🎲 Probability Fundamentals Review

Before we can understand probabilistic classifiers, we need to be comfortable with three key probability concepts — prior, likelihood, and posterior.

Prior Probability P(h)

Your initial belief about hypothesis h before seeing any data.

Example: Before any test, the probability someone has a disease is P(disease) = 0.001 (1 in 1000).

Reflects background knowledge.

Likelihood P(D|h)

The probability of observing data D if hypothesis h is true.

Example: P(positive test | disease) = 0.99 (the test is 99% sensitive).

How well the hypothesis explains the data.

Posterior Probability P(h|D)

Your updated belief about h after seeing data D.

Example: P(disease | positive test) = ? (what we want to compute)

The goal of Bayesian inference.

Medical Test Worked Example

This is one of the most counter-intuitive results in probability — the base-rate fallacy.

Given:
P(disease) = 0.001  (1 in 1000 people have the disease)
P(+ | disease) = 0.99  (sensitivity: 99% — test correctly identifies sick people)
P(+ | no disease) = 0.01  (1 − specificity: 1% false positive rate)

Question: If you test positive, what is the probability you actually have the disease?

Step-by-Step Solution

1
Compute P(D) — the total probability of testing positive (evidence):
P(+) = P(+|disease)×P(disease) + P(+|no disease)×P(no disease)
P(+) = 0.99×0.001 + 0.01×0.999
P(+) = 0.00099 + 0.00999 = 0.01098
2
Apply Bayes' Theorem:
P(disease | +) = P(+ | disease) × P(disease) / P(+)
P(disease | +) = (0.99 × 0.001) / 0.01098
P(disease | +) = 0.00099 / 0.01098 ≈ 0.0902 ≈ 9%
Surprising result! Even with a 99%-accurate test, a positive result only means ~9% chance of disease. Why? Because the disease is so rare (0.1%) that false positives (1% of the 99.9% healthy people) outnumber true positives. This is the base-rate fallacy — ignoring the prior can lead to massively incorrect intuitions.

📐 Bayes' Theorem

Bayes' Theorem is the mathematical backbone of probabilistic machine learning. It tells us how to update our beliefs in light of new evidence.

P(h|D) = P(D|h) × P(h) / P(D)

Each Term Explained

  • P(h|D) — Posterior: probability of hypothesis given data
  • P(D|h) — Likelihood: probability of data given hypothesis
  • P(h) — Prior: initial belief about hypothesis
  • P(D) — Evidence/Normaliser: total probability of the data

The normaliser P(D):

P(D) = Σ P(D|hᵢ)×P(hᵢ) over all hypotheses hᵢ

For binary hypotheses: P(D) = P(D|h)P(h) + P(D|¬h)P(¬h)

This ensures P(h|D) is a valid probability (sums to 1).

Intuition: Belief Update

Think of Bayes' theorem as a belief update machine:

Posterior ∝ Likelihood × Prior

The evidence P(D) is just a normalising constant — we can think of it as:

New Belief = How well data fits × Old Belief

Each new observation updates our beliefs. With enough data, the prior becomes irrelevant — the likelihood dominates.

Key insight: Bayesian learning is incremental. Today's posterior becomes tomorrow's prior as new evidence arrives.

Toy Example: Is a Coin Fair?

Suppose we have a coin. We want to know: is it fair (P(H)=0.5) or biased (P(H)=0.9)?

ObservationPrior P(fair)Likelihood updatePosterior P(fair)
No flips yet0.50—0.50
Flip 1: Heads0.50P(H|fair)=0.5, P(H|biased)=0.90.50×0.5 / [0.50×0.5+0.50×0.9] ≈ 0.36
Flip 2: Heads0.36Another heads update≈ 0.25
Flip 3: Heads0.25Another heads update≈ 0.16
Flip 4: Tails0.16P(T|fair)=0.5, P(T|biased)=0.1≈ 0.44 (jumps back up)

Each new observation updates our belief that the coin is fair.

🎯 MAP vs MLE

In machine learning, we often want to find the best hypothesis h given data D. There are two main approaches.

Maximum Likelihood Estimation (MLE)

h_MLE = argmax P(D|h)

Find the hypothesis that makes the observed data most probable. Ignores the prior entirely — just maximises how well the hypothesis explains the data.

Intuition: "Which coin bias makes these 10 flips most likely?"

If you observe 7 heads in 10 flips:
MLE says θ = 7/10 = 0.7 (even if you think coins should be fair)

Equivalent to: Set prior P(h) = uniform (all hypotheses equally likely), then MAP = MLE.

Maximum A Posteriori (MAP)

h_MAP = argmax P(D|h) × P(h)

Find the hypothesis with the highest posterior probability. Incorporates prior beliefs — balances the evidence with what you already know.

Intuition: "Which coin bias is most likely given both the flips AND my prior belief that coins are usually fair?"

Same 7 heads in 10 flips:
MAP (with strong fair-coin prior) might give θ ≈ 0.6 (pulled toward fair).

Effect of prior: Acts as regularisation — prevents overfitting to small samples. With large data, MAP → MLE.

🏷️ Naive Bayes Classifier

Naive Bayes is a probabilistic classifier based on Bayes' theorem. It is called "naive" because it makes a strong (and usually wrong, but surprisingly useful) assumption about features.

The Core Idea

We want to classify a new example x = (x₁, x₂, ..., xₙ) into one of the classes y ∈ {c₁, c₂, ...}. Using Bayes:

P(y | x₁,...,xₙ) = P(x₁,...,xₙ | y) × P(y) / P(x₁,...,xₙ)

The denominator P(x₁,...,xₙ) is constant for all classes, so we can ignore it for argmax:

ŷ = argmax_y P(y) × P(x₁,x₂,...,xₙ | y)

The Naive Independence Assumption

Computing P(x₁,...,xₙ | y) is exponentially hard — for n binary features, you'd need 2ⁿ parameters! The key assumption:

Conditional Independence Assumption: Given the class y, all features are independent of each other.
P(x₁,x₂,...,xₙ | y) = P(x₁|y) × P(x₂|y) × ... × P(xₙ|y) = ∏ P(xᵢ|y)

This reduces 2ⁿ parameters to just n×|classes| parameters — a massive simplification!

Full Naive Bayes Classifier

ŷ = argmax_y P(y) × ∏ᵢ P(xᵢ | y)

Why "Naive"?

The independence assumption is almost always violated in practice:

  • In emails, words co-occur ("free" and "offer" appear together)
  • Medical symptoms are correlated (fever and chills)
  • Pixels in images are spatially correlated

Despite this, NB works surprisingly well in many real applications!

Why it still works

Naive Bayes often gives excellent ranking even if exact probabilities are wrong. The independence assumption is violated symmetrically — the errors partially cancel out. What matters is the argmax, not the exact value.

Practical evidence: NB is still competitive with SVM on text classification despite the strong assumption.

Advantages and Limitations

AdvantagesLimitations
Fast: O(n) training and O(n) prediction Independence assumption: rarely true in practice
Small data: works well even with few samples Zero probability: if any P(xᵢ|y)=0, whole product = 0
High dimensions: handles many features gracefully Poor probability estimates: calibration can be off
Incremental: easy to update with new data Correlated features: counted multiple times
Interpretable: probabilities are human-readable Continuous features: require distribution assumption (e.g. Gaussian)

Log Trick — Avoiding Underflow

Multiplying many small probabilities leads to numerical underflow (e.g., 0.001^100 → 0). Solution: take logs.

ŷ = argmax_y log P(y) + Σᵢ log P(xᵢ | y)

Since log is monotonically increasing, argmax is preserved. This transforms products into sums — numerically stable and faster.

🧮 Laplace Smoothing

The Zero Probability Problem

If a word never appears in training documents of a given class, its estimated probability is 0. This causes the entire product to collapse to 0:

P("unicorn" | spam) = 0 → P(y=spam | document) = 0 regardless of other words

One unseen word completely kills the classification — clearly wrong!

Example without smoothing:
Training: 10 "sports" documents, word "cricket" appears in 0 of them.
P("cricket" | sports) = 0/10 = 0
A new document mentioning "cricket" once along with many sport words will be classified as NOT sports. Wrong!

Laplace (Add-k) Smoothing

Add a small constant k (usually k=1) to every count before normalising. This ensures no probability is ever exactly zero.

P(xᵢ = v | y = c) = (count(xᵢ=v, y=c) + k) / (count(y=c) + k × |V|)

Where |V| is the vocabulary size (number of distinct values xᵢ can take).

Without Smoothing (k=0)

Vocabulary: {A, B, C, D} (|V| = 4)

Class "positive" has 10 training samples:
A appears 4 times, B appears 3 times, C appears 3 times, D appears 0 times.

P(A|+) = 4/10 = 0.40
P(B|+) = 3/10 = 0.30
P(C|+) = 3/10 = 0.30
P(D|+) = 0/10 = 0.00 ← PROBLEM

Any test example with D → P(+) = 0!

With Laplace Smoothing (k=1)

Vocabulary: {A, B, C, D} (|V| = 4)

Class "positive" has 10 training samples (same as left).

P(A|+) = (4+1)/(10+4) = 5/14 ≈ 0.357
P(B|+) = (3+1)/(10+4) = 4/14 ≈ 0.286
P(C|+) = (3+1)/(10+4) = 4/14 ≈ 0.286
P(D|+) = (0+1)/(10+4) = 1/14 ≈ 0.071 ← Fixed!

No zero probabilities. D gets small but non-zero probability.

Effect of k on Smoothing

k valueEffectWhen to use
k = 0No smoothing — zero probability possibleNever (unless very large dataset)
k = 1Laplace smoothing — each value gets count of 1Most common default
0 < k < 1Lidstone smoothing — gentler than LaplaceWhen vocabulary is large
k → ∞Uniform distribution — ignores training dataNever practical
Key insight: Laplace smoothing can be interpreted as adding |V| "hallucinated" training examples — one for each possible feature value. It's a form of regularisation that expresses prior belief that all values are possible.

📈 Gaussian Naive Bayes

When features are continuous (e.g., temperature, height, age), we can't count occurrences. Instead, we model each feature as a Gaussian (normal) distribution per class.

Gaussian Probability Density Function

P(xᵢ | y) = 1/√(2πσ²) × exp(−(xᵢ − μ)² / (2σ²))

For each class c and each feature i, we learn:

  • μ_{i,c} = mean of feature xᵢ among all training examples with class c
  • σ²_{i,c} = variance of feature xᵢ among all training examples with class c

Step 1: Parameter Estimation

Class: Sick (4 patients)

Temperature: {38.8, 38.2, 38.6, 38.4}

μ_temp,sick = (38.8+38.2+38.6+38.4)/4 = 38.5°C

σ²_temp,sick = Var = [(38.8−38.5)²+(38.2−38.5)²+(38.6−38.5)²+(38.4−38.5)²]/4

= [0.09+0.09+0.01+0.01]/4 = 0.05

Headache: {8, 7, 9, 6} → μ = 7.5, σ² = 1.25

P(Sick) = 4/8 = 0.5

Class: Healthy (4 patients)

Temperature: {36.6, 36.9, 37.1, 36.8}

μ_temp,healthy = (36.6+36.9+37.1+36.8)/4 = 36.85°C

σ²_temp,healthy = [(36.6−36.85)²+(36.9−36.85)²+(37.1−36.85)²+(36.8−36.85)²]/4

= [0.0625+0.0025+0.0625+0.0025]/4 = 0.0325

Headache: {2, 3, 1, 2} → μ = 2.0, σ² = 0.5

P(Healthy) = 4/8 = 0.5

Step 2: Classify New Patient

New patient: Temperature = 38.3°C, Headache = 7
1
Compute Gaussian likelihoods for "Sick":
P(38.3 | Sick) = 1/√(2π×0.05) × exp(−(38.3−38.5)²/(2×0.05))
= 1/√(0.3142) × exp(−0.04/0.1)
= 1.784 × exp(−0.4) = 1.784 × 0.6703 ≈ 1.196

P(7 | Sick) = 1/√(2π×1.25) × exp(−(7−7.5)²/(2×1.25))
= 0.3568 × exp(−0.1) = 0.3568 × 0.9048 ≈ 0.3228
2
Compute Gaussian likelihoods for "Healthy":
P(38.3 | Healthy) = 1/√(2π×0.0325) × exp(−(38.3−36.85)²/(2×0.0325))
= 2.211 × exp(−32.1) ≈ ≈ 0 (essentially zero)

P(7 | Healthy) = 1/√(2π×0.5) × exp(−(7−2.0)²/(2×0.5))
= 0.5642 × exp(−12.5) ≈ ≈ 0 (essentially zero)
3
Compare posteriors (∝ prior × likelihood):
Score(Sick) = P(Sick) × P(38.3|Sick) × P(7|Sick) = 0.5 × 1.196 × 0.3228 ≈ 0.193
Score(Healthy) = P(Healthy) × P(38.3|Healthy) × P(7|Healthy) ≈ 0.5 × 0 × 0 ≈ ≈ 0
Prediction: SICK — the patient's 38.3°C temperature and headache intensity of 7 strongly match the Sick class parameters.

🔤 Text Preprocessing for Naive Bayes

Raw text must be transformed before NB can process it. Good preprocessing dramatically improves accuracy.

1. Tokenisation

Split text into individual tokens (words/subwords).

"Hello, World! It's great."
→ ["Hello", "World", "It", "s", "great"]

Considerations: punctuation, apostrophes, numbers, case.

2. Stop Word Removal

Remove high-frequency words that carry little semantic meaning.

["the", "is", "at", "which", "on", "a", "an", "the"]
→ removed from document

Reduces noise and computation. Domain-specific stop words may differ.

3. Stemming / Lemmatisation

Reduce words to their base/root form to consolidate vocabulary.

running → run
better → good (lemma)
studies → studi (stem)

Stemming is fast but crude; lemmatisation is accurate but slower.

TF-IDF: A Better Feature Representation

Bag of Words treats all words equally. TF-IDF (Term Frequency–Inverse Document Frequency) down-weights common words and up-weights rare but informative ones.

TF-IDF(w, d) = TF(w, d) × IDF(w) TF(w, d) = count(w in d) / total words in d IDF(w) = log(N / df(w)) where df(w) = documents containing w
Intuition for IDF: If "the" appears in all 1000 docs, IDF = log(1000/1000) = 0 — "the" is useless. If "algorithm" appears in only 10 docs, IDF = log(1000/10) = log(100) ≈ 4.6 — "algorithm" is highly informative.
When to use TF-IDF over raw counts: When documents are of varying lengths, when common domain words dominate (e.g., "patient" in medical NB), or when you want better feature discrimination. Note: standard Naive Bayes uses raw counts with Laplace smoothing.
