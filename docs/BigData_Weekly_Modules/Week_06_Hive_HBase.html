1️⃣ Introduction to Apache Hive

Hive is a data warehouse infrastructure built on top of Hadoop. It provides SQL-like querying for analyzing large datasets stored in HDFS (or HBase).

Purpose

  • Makes Hadoop accessible to users familiar with SQL
  • Projects structure onto unstructured/semi-structured data
  • Enables ad-hoc querying, data analysis, and ETL at scale

Key Features

📝 HiveQL

SQL-like query language. Familiar to anyone who knows SQL.

⚙️ Translates to MR/Tez/Spark

Hive queries get compiled to underlying execution engines.

🗄️ Storage Flexibility

Supports HDFS, HBase, S3, and more.

📊 RDBMS-like Concepts

Defines databases, tables, partitions just like SQL.

⚠️ Not for OLTP: Hive is for analytics (OLAP), not real-time transactions. Queries take seconds to minutes.

3️⃣ Hive File Formats

File format dramatically affects query performance, storage efficiency, and schema evolution.

📄 TextFile (default)

Pros: Plain text, human-readable, easy debugging

Cons: No compression, poor query performance

Use when: Initial loading, external tools producing CSV

📦 SequenceFile

Pros: Binary K/V format, compact, block-compressible, splittable

Cons: Not as efficient as columnar formats

Use when: Need binary storage but not column-specific

📑 RCFile

Pros: Column-oriented — good for queries reading specific columns

Cons: Older format, mostly replaced by ORC

🏆 ORC (Optimized Row Columnar)

Pros: Highly optimized columnar, Zlib/Snappy compression, stripe-level indexes, predicate pushdown, supports complex types & schema evolution

Often the best choice for Hive

📊 Parquet

Pros: Popular columnar, used in Spark/Impala/Hive

Use when: Cross-tool data sharing (Hive + Spark)

🔄 Avro

Pros: Row-based, schema stored with data (in JSON), schema evolution friendly

Use when: Schema changes frequently

Quick Comparison

SequenceFileRowYes (block)LimitedBinary K/V

ParquetColumnExcellentYesMulti-tool (Spark)

4️⃣ HiveQL — DDL (Data Definition Language)

DDL defines/modifies database/table structures.

Database Operations

CREATE DATABASE sales; SHOW DATABASES; USE sales; DROP DATABASE sales CASCADE; -- CASCADE drops all tables too

Table Operations

CREATE TABLE orders ( order_id INT, customer STRING, amount DOUBLE, order_date STRING ) PARTITIONED BY (region STRING) STORED AS ORC; -- View schema DESCRIBE orders; DESCRIBE FORMATTED orders; -- detailed -- Alter table ALTER TABLE orders ADD COLUMNS (currency STRING); ALTER TABLE orders RENAME TO customer_orders; ALTER TABLE orders SET TBLPROPERTIES ('comment' = 'Updated 2024'); -- Drop / truncate TRUNCATE TABLE orders; -- removes data, keeps schema DROP TABLE orders; -- removes everything

External vs Managed Tables

TypeHive Owns Data?DROP behavior
Managed (default)YesDeletes data + metadata
ExternalNoDeletes only metadata, data stays
CREATE EXTERNAL TABLE logs ( ts STRING, level STRING, message STRING ) LOCATION '/data/external/logs';

Partitioning

Partitioning splits a table into directories by partition column values — speeds up queries that filter by the partition.

CREATE TABLE sales_data ( product STRING, amount DOUBLE ) PARTITIONED BY (year INT, month INT); -- Add a partition ALTER TABLE sales_data ADD PARTITION (year=2024, month=5); -- Query with partition filter (fast!) SELECT * FROM sales_data WHERE year=2024 AND month=5;

5️⃣ HiveQL — DML (Data Manipulation Language)

Loading Data

-- Load from HDFS file LOAD DATA INPATH '/data/raw/orders.csv' INTO TABLE orders; -- Load from local LOAD DATA LOCAL INPATH '/home/user/orders.csv' INTO TABLE orders; -- Overwrite (replaces existing data) LOAD DATA INPATH '/data/orders.csv' OVERWRITE INTO TABLE orders;

INSERT

-- Insert into table from query INSERT INTO TABLE orders_summary SELECT customer, SUM(amount) FROM orders GROUP BY customer; -- Insert overwrite (replaces existing data) INSERT OVERWRITE TABLE orders_summary SELECT customer, SUM(amount) FROM orders GROUP BY customer; -- Insert into partitioned table INSERT INTO TABLE sales_data PARTITION (year=2024, month=5) SELECT product, amount FROM staging;

SELECT

-- Basic SELECT * FROM orders LIMIT 10; -- Filter + aggregate SELECT region, SUM(amount) AS total FROM orders WHERE order_date >= '2024-01-01' GROUP BY region ORDER BY total DESC LIMIT 5; -- Joins SELECT c.name, o.amount FROM customers c JOIN orders o ON c.id = o.customer_id; -- Subqueries SELECT * FROM ( SELECT customer, SUM(amount) AS total FROM orders GROUP BY customer ) t WHERE total > 1000;

SUM, AVG, MIN, MAXStandard aggregates

6️⃣ HBase Architecture (Recap)

Quick recap from Week 5 — HBase uses master-slave:

👑 HMaster
Assigns regions, monitors via ZooKeeper
🦓 ZooKeeper
Coordination
🔧 RegionServer
MemStore + BlockCache + HFile + WAL

Write Flow

  1. Client → finds RegionServer via ZooKeeper
  2. Data → WAL (durability log on HDFS)
  3. Data → MemStore (in-memory write cache)
  4. When MemStore fills → flush to HFile on HDFS
  5. Periodic compaction merges small HFiles

Read Flow

  1. Client → finds RegionServer via ZooKeeper
  2. Check BlockCache (hot data)
  3. Check MemStore (recent writes not flushed yet)
  4. Check HFiles on HDFS
  5. Return combined results
💡 Why WAL? If RegionServer crashes before MemStore is flushed, the WAL replays writes to recover.

7️⃣ HBase Shell — DDL Commands

Access via: hbase shell

Table Management

# Create table with column families create 'users', 'profile', 'activity' # Example: create with column family options create 'users', {NAME => 'profile', VERSIONS => 3}, {NAME => 'activity'} # List all tables list # Describe table schema describe 'users' # Check if table exists exists 'users'

Alter Table

# Add column family alter 'users', {NAME => 'new_cf'} # Modify column family (versions) alter 'users', {NAME => 'profile', VERSIONS => 5} # Delete column family alter 'users', {NAME => 'activity', METHOD => 'delete'}

Disable / Enable / Drop

# IMPORTANT: must disable before drop disable 'users' enable 'users' # Drop table (must be disabled first) disable 'users' drop 'users'
⚠️ Common mistake: Forgetting to disable a table before dropping! HBase will error out.

8️⃣ HBase Shell — DML Commands

PUT — Insert/Update

put 'users', 'user123', 'profile:name', 'John Doe' put 'users', 'user123', 'profile:email', 'john@example.com' put 'users', 'user123', 'activity:last_login', '2024-05-15'

GET — Retrieve

# Get entire row get 'users', 'user123' # Get specific column get 'users', 'user123', {COLUMN => 'profile:email'} # Get multiple columns get 'users', 'user123', {COLUMNS => ['profile:name', 'profile:email']} # Get with version (timestamp) get 'users', 'user123', {COLUMN => 'profile:email', VERSIONS => 3} # Get within time range get 'users', 'user123', {TIMERANGE => [1620000000000, 1630000000000]}

SCAN — Range Query

# Scan whole table scan 'users' # Limit rows scan 'users', {LIMIT => 10} # Range scan scan 'users', {STARTROW => 'user100', STOPROW => 'user200'} # Filter by column scan 'users', {COLUMNS => 'profile:email'} # Filter expression scan 'users', {FILTER => "ValueFilter(=, 'binary:John Doe')"}

DELETE

# Delete specific column delete 'users', 'user123', 'profile:name' # Delete all data for a row deleteall 'users', 'user123' # Delete a specific version (timestamp) delete 'users', 'user123', 'profile:name', 1620000000000

INCR — Atomic Counter

# Increment counter (atomic operation) incr 'counters', 'page_views', 'stats:count', 1 incr 'counters', 'page_views', 'stats:count', 5 # increment by 5

Counts & Stats

count 'users' # row count count 'users', INTERVAL => 1000 # show progress every 1000 rows

9️⃣ HBase vs RDBMS — Detailed Comparison

FeatureHBaseRDBMS (e.g., MySQL)
Data ModelColumn-family (wide-column)Row-oriented (tables/relations)
SchemaSchema-on-read (flexible, dynamic cols)Schema-on-write (fixed, predefined)
ScalabilityHorizontal (add nodes)Vertical (bigger server); sharding is complex
Data SizeDesigned for PB-scaleTypically GB to a few TB
Query LanguageJava API, REST, Thrift, Shell (no SQL)SQL
JoinsLimited / Not nativeSupported (INNER, OUTER, etc.)
TransactionsRow-level atomicity onlyFull ACID transactions
NormalizationDenormalized is commonNormalized (3NF) preferred
Sparse DataExcellent (empty cells = 0 bytes)Poor (NULL takes space)
IndexesPrimary RowKey + secondary (via Phoenix)Multiple indexes natively
Use CaseReal-time random access, sparse dataTransactional apps, complex queries

When to Choose What?

Choose HBase if...

  • Data is in PB scale
  • Need real-time random reads/writes
  • Data is sparse (varying columns)
  • Already in Hadoop ecosystem
  • Time-series data (logs, sensors)

Choose RDBMS if...

  • Data fits in single server (TB scale)
  • Need full ACID transactions
  • Complex SQL queries with joins
  • Schema is stable and well-defined
  • Strong relational integrity required

📝 Summary & Exam Tips

🎯 Key Takeaways

  • Hive = SQL-like warehouse on Hadoop (HiveQL → MR/Tez/Spark)
  • Hive components: Metastore + Driver + Compiler + Optimizer + Executor
  • File formats: TextFile (default), SequenceFile, RCFile, ORC ⭐ (best for Hive), Parquet, Avro
  • DDL: CREATE/ALTER/DROP DATABASE/TABLE
  • DML: LOAD DATA, INSERT INTO/OVERWRITE, SELECT
  • External tables: data not owned by Hive — DROP keeps the data
  • HBase Shell DDL: create, disable → drop, describe, alter
  • HBase Shell DML: put, get, scan, delete, incr
  • HBase write: WAL → MemStore → HFile (compaction merges)
  • HBase vs RDBMS: HBase for sparse, big, real-time; RDBMS for joins, ACID
