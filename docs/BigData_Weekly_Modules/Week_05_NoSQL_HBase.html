1️⃣ What is NoSQL?

NoSQL stands for "Not Only SQL". It's a diverse category of database management systems that deviate from the traditional relational (SQL) model.

Key Differences from SQL

  • Don't enforce strict tabular structures (rows/columns)
  • No predefined schemas
  • Offer flexible ways to store/retrieve data
  • Designed for large volumes, varied types (unstructured/semi-structured), high-velocity data
💡 The name is intentional: "Not Only SQL" means NoSQL doesn't replace SQL — it complements it for use cases where relational databases struggle.

2️⃣ Why NoSQL? The Motivation

NoSQL emerged because traditional RDBMS struggle with the scale and agility requirements of modern web applications and Big Data.

Key Motivations

📈 Scalability

Need to scale beyond a single server. RDBMS hits limits when vertical scaling becomes too expensive.

🎭 Data Flexibility

Modern data is messy — JSON, logs, social posts, sensor data. Schema must evolve quickly.

⚡ Availability & Performance

24/7 web apps can't afford downtime; need low-latency responses worldwide.

🐘 Handling Big Data

Petabytes of data, billions of records, millions of writes/sec.

3️⃣ Key Characteristics of NoSQL

🚫 Non-Relational

Data isn't forced into table structures with rows/columns.

🔄 Schema Flexibility

Dynamic or implicit schemas — each document/row can have different fields.

📈 Horizontal Scalability

Designed to scale OUT (add more servers) instead of UP (bigger server).

🌐 Distributed Architecture

Often inherently distributed across nodes for availability and fault tolerance.

🎛️ Varied Consistency Models

Many NoSQL DBs offer tunable consistency based on CAP theorem — trade strong consistency for availability/partition tolerance.

SQL vs NoSQL on consistency: SQL typically emphasizes strong consistency (ACID). NoSQL often prioritizes availability + partition tolerance, sometimes at the cost of strict consistency.

4️⃣ Four Types of NoSQL Databases

📄 ① Document Databases

Store: Document-like structures (JSON, BSON, XML)

Best for: Content management, e-commerce, analytics where data structure varies

Examples: MongoDB, CouchDB

🔑 ② Key-Value Stores

Store: Simplest form — unique keys paired with values

Best for: Caching, user profiles, session management (fast lookups)

Examples: Redis, DynamoDB, Riak

📊 ③ Column-Oriented (Wide-Column)

Store: Data in columns rather than rows

Best for: Queries over large datasets that only need specific columns

Examples: HBase, Cassandra

🕸️ ④ Graph Databases

Store: Data as nodes and edges

Best for: Social networks, recommendation engines, fraud detection

Examples: Neo4j, ArangoDB

Choosing the Right Type

Use CaseBest TypeWhy
User session cacheKey-ValueSimple lookups, fast TTL
Product catalogDocumentProducts have varying attributes
IoT sensor dataColumn-orientedTime-series + sparse columns
Social network friendsGraphHeavy relationship queries
Real-time analyticsColumn-orientedAggregate over specific columns

5️⃣ CAP Theorem — The Distributed Database Trade-off

The CAP theorem states that a distributed data store can guarantee at most 2 out of 3 of:

C — Consistency
all nodes see same data
A — Availability
every request gets a response
P — Partition Tolerance
system works despite network failures

Combinations

PickWhat you give upExamples
CP (Consistency + Partition)AvailabilityHBase, MongoDB (default), Redis
AP (Availability + Partition)Strong ConsistencyCassandra, CouchDB, DynamoDB
CA (Consistency + Availability)Partition ToleranceTraditional RDBMS (single-node)
⚠️ Reality: In a real distributed system, network partitions WILL happen. So you must choose between CP or AP. CA is only theoretical (single node).

6️⃣ SQL vs NoSQL — Full Comparison

FeatureSQL (Relational)NoSQL (Non-Relational)
Data ModelTables with rows/colsDocument, Key-Value, Column, Graph
SchemaPredefined, rigidDynamic, flexible
ScalabilityPrimarily Vertical (Scale-up)Primarily Horizontal (Scale-out)
Data StructureStructured onlyStructured + Semi + Unstructured
ConsistencyStrong (ACID)Tunable (often Eventual / BASE)
JoinsNative supportLimited or none
TransactionsFull ACIDLimited (often row-level)
Query LanguageSQL (standardized)Varies per DB (CQL, MongoDB query, etc.)
Use CasesTransactional apps, complex queriesBig data, real-time apps, high scalability
ExamplesMySQL, PostgreSQL, OracleMongoDB, Cassandra, HBase, Redis
💡 BASE vs ACID:
  • ACID (SQL): Atomicity, Consistency, Isolation, Durability
  • BASE (many NoSQL): Basically Available, Soft state, Eventual consistency

7️⃣ Apache HBase

HBase is a NoSQL database modeled after Google's Bigtable. It runs on top of HDFS.

Key Features

🗄️ Built on HDFS

Inherits HDFS's reliability + scalability

📊 Column-Oriented

Stores data by column family for efficient column-specific queries

⚡ Real-Time R/W

Random access to very large datasets

🚀 Overcomes HDFS Limits

HDFS is write-once + sequential; HBase adds random access + in-place updates

📦 Sparse Data Friendly

Empty cells take no space — great for sparse data with thousands of optional columns

🛡️ Fault Tolerant

Built-in replication via HDFS underneath

When to Use HBase?

  • Need real-time random reads/writes on huge tables (billions of rows)
  • Data has varying columns per row (sparse data)
  • Want to integrate with Hadoop ecosystem
  • Need strong consistency (HBase is CP in CAP)
⚠️ When NOT to use HBase:
  • You need joins or complex SQL queries → use Hive or RDBMS
  • You have small data (few GB) → use RDBMS
  • You need true ACID transactions across rows → use RDBMS

8️⃣ HBase vs Hive — Common Confusion

Both are in the Hadoop ecosystem but serve different purposes:

📊 HBase

What: NoSQL database on HDFS

Use: Real-time, random R/W

Query: Java API, REST, shell (no SQL)

Strength: Sparse data, fast lookups, updates

🐝 Hive

What: Data warehouse on Hadoop

Use: Analytics, reporting, ETL (batch)

Query: HiveQL (SQL-like)

Strength: Complex SQL queries on big data

Side-by-side

AspectHBaseHive
PurposeReal-time data storeBatch analytics warehouse
LatencyLow (ms)High (seconds to minutes)
Query LanguageNo SQLHiveQL (SQL-like)
SchemaSchema-on-readSchema-on-read
UpdatesSingle row updatesLimited (append/overwrite)
JoinsLimitedYes (translates to MR/Tez/Spark)

9️⃣ HBase Architecture

HBase uses a master-slave architecture:

👑 HMaster
Master server. Assigns regions, schema changes, monitors via ZooKeeper, load balancing.
🦓 ZooKeeper
Coordination service. Tracks cluster state, holds config.
🔧 RegionServer 1
Serves regions. Has MemStore + BlockCache + HFile + WAL.
🔧 RegionServer 2
Same as above. Each manages multiple regions.

HMaster (Master)

  • Manages the cluster
  • Assigns regions (data partitions) to RegionServers
  • Handles schema changes & metadata operations
  • Monitors RegionServers via ZooKeeper
  • Manages load balancing across RegionServers

RegionServers (Workers)

  • Worker nodes that serve data regions
  • Handle client read/write requests for assigned regions
  • Each manages multiple regions (a region = range of table rows)

RegionServer Internal Components

💾 MemStore

In-memory write cache. New data goes here first. Flushed to HFiles when full.

📚 BlockCache

In-memory read cache. Stores frequently accessed data for fast reads.

📁 HFile

Actual storage file on HDFS. Stores sorted key-values persistently.

📝 WAL (Write Ahead Log)

Durability log on HDFS. New data → WAL first, then MemStore. If RegionServer crashes, WAL recovers data.

ZooKeeper's Role

  • Maintains cluster state
  • Tracks active/failed RegionServers
  • Holds configuration information
  • Used for leader election (in case HMaster fails)

Architecture Diagram (Textual)

+--------------+ (find RegionServer) +-----------+ | HBase Client | -----------------------> | ZooKeeper | <---> +---------+ +--------------+ +-----------+ | HMaster | | ^ +---------+ | (read/write) | (heartbeat) | v | | (monitor) +--------------+ (manages regions) +--------------+ (stores HFiles) +------+ | RegionServer | <---------------> | RegionServer | --------------> | HDFS | +--------------+ +--------------+ +------+

🔟 HBase Data Model

HBase uses a column-family-based data model. Don't confuse it with relational tables!

Core Concepts

ConceptDescription
TableStores data, column-oriented (not row-oriented like RDBMS)
RowKeyUnique row identifier. Acts as the primary index. Rows sorted lexicographically.
Column FamilyGroups related columns. Defined at table creation. Columns within a family stored together physically.
Column QualifierSpecific column name within a family. Can be added dynamically per row.
CellIntersection of (RowKey, Family:Qualifier) — contains the actual value.
TimestampEvery cell value is versioned with a timestamp. HBase stores multiple versions.

Visual Example

RowKey | Family: profile | Family: activity | name | email | last_login | clicks ------------+-------+-----------------+-------------+-------- user001 | Alice | alice@x.com | 2024-05-01 | 42 user002 | Bob | bob@y.com | 2024-05-15 | (empty) user003 | Carol | carol@z.com | (empty) | 5
💡 Key insight: Empty cells don't take space — HBase efficiently stores sparse data with millions of optional columns.

Common Use Cases

  • Time-series data — append-heavy workloads with timestamps
  • Messaging systems — store message history per user
  • Web indexes — Bigtable was originally built for Google's web crawl
  • IoT data — sparse sensor data with millions of devices

📝 Summary & Exam Tips

🎯 Key Takeaways

  • NoSQL = "Not Only SQL" — flexible, scalable, non-relational
  • 4 types: Document (MongoDB), Key-Value (Redis), Column (HBase, Cassandra), Graph (Neo4j)
  • CAP theorem: Pick 2 of Consistency, Availability, Partition tolerance — in practice CP or AP
  • HBase = Column-oriented NoSQL DB on HDFS for real-time R/W of sparse big data
  • HBase architecture: HMaster (master) + RegionServers (workers) + ZooKeeper (coord)
  • RegionServer internals: MemStore (write), BlockCache (read), HFile (storage), WAL (durability)
  • Data model: Table → RowKey → Column Family → Column Qualifier → Cell (with Timestamp)
  • HBase vs Hive: HBase = DB for real-time; Hive = warehouse for batch SQL analytics
📌 Common Exam Pitfalls:
  • Confusing HBase (column-oriented DB) with Hive (SQL warehouse)
  • Saying NoSQL means "no schema" (it means "flexible schema")
  • Forgetting that HBase is CP in CAP (strong consistency)
  • Cassandra and HBase BOTH column-oriented but DIFFERENT (Cassandra is AP, HBase is CP)
