1️⃣ MapReduce Flow — Visual Recap

This diagram shows the canonical MapReduce flow:

  1. Input files in HDFS are split into chunks
  2. Each chunk → Mapper instance (parallel)
  3. Mapper output → intermediate key-value pairs (stored on local disk)
  4. Shuffle phase transfers data to Reducers, sorts/merges by key
  5. Each Reducer processes one or more keys → final output back to HDFS

2️⃣ Example 1 — Word Count (The "Hello World" of MR)

Problem: Count the frequency of each word in a text corpus.

Pseudocode

// Map function map(line_number, line_text): for each word in line_text.split(): emit(word, 1) // Reduce function reduce(word, list_of_counts): total = sum(list_of_counts) emit(word, total)

Simple Trace

Input: "A B R" "C C R" "A C B" After Map: (A,1)(B,1)(R,1) (C,1)(C,1)(R,1) (A,1)(C,1)(B,1) After Shuffle/Sort (group by key): A → [1,1] B → [1,1] C → [1,1,1] R → [1,1] After Reduce: (A,2)(B,2)(C,3)(R,2)

3️⃣ Detailed Walkthrough — Animal Words

Let's walk through the canonical example from the PDF: "Deer Bear River | Car Car River | Deer Car Bear"

Input lines (split into 3 chunks): Chunk 1: "Deer Bear River" Chunk 2: "Car Car River" Chunk 3: "Deer Car Bear" ============== MAP PHASE ============== Mapper 1 ("Deer Bear River"): emit (Deer, 1), (Bear, 1), (River, 1) Mapper 2 ("Car Car River"): emit (Car, 1), (Car, 1), (River, 1) Mapper 3 ("Deer Car Bear"): emit (Deer, 1), (Car, 1), (Bear, 1) ============ SHUFFLE & SORT ============ Group by key (with optional combiner): Bear → [1, 1] Car → [1, 1, 1] Deer → [1, 1] River → [1, 1] ============== REDUCE PHASE ============ Reducer: (Bear, 2) ← sum of [1,1] (Car, 3) ← sum of [1,1,1] (Deer, 2) (River, 2) ============= FINAL OUTPUT ============= Bear 2 Car 3 Deer 2 River 2
💡 Notice: Each Mapper works independently in parallel. The Shuffle/Sort stage is automatic and handles all the cross-machine data movement.

Where would a Combiner help?

If Mapper 2 saw "Car Car River":

  • Without Combiner: emits (Car,1)(Car,1)(River,1) — 3 pairs shuffled
  • With Combiner: combines locally → (Car,2)(River,1) — only 2 pairs shuffled

Combiner reduces network traffic. Since sum is associative & commutative, it's safe.

4️⃣ Example 2 — Matrix Multiplication

Problem: Compute C = A × B, where:

C[i][j] = Σ_k A[i][k] × B[k][j]

Approach

The key insight: for each element C[i][j] of the result, we need to multiply rows of A by columns of B. We can emit (i,j) as the key so all relevant values converge at one Reducer.

Map Function

// For matrix A of size m×n for each element A[i][k]: for each j in 1..p (where B is n×p): emit ((i,j), ("A", k, A[i][k])) // For matrix B of size n×p for each element B[k][j]: for each i in 1..m: emit ((i,j), ("B", k, B[k][j]))

Reduce Function

reduce((i,j), values): A_dict = {} // k -> value B_dict = {} // k -> value for (source, k, v) in values: if source == "A": A_dict[k] = v else: B_dict[k] = v result = 0 for k in A_dict: if k in B_dict: result += A_dict[k] * B_dict[k] emit((i,j), result)

Concrete Example

A = [[1,2],[3,4]] (2×2), B = [[5,6],[7,8]] (2×2)

Expected: C = [[19,22],[43,50]]

For element C[0][0]: A[0][0]×B[0][0] + A[0][1]×B[1][0] = 1×5 + 2×7 = 5+14 = 19 ✓

Map emits for key (0,0):

  • ("A", 0, 1) — from A[0][0]
  • ("A", 1, 2) — from A[0][1]
  • ("B", 0, 5) — from B[0][0]
  • ("B", 1, 7) — from B[1][0]

Reduce for (0,0): A_dict={0:1, 1:2}, B_dict={0:5, 1:7}
Result = 1×5 + 2×7 = 19 ✓

5️⃣ Example 3 — SQL Join via MapReduce

Implement: SELECT C.name, O.amount FROM Customers C JOIN Orders O ON C.id = O.cust_id

Approach (Reduce-Side Join)

Both tables emit their records keyed on the JOIN attribute. Reducer combines them.

Map Function

// Tag each record with source for row in Customers (id, name): emit(id, ("C", name)) for row in Orders (cust_id, amount): emit(cust_id, ("O", amount))

Reduce Function

reduce(cust_id, values): customer_names = [] order_amounts = [] for (tag, val) in values: if tag == "C": customer_names.append(val) else: order_amounts.append(val) // Cross product (one customer × all their orders) for name in customer_names: for amount in order_amounts: emit(cust_id, (name, amount))

6️⃣ MapReduce Design Patterns

Beyond word count, many problems follow recognizable MR patterns:

🔢 Summarization

Aggregating values: counts, averages, sums per key.
Examples: Word count, sales totals.

🔍 Filtering

Pull subset of data: top-K, bottom-K, deduplication.
Examples: Top 10 customers, find outliers.

🔗 Joins

Combine 2+ datasets on a common key.
Examples: Customer + Orders, SQL-style joins.

📊 Organization

Reorganize data for downstream use.
Examples: Sort, bin/bucket, partition.

📈 Index/Inverted Index

Build search indexes.
Example: For each word → list of documents containing it.

🧬 Linear/Matrix Math

Linear algebra operations.
Examples: Matrix multiplication, PageRank.

7️⃣ Important Considerations

Combiner — When to Use

  • Sum, min, max, count → ✅ safe to use combiner
  • Average → ❌ NOT safe (combiner would compute partial averages incorrectly)
  • Distinct count → ⚠️ tricky (need approximate algorithms like HyperLogLog)

Partitioner — When Custom?

  • To handle data skew (some keys have way more values than others)
  • To group related keys for downstream processing
  • To implement secondary sort

Performance Tuning

📊 Use Combiners

Cut down shuffle data

🎯 Right Number of Reducers

Too few = bottleneck; too many = overhead

📁 Input Format

Use SequenceFile or compressed formats for large data

📦 Block Size

Match HDFS block size to optimize splits

📝 Summary & Exam Tips

🎯 Key Takeaways

  • Word Count: emit (word, 1) → sum in Reducer
  • Matrix Multiplication: key = result position (i,j); tag values "A" or "B" with index k
  • SQL Join: emit on join key, tag with source table, combine in Reducer
  • Combiner safe for: sum, min, max, count (associative + commutative)
  • Combiner NOT safe for: average (use sum + count separately)
  • Design patterns: summarization, filtering, joins, organization, index, math
