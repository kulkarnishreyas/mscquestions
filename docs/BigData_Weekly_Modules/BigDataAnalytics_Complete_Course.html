2. HDFS Architecture — Master/Slave

HDFS uses a master-slave architecture. The NameNode stores metadata; DataNodes store the actual blocks.

👑 NameNode
Master • Stores metadata (filenames, permissions, block locations) • Does NOT store data
🔧 DataNode 1
Stores actual blocks • Serves R/W requests • Reports to NameNode
🔧 DataNode 2
Stores replicas • Block creation/deletion under NameNode commands
🔧 DataNode 3
3rd replica (default replication factor = 3)

3. Blocks & Replication

  • Default block size: 128 MB (or 256 MB)
  • Default replication factor: 3
  • Replication ensures: if 1 DataNode fails, data still available on 2 others

4. HDFS Write Operation (6 steps)

  1. Client → write request → NameNode
  2. NameNode checks if file exists & permissions
  3. NameNode → list of DataNodes → Client
  4. Client streams data to first DataNode → pipelined to other DataNodes
  5. DataNodes send ACKs back
  6. NameNode updates metadata

5. HDFS Read Operation (4 steps)

  1. Client → read request → NameNode
  2. NameNode returns DataNode locations
  3. Client reads blocks directly from DataNodes (parallel, closest first)
  4. Client assembles blocks in order

2. Key Components

ComponentRole
MapperProcesses input → emits intermediate (key, value) pairs
Combiner (optional)Local reduce — reduces data transferred across network
PartitionerDecides which Reducer gets which key (default: hash function)
ReducerAggregates values for each key → final output

3. Classic vs YARN Framework

Classic MapReduce

  • JobTracker (master): job scheduling + resource management (single point of failure)
  • TaskTrackers: execute map/reduce tasks on workers

Limitations: Tight coupling, scalability issues, only batch processing.

YARN (Yet Another Resource Negotiator)

  • ResourceManager (master): cluster-wide resource scheduling
  • NodeManager (per node): manages local resources, launches containers
  • ApplicationMaster (per app): negotiates resources, manages app lifecycle
  • Container: unit of resource (CPU + memory)

Benefit: Decouples resource mgmt from processing → Spark, Tez, etc. can also run on Hadoop.

4. Word Count Example

Problem: Count frequency of each word in a text.

// Map: each word → (word, 1) "deer bear river" → ("deer",1), ("bear",1), ("river",1) "car car river" → ("car",1), ("car",1), ("river",1) // Shuffle/Sort groups same keys: deer → [1] bear → [1] river → [1,1] car → [1,1] // Reduce: sum values for each key deer → 1 bear → 1 river → 2 car → 2

5. Matrix Multiplication via MapReduce

  • Map: For each element of A & B, emit (result_position, [matrix, coord, value])
  • Reduce: For each result position, multiply corresponding elements and sum

Week 4 📊 MapReduce — Worked Examples

Visual MapReduce Flow

Example 1: Word Count (Deer/Bear/River)

Input: "Deer Bear River | Car Car River | Deer Car Bear" Split → Map → Shuffle → Reduce ================================= "Deer Bear River" → (Deer,1)(Bear,1)(River,1) "Car Car River" → (Car,1)(Car,1)(River,1) "Deer Car Bear" → (Deer,1)(Car,1)(Bear,1) After Shuffle: Bear → [1,1] Car → [1,1,1] Deer → [1,1] River → [1,1] Final Output: Bear=2, Car=3, Deer=2, River=2

Example 2: Matrix Multiplication

Compute C = A × B where C[i][j] = Σₖ A[i][k] × B[k][j]

// Map phase: for each A[i][k] and B[k][j] A[i][k] emits: ((i,j), ("A", k, value)) for all j B[k][j] emits: ((i,j), ("B", k, value)) for all i // Reduce phase: for each (i,j), sort by k, multiply pairs, sum C[i][j] = sum of A[i][k] × B[k][j]

Example 3: SQL Join via MapReduce

Joining two tables on a common key:

// Map: tag each record with source Customers row → emit (customer_id, ("C", name, ...)) Orders row → emit (customer_id, ("O", order_id, amount)) // Shuffle groups records by customer_id // Reduce: combine C and O records for each customer_id For each key, produce joined row by cross-multiplying C × O records

Week 5 📂 NoSQL & HBase

1. What is NoSQL?

"Not Only SQL" — flexible, schema-less databases for handling large volumes of varied (unstructured/semi-structured) data.

2. Four Types of NoSQL

📄 Document

JSON/BSON docs — MongoDB. Good for content mgmt, varying structures.

🔑 Key-Value

Simple key→value — Redis. Caching, sessions, fast lookups.

📊 Column-Oriented

Wide columns — HBase, Cassandra. Time-series, analytics on specific cols.

🕸️ Graph

Nodes & edges — Neo4j. Social networks, recommendations, fraud.

3. SQL vs NoSQL

FeatureSQL (Relational)NoSQL
Data ModelTables (rows/cols)Document/Key-Value/Column/Graph
SchemaPredefined, rigidDynamic, flexible
ScalingVertical (scale-up)Horizontal (scale-out)
ConsistencyStrong (ACID)Often tunable (CAP theorem)
Use CasesTransactions, complex queriesBig data, real-time apps

4. CAP Theorem

C — Consistency
A — Availability
P — Partition Tolerance

You can have only 2 of 3. NoSQL DBs typically choose AP, while SQL chooses CP.

5. HBase vs Hive

HBase

NoSQL database on HDFS. Real-time random R/W. Column-oriented. No SQL.

Hive

Data warehouse on Hadoop. SQL-like (HiveQL) for batch analytics. Translates to MapReduce/Tez/Spark.

6. HBase Architecture

👑 HMaster
Assigns regions to RegionServers, schema changes, load balancing
🦓 ZooKeeper
Cluster coordination, tracks active/failed nodes
🔧 RegionServer 1
Serves regions • MemStore (RAM) + BlockCache + HFile (HDFS) + WAL
🔧 RegionServer 2
Handles R/W requests for its regions

7. HBase Data Model

ConceptDescription
TableColumn-oriented collection of rows
RowKeyUnique identifier (primary index, lexicographic order)
Column FamilyGroup of related columns, defined at table creation

Cell(RowKey, Family:Qualifier) → value

5. MongoDB Replication — Replica Sets

👑 Primary
Handles ALL writes • Records to oplog
🔄 Secondary 1
Async replicates from primary's oplog • Can serve reads
🔄 Secondary 2
Replica copy • Vote in elections
💡 Failover: If primary fails (no heartbeat ~10-12s), eligible secondaries hold an election. A secondary with majority votes becomes the new primary. The old primary rejoins as a secondary when it recovers.

Week 9 🌊 Cassandra & Streaming Analytics

1. Streaming vs Batch Analytics

AspectStreamingBatch
DataIn motion (continuous)Static blocks
LatencyReal-time / near real-timeMinutes to hours
ToolsKafka, Flink, Spark StreamingHadoop MapReduce
Use CaseFraud detection, live dashboardsReports, ETL

2. Cassandra — Key Concepts

🌐 Peer-to-Peer

No master — every node identical. No single point of failure.

📈 Linear Scalability

Add nodes → throughput grows linearly

⚡ High Write Throughput

Optimized for fast writes — perfect for streaming

🛡️ Fault Tolerance

Data replicated across nodes

🎛️ Tunable Consistency

Per-query trade-off: consistency vs availability

⏰ Time-Series

Ideal for IoT, sensor data, logs

3. Cassandra Architecture (Ring)

4. Replication Factor

  • Set at keyspace level
  • RF = 3 means: 3 copies of every piece of data across nodes
  • If one node fails → data still readable from other replicas

5. Indexes

🔑 Primary Key

= Partition Key + Clustering Columns. Drives distribution & sorting.

📑 Secondary Index

Query non-key columns. Best for low-medium cardinality. Slow on big data!

4. Keyspaces & Tables

-- Create keyspace CREATE KEYSPACE my_app WITH REPLICATION = { 'class': 'SimpleStrategy', 'replication_factor': 3 }; -- Multi-datacenter CREATE KEYSPACE retail WITH REPLICATION = { 'class': 'NetworkTopologyStrategy', 'DC1': 3, 'DC2': 2 } AND DURABLE_WRITES = TRUE; -- Composite primary key (partition + clustering) CREATE TABLE temperature_readings ( device_id UUID, event_time TIMESTAMP, temperature FLOAT, PRIMARY KEY ((device_id), event_time) ) WITH CLUSTERING ORDER BY (event_time DESC);

5. Streaming Data Patterns

🌐 IoT Sensors

Time-series readings, RF=3, partition by device_id

📰 News Feed

User activities + popularity counters

💬 Messaging

Chat history, partition by conversation_id

📊 Monitoring

Application metrics & logs

6. Time-Series Best Practices

  • Use (device_id, event_time) as composite PK
  • Add WITH CLUSTERING ORDER BY (event_time DESC) for fast "recent N" queries
  • Use COUNTER columns for fast atomic counters (likes/views)
  • Apply TTL (Time To Live) to automatically delete old data

7. Counters Example (News Feed)

CREATE TABLE article_popularity ( article_id UUID, event_date TEXT, views COUNTER, likes COUNTER, shares COUNTER, PRIMARY KEY ((article_id, event_date)) ); -- Increment counters UPDATE article_popularity SET views = views + 1 WHERE article_id = ? AND event_date = '2024-05-15';
⚠️ Tombstones: Cassandra deletes are "soft" — they leave tombstone markers. Too many can slow queries. Use TTL or design data model to avoid excessive deletes.
