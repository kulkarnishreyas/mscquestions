1️⃣ Cassandra Collection Types

Cassandra collections store multiple values within a single column. They're useful for denormalizing data and grouping related items together — efficient for reads since the entire collection sits in one cell.

Why Use Collections in Streaming?

  • Streaming data often has multiple related attributes per entity (e.g., multiple sensor readings, tags for an article)
  • Collections store this data together — avoiding joins (Cassandra doesn't support them)
  • Faster reads — fewer round trips

Three Collection Types

📚 SET<type>

Unordered, unique elements

Good for: tags, categories, role memberships

Example: set<text>

📋 LIST<type>

Ordered, allows duplicates

Good for: activity logs, ordered preferences

Example: list<timestamp>

🗺️ MAP<K,V>

Key-value pairs, unique keys

Good for: settings, dynamic attributes

Example: map<text, int>

Quick Comparison

PropertySETLISTMAP
OrderNonePreservedNone
DuplicatesNoYesKeys unique
Indexed by—PositionKey
Use caseTagsLogsSettings
⚠️ Important constraints:
  • Collections should be small (≤ 64KB) for performance
  • Elements must be of primitive types (not nested collections)
  • Reading a collection always retrieves the WHOLE collection — no partial reads

2️⃣ CQL for Collections

Defining Columns with Collections

CREATE TABLE user_profiles ( user_id UUID PRIMARY KEY, emails SET<TEXT>, login_history LIST<TIMESTAMP>, settings MAP<TEXT, TEXT> );

Inserting Collections

-- SET INSERT INTO user_profiles (user_id, emails) VALUES (uuid(), {'a@example.com', 'b@example.com'}); -- LIST INSERT INTO user_profiles (user_id, login_history) VALUES (uuid(), ['2024-05-15 10:00:00', '2024-05-16 14:30:00']); -- MAP INSERT INTO user_profiles (user_id, settings) VALUES (uuid(), {'theme': 'dark', 'language': 'en'});

Querying Collections

-- Always retrieves the WHOLE collection SELECT emails FROM user_profiles WHERE user_id = ?; -- Result example: -- emails: {'a@example.com', 'b@example.com'}
⚠️ Filtering on collection elements has limitations and performance impact. Use sparingly or design your data model differently.

Updating Collections — SET

-- Add to set UPDATE user_profiles SET emails = emails + {'new@example.com'} WHERE user_id = ?; -- Remove from set UPDATE user_profiles SET emails = emails - {'old@example.com'} WHERE user_id = ?; -- Replace entire set UPDATE user_profiles SET emails = {'only@example.com'} WHERE user_id = ?;

Updating Collections — LIST

-- Append (most common for streaming) UPDATE user_profiles SET login_history = login_history + [toTimestamp(now())] WHERE user_id = ?; -- Prepend UPDATE user_profiles SET login_history = [toTimestamp(now())] + login_history WHERE user_id = ?; -- Remove element by value (removes ALL occurrences) UPDATE user_profiles SET login_history = login_history - ['2024-05-15 10:00:00'] WHERE user_id = ?; -- Update element by index (use cautiously) UPDATE user_profiles SET login_history[0] = toTimestamp(now()) WHERE user_id = ?;

Updating Collections — MAP

-- Add or update map entry UPDATE user_profiles SET settings['notifications'] = 'sms' WHERE user_id = ?; -- Remove map entry DELETE settings['theme'] FROM user_profiles WHERE user_id = ?; -- OR UPDATE user_profiles SET settings = settings - {'theme'} WHERE user_id = ?;

3️⃣ CSV Import / Export with COPY

Cassandra provides the COPY command for bulk CSV import/export — essential for ETL and migrating data.

Why CSV?

  • Universal interchange format
  • Easy to share across tools
  • Most analytical tools (Excel, Pandas) support CSV natively
  • Fast bulk loading via COPY FROM

Importing CSV — COPY FROM

-- Assume: CREATE TABLE sensor_data ( -- device_id UUID, event_time TIMESTAMP, temperature FLOAT, -- PRIMARY KEY (device_id, event_time) -- ); -- Assume: sensor_readings.csv -- device_id,event_time,temperature -- 123e4567-e89b-12d3-a456-426614174000,2024-05-15T10:00:00Z,25.5 -- 123e4567-e89b-12d3-a456-426614174001,2024-05-15T10:01:00Z,26.1 COPY sensor_data (device_id, event_time, temperature) FROM 'sensor_readings.csv' WITH HEADER = TRUE;

Exporting CSV — COPY TO

-- Export whole table COPY sensor_data TO 'all_sensor_data.csv' WITH HEADER = TRUE; -- Export with filter COPY (SELECT device_id, temperature FROM sensor_data WHERE event_time > '2024-05-15T00:00:00Z') TO 'recent_temps.csv' WITH HEADER = TRUE AND DELIMITER = '|';

COPY Options

OptionPurpose
HEADER = TRUEInclude/expect column headers
DELIMITER = ','Field separator (default comma)
QUOTE = '"'Quote character
NULL = 'NULL'String representing NULL
CHUNKSIZE = 1000Rows per chunk (default 5000)
NUMPROCESSES = 4Parallel processes for import/export
💡 For huge imports: Use sstableloader instead of COPY FROM — it bypasses CQL and loads data directly into the storage format, much faster.

4️⃣ Creating & Altering Keyspaces / Tables

Keyspaces (= Databases)

A keyspace is the top-level container for tables, similar to a database in SQL.

Creating a Keyspace

-- Simple (single data center) CREATE KEYSPACE my_app_keyspace WITH REPLICATION = { 'class': 'SimpleStrategy', 'replication_factor': 3 }; -- Multi-data center CREATE KEYSPACE retail_analytics WITH REPLICATION = { 'class': 'NetworkTopologyStrategy', 'DC1': 3, 'DC2': 2 } AND DURABLE_WRITES = TRUE; -- DURABLE_WRITES = TRUE (default) → use commit log -- DURABLE_WRITES = FALSE → skip commit log (faster but risky)

Other Keyspace Operations

-- List keyspaces DESCRIBE KEYSPACES; -- Switch to keyspace USE my_app_keyspace; -- Show current keyspace DESCRIBE KEYSPACE; -- Alter replication strategy ALTER KEYSPACE my_app_keyspace WITH REPLICATION = { 'class': 'SimpleStrategy', 'replication_factor': 5 }; -- Drop (irreversible!) DROP KEYSPACE my_app_keyspace;

Tables — Simple Primary Key

CREATE TABLE users ( user_id UUID, username TEXT, email TEXT, registration_date TIMESTAMP, PRIMARY KEY (user_id) -- user_id is partition key );

Tables — Composite Primary Key

CREATE TABLE temperature_readings ( device_id UUID, event_time TIMESTAMP, temperature FLOAT, location TEXT, PRIMARY KEY ((device_id), event_time) ) WITH CLUSTERING ORDER BY (event_time DESC); -- PRIMARY KEY ((device_id), event_time): -- device_id = partition key (which node) -- event_time = clustering column (sort order within partition) -- DESC ordering → newest readings first

Composite Partition Key

CREATE TABLE events_by_month ( year INT, month INT, event_time TIMESTAMP, event_data TEXT, PRIMARY KEY ((year, month), event_time) ) WITH CLUSTERING ORDER BY (event_time DESC); -- Partition key = (year, month) → data grouped per month -- Useful: avoid huge partitions for time-series

Altering Tables

-- Add column ALTER TABLE users ADD age INT; -- Drop column ALTER TABLE users DROP age; -- Change column type (limited; new type must be compatible) ALTER TABLE users ALTER email TYPE TEXT; -- Modify table options ALTER TABLE users WITH comment = 'Active users'; -- Drop table DROP TABLE users; -- Truncate (remove all data, keep schema) TRUNCATE users;

5️⃣ Reading, Modifying & Deleting Data

SELECT — Reading Data

-- All from table (use LIMIT for safety) SELECT * FROM users LIMIT 100; -- Specific columns SELECT username, email FROM users; -- WHERE on primary key (efficient) SELECT * FROM users WHERE user_id = ?; -- Range query on clustering column (within a partition) SELECT * FROM temperature_readings WHERE device_id = ? AND event_time > '2024-05-15T00:00:00Z' AND event_time < '2024-05-16T00:00:00Z'; -- Limit + sort (sort works if clustering order matches) SELECT * FROM temperature_readings WHERE device_id = ? ORDER BY event_time DESC LIMIT 10; -- COUNT (slow on large data) SELECT COUNT(*) FROM users;

ALLOW FILTERING — Use Carefully

-- Cassandra refuses inefficient queries by default -- ALLOW FILTERING forces the query but does a full scan SELECT * FROM users WHERE email = 'alice@example.com' ALLOW FILTERING;
⚠️ Avoid ALLOW FILTERING in production! It scans every node — fine for small data, terrible at scale. Instead: model your data with a secondary table partitioned by email.

INSERT & UPDATE — "Upsert"

In Cassandra, INSERT and UPDATE are interchangeable:

  • If row with primary key exists, INSERT acts like UPDATE
  • If row does not exist, UPDATE acts like INSERT
-- INSERT (acts as upsert) INSERT INTO users (user_id, username, email) VALUES (uuid(), 'newuser', 'new@example.com'); -- INSERT with TTL (auto-delete after 1 hour) INSERT INTO sessions (id, token) VALUES (uuid(), 'abc123') USING TTL 3600; -- UPDATE UPDATE users SET email = 'updated@example.com' WHERE user_id = ?; -- UPDATE multiple columns UPDATE users SET email = ?, last_login = toTimestamp(now()) WHERE user_id = ?; -- UPDATE with IF (lightweight transaction) UPDATE users SET email = 'new@example.com' WHERE user_id = ? IF email = 'old@example.com';

DELETE — Various Forms

-- Delete specific row DELETE FROM users WHERE user_id = ?; -- Delete specific column DELETE email FROM users WHERE user_id = ?; -- Delete with clustering filter (range) DELETE FROM temperature_readings WHERE device_id = ? AND event_time < '2024-01-01T00:00:00Z'; -- Delete from collection (covered earlier) UPDATE user_profiles SET emails = emails - {'old@example.com'} WHERE user_id = ?; -- Truncate (all data, keep schema) TRUNCATE users;

Lightweight Transactions (LWT)

Cassandra supports compare-and-set via IF clauses:

-- INSERT only if not exists INSERT INTO users (user_id, username) VALUES (?, 'alice') IF NOT EXISTS; -- UPDATE only if condition matches UPDATE accounts SET balance = balance - 100 WHERE id = ? IF balance >= 100;
⚠️ LWTs are SLOW — use Paxos for coordination. Don't use them for high-throughput operations. Reserve for true atomicity needs (e.g., unique constraint enforcement).

6️⃣ Example 1 — IoT Sensor Data

Problem

Ingest temperature readings from thousands of sensors, updating every few seconds. Queries needed:

  • Recent readings for a specific device
  • Temperature for a device over a time range
  • Latest reading per device

Schema Design

CREATE TABLE sensor_readings ( sensor_id UUID, reading_time TIMESTAMP, temperature DOUBLE, humidity DOUBLE, location TEXT, PRIMARY KEY ((sensor_id), reading_time) ) WITH CLUSTERING ORDER BY (reading_time DESC);

Why This Design?

📍 Partition by sensor_id

All readings for one sensor end up on the same node — fast queries per device

⏰ Cluster by reading_time DESC

Newest readings stored first — efficient "latest N readings" queries

🚀 Fast Writes

Streaming data → append-only writes, perfect for Cassandra's LSM design

Common Queries

-- Latest 10 readings for a sensor SELECT * FROM sensor_readings WHERE sensor_id = ? LIMIT 10; -- Readings in last hour SELECT * FROM sensor_readings WHERE sensor_id = ? AND reading_time > toTimestamp(now()) - 3600000; -- Insert new reading INSERT INTO sensor_readings (sensor_id, reading_time, temperature, humidity, location) VALUES (?, toTimestamp(now()), 25.5, 60.0, 'Room A');

Watch Out For — Wide Partitions

⚠️ Partition size limit: A single partition shouldn't exceed ~100 MB. If a sensor generates millions of readings, the partition grows huge!

Fix: Bucket by Time

CREATE TABLE sensor_readings_bucketed ( sensor_id UUID, bucket_date DATE, -- e.g., '2024-05-15' reading_time TIMESTAMP, temperature DOUBLE, PRIMARY KEY ((sensor_id, bucket_date), reading_time) ) WITH CLUSTERING ORDER BY (reading_time DESC); -- Now partition = (sensor_id, day) — bounded size

7️⃣ Example 2 — User Activity for a News Feed

Problem

Track user interactions (likes, views, shares) on news articles for:

  • Personalize each user's feed
  • Identify trending content
  • High velocity — many writes per second

Table 1 — User Activity Stream

CREATE TABLE user_article_interactions ( user_id UUID, interaction_time TIMESTAMP, article_id UUID, interaction_type TEXT, -- 'view', 'like', 'share' PRIMARY KEY ((user_id), interaction_time) ) WITH CLUSTERING ORDER BY (interaction_time DESC); -- Use case: "What did user X do recently?" SELECT * FROM user_article_interactions WHERE user_id = ? LIMIT 20;

Table 2 — Article Popularity Counters

CREATE TABLE article_popularity_counters ( article_id UUID, event_date TEXT, -- e.g., '2024-05-15' views COUNTER, likes COUNTER, shares COUNTER, PRIMARY KEY ((article_id, event_date)) ); -- Increment counters atomically UPDATE article_popularity_counters SET views = views + 1 WHERE article_id = ? AND event_date = '2024-05-15'; UPDATE article_popularity_counters SET likes = likes + 1 WHERE article_id = ? AND event_date = '2024-05-15';

Why Two Tables?

  • Denormalization is standard in Cassandra — model per query pattern
  • Table 1 answers "what did user X do?" (per-user view)
  • Table 2 answers "what's trending?" (per-article counters)
  • Each table optimized for its query — no joins needed

Trending Articles Query

-- Get top articles for today SELECT article_id, views FROM article_popularity_counters WHERE event_date = '2024-05-15' ALLOW FILTERING; -- Better approach: use a secondary table indexed by date CREATE TABLE trending_by_date ( event_date TEXT, views COUNTER, article_id UUID, PRIMARY KEY ((event_date), views, article_id) ) WITH CLUSTERING ORDER BY (views DESC);

8️⃣ Tombstones & TTL

What are Tombstones?

Cassandra deletes are "soft" — they don't immediately remove data. Instead, a tombstone marker is written saying "this data is deleted."

Why Tombstones?

  • Cassandra is distributed; can't guarantee all replicas got the delete
  • Tombstone propagates via gossip/repair
  • Actual data removed during compaction

Tombstone Lifecycle

DELETE
→
🪦 Tombstone written
propagate
All replicas marked
after gc_grace_seconds (default 10 days)
Compaction removes

Tombstone Problems

⚠️ Too many tombstones slow down reads:
  • Read queries must process tombstones
  • If > 1000 tombstones in a query → warning
  • If > 100,000 → query fails

Avoiding Tombstone Issues

  • Use TTL instead of explicit deletes for expiring data
  • Avoid inserting NULL values
  • Avoid deleting entire collections — replace with new content instead
  • Use time-bucketed partitions so old data ages out cleanly

TTL — Time To Live

TTL automatically deletes data after a specified time (in seconds):

-- INSERT with TTL of 1 hour INSERT INTO sessions (id, token) VALUES (uuid(), 'abc123') USING TTL 3600; -- UPDATE with TTL (also resets TTL on existing column) UPDATE users USING TTL 86400 SET status = 'temporary' WHERE user_id = ?; -- Check remaining TTL on a column SELECT TTL(status) FROM users WHERE user_id = ?; -- Set table default TTL CREATE TABLE temp_data ( id UUID PRIMARY KEY, data TEXT ) WITH default_time_to_live = 604800; -- 7 days

TTL Use Cases

🔐 Sessions

Auto-expire user sessions after timeout

📊 Recent Activity

Keep only last 30 days

🔄 Caches

Auto-invalidate stale data

📅 Promotions

Auto-remove after end date

💡 Why is TTL better than DELETE? TTL becomes a tombstone too, BUT it's predictable — Cassandra knows the expiry time. Schedulers can optimize compaction around it. Explicit DELETEs can pile up unexpectedly.

📝 Summary & Exam Tips

🎯 Key Takeaways

  • 3 collection types: SET (unique, unordered), LIST (ordered, dups), MAP (key-value)
  • COPY FROM / TO for CSV bulk import/export
  • Keyspace = top-level container (like database); RF set here
  • SimpleStrategy for single DC; NetworkTopologyStrategy for multi-DC
  • Composite primary key: PRIMARY KEY ((partition_key), clustering_col)
  • INSERT and UPDATE are interchangeable (upsert)
  • COUNTER type for atomic counters (likes, views) — only UPDATE, no INSERT
  • Tombstones: soft delete markers; removed via compaction after gc_grace_seconds
  • TTL: auto-expire data — better than explicit DELETE for time-bound data
  • Time-series: partition by entity, cluster by time DESC; bucket to avoid wide partitions
  • Denormalize: create separate tables per query pattern (no joins!)
📌 Common Exam Pitfalls:
  • Using ALLOW FILTERING in production (terrible at scale)
  • Trying to mix counter and regular columns in same table
  • Using explicit DELETE when TTL is better
  • Forgetting that collections must be small (~64KB) and primitive
  • Designing one big table — Cassandra wants one table per query pattern
