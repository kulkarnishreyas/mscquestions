1. Introduction to RMI

Definition: Java RMI (Remote Method Invocation) is a Java API that allows an object running in one Java Virtual Machine (JVM) to invoke methods on an object running in another JVM — on the same machine or across a network.

Before RMI, distributed computing in Java required developers to manually handle low-level socket programming, serialization, and protocol design. RMI abstracts all of that away, making remote calls look almost identical to local method calls.

Core Idea

Machine A (Client)

Has a stub — a proxy object that looks like the real remote object. When a method is called on the stub, it secretly packages the call and sends it over the network.

Machine B (Server)

Has the real object. A skeleton receives the packaged call, unpacks it, invokes the actual method, and sends results back.

Key Facts

  • Package: java.rmi
  • The remote server object must be publicly accessible
  • Used for client-server communication in distributed Java applications
  • Parameters and return values are transferred via serialization (marshalling)
  • Default port: 1099 (RMI Registry)

RMI in Context

🖥️

Local Call

Object A calls method on Object B in same JVM. Instantaneous, no network.

🌐

RMI Call

Object A calls method on Object B in different JVM. Transparent over network.

🔌

Socket Call

Raw byte streams. Developer manages all protocol details manually.

2. RMI Architecture

RMI is built on four distinct layers. Each layer has a well-defined responsibility, allowing the system to be modular and replaceable.

Layer Explanations

Application Layer

The actual client and server programs written by developers. Client calls methods; server implements them.

Stub / Skeleton

Proxy objects. Stub is on the client, Skeleton on the server. They handle marshalling/unmarshalling.

Remote Reference

Manages the lifecycle of remote object references. Decides how calls are delivered (one-to-one, one-to-many).

Transport Layer

Raw TCP/IP socket management. Sets up connections, routes packets, handles failures at the network level.

3. Stub (Client Side)

A Stub is a client-side proxy/representative of the remote object. From the client's perspective, it looks and behaves exactly like the real object — but under the hood it forwards every call over the network.

When the client calls a method on the stub, the following sequence occurs:

1
Initiate Connection — The stub opens (or reuses) a TCP connection to the remote JVM where the real object lives.
2
Marshal Parameters — Arguments are serialized into a byte stream (marshalling). Primitive types are encoded directly; objects must implement Serializable.
3
Send & Wait — The marshalled data is sent to the skeleton. The stub thread blocks, waiting for a response.
4
Unmarshal Return Value — When the skeleton sends back the result, the stub deserializes the byte stream back into a Java object (unmarshalling).
5
Return to Caller — The deserialized return value (or exception) is handed back to the client code as if it came from a local method.
Key insight: The stub implements the same Remote Interface as the real object. This is how the client code remains completely unaware that the object is remote.

4. Skeleton (Server Side)

A Skeleton is the server-side counterpart to the stub. It acts as a gateway into the real remote object — receiving calls from the stub, dispatching them to the actual implementation, and sending results back.

The skeleton's responsibilities in order:

1
Receive Incoming Data — The skeleton listens on a TCP port for incoming marshalled calls from any connected stub.
2
Unmarshal Parameters — The byte stream is deserialized back into Java types (integers, strings, objects, etc.).
3
Invoke Actual Method — The skeleton calls the real method on the server-side implementation object with the deserialized parameters.
4
Marshal Result or Exception — The return value (or any thrown exception) is serialized into bytes.
5
Send Back to Stub — The serialized response is transmitted back to the client's stub over the same TCP connection.

In Java 1.1

Skeletons were generated as separate _Skel.class files by the rmic compiler. You had to run rmic MyImpl explicitly.

Java 1.2+ (Modern RMI)

Skeleton logic was moved into the RMI runtime itself via reflection. No need to run rmic — skeleton behavior is automatic.

5. RMI Registry

The RMI Registry is a simple name server — a phone book for remote objects. Before a client can call a remote object, it must first find it. The registry enables this discovery.

How the Registry Works

1
Server starts up and creates (or locates) the registry on port 1099.
2
Server registers its remote object: registry.rebind("Calculator", new CalculatorImpl())
3
Client looks up the object by name: Naming.lookup("//localhost/Calculator")
4
Registry returns a stub reference to the client.
5
Client calls methods directly on the stub — the registry is no longer involved.
Important: The registry acts as a one-time broker. Once the client has the stub, all subsequent method calls go directly from client stub to server skeleton — bypassing the registry entirely.

6. The Naming Class

The java.rmi.Naming class provides static methods to interact with the RMI registry using URL-style names.

Method Signature Description
bind() bind(String name, Remote obj) Binds a name to a remote object. Throws AlreadyBoundException if the name is taken.
rebind() rebind(String name, Remote obj) Binds or replaces an existing binding. Safe to call even if name is already bound.
lookup() lookup(String name) : Remote Returns the stub for a named remote object. Used by clients.
unbind() unbind(String name) Removes a name-to-object binding from the registry.
list() list(String host) : String[] Returns an array of all names currently registered in the registry.

URL Format

rmi://hostname:port/objectName

// Examples:
"//localhost/Calculator" // port defaults to 1099
"//192.168.1.10:1099/Hi" // explicit IP and port
bind() vs rebind(): Always prefer rebind() on the server. If you restart your server and use bind(), it throws AlreadyBoundException because the old binding still exists. rebind() simply replaces the old one.

7. Four Required Programs

Every RMI application consists of exactly four components. Understanding the role of each is essential before writing any code.

1

Remote Interface

Declares the methods that can be called remotely. Must extend java.rmi.Remote. Every declared method must throw RemoteException. This interface is shared between client and server.

2

Implementation Class

The actual server-side object. Must extend UnicastRemoteObject and implement the remote interface. Contains the real business logic. Constructor must throw RemoteException.

3

Server Program

Creates (or locates) the RMI registry using LocateRegistry.createRegistry(1099), instantiates the implementation object, and registers it with rebind(). Must run before any client.

4

Client Program

Calls Naming.lookup() to get a stub reference, casts it to the remote interface type, then calls methods as if the object were local.

Compilation & Run Order

# 1. Compile all Java files
javac *.java
# 2. Start the RMI registry (separate terminal)
rmiregistry &
# 3. Start the server
java CalServer
# 4. Run the client (separate terminal)
java CalClient1

import java.rmi.Remote;

    public String sayHi() throws RemoteException;

}
Output:
Compiled successfully. (Interface — no runtime output)
Rules for Remote Interface:
  • Must extend java.rmi.Remote (marker interface)
  • Every method must declare throws RemoteException
  • This interface is the contract shared by client and server
JAVA 
import java.rmi.RemoteException;
import java.rmi.server.UnicastRemoteObject;

// Step 2: Implement the remote interface
public class Hi extends UnicastRemoteObject
              implements HiInterface {

    private String message;

    // Constructor MUST throw RemoteException
    public Hi(String msg) throws RemoteException {
        super();          // exports this object on an anonymous port
        message = msg;
    }

    @Override
    public String sayHi() throws RemoteException {
        return message;   // actual business logic
    }
}
Output:
Compiled successfully. (Implementation class — run via server)
UnicastRemoteObject: Extending this class exports the object (makes it reachable over the network) automatically in the constructor. It also handles deep cloning of parameters.
JAVA 
import java.rmi.registry.LocateRegistry;
import java.rmi.registry.Registry;

public class HiServer {
    public static void main(String[] args) {
        try {
            // Create and start the RMI registry on port 1099
            Registry r = LocateRegistry.createRegistry(1099);

            // Create the remote object with our message
            Hi hiObj = new Hi("I am from VIT, Chennai.");

            // Register it in the registry under the name "Hi"
            r.rebind("Hi", hiObj);

            System.out.println("Server is up");

        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}
Output:
Server is up
JAVA 
import java.rmi.Naming;

public class HiClient {
    public static void main(String[] args) {
        try {
            // Look up the remote object in the registry
            HiInterface hiObj =
                (HiInterface) Naming.lookup("//localhost/Hi");

            // Call the remote method — looks just like a local call!
            String result = hiObj.sayHi();

            System.out.println(result);

        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}
Output:
I am from VIT, Chennai.

public interface CalInterface extends Remote {

Output:

JAVA — CalculatorImpl.java 
import java.rmi.RemoteException;
import java.rmi.server.UnicastRemoteObject;

public class CalculatorImpl extends UnicastRemoteObject
                           implements CalInterface {

    // Constructor must throw RemoteException (required by UnicastRemoteObject)
    public CalculatorImpl() throws RemoteException {
        super();
    }

    @Override
    public int addition(int a, int b) throws RemoteException {
        return a + b;
    }

    @Override
    public int subtraction(int a, int b) throws RemoteException {
        return a - b;
    }

    @Override
    public int multiplication(int a, int b) throws RemoteException {
        return a * b;
    }

    @Override
    public int division(int a, int b) throws RemoteException {
        if (b == 0) throw new ArithmeticException("Division by zero");
        return a / b;
    }
}

import java.rmi.registry.LocateRegistry;
import java.rmi.registry.Registry;

public class CalServer {

            // Start RMI registry on port 1099

}

10. RMI vs Socket Programming

Both RMI and raw sockets enable distributed communication in Java. Understanding their trade-offs helps you choose the right tool.

Feature RMI Sockets
Abstraction Level High — method calls Low — byte streams
Object Passing Yes — objects serialized automatically No — manual serialization required
Developer Complexity Lower — protocol handled by RMI runtime Higher — developer writes all protocol logic
Performance Slightly slower (serialization overhead) Faster (raw bytes, minimal overhead)
Error Handling RemoteException hierarchy, automatic Manual IOException handling
Language Restriction Java-to-Java only Language agnostic
Security Built-in RMI security manager Developer must implement manually
Use Case Distributed Java enterprise apps High-performance, cross-language systems

When to Choose RMI

  • Pure Java distributed system
  • Need to pass complex Java objects
  • Rapid development is priority
  • Method-oriented API design
  • Enterprise application servers (EJB uses RMI)

When to Choose Sockets

  • Multi-language client-server system
  • Maximum performance required
  • Custom binary protocol needed
  • Embedded/constrained environments
  • Low-level network programming

11. How RMI Works — Detailed Walkthrough

Let's trace a complete RMI call from start to finish, following every byte and every object across the network.

Step-by-Step Explanation

1
Client calls method on stub — The client code holds a reference to the stub (obtained from the registry). Calling calObj.subtraction(20, 10) looks identical to a local call.
2
Stub marshals parameters — The stub intercepts the call. It serializes the method identifier and parameters (integers 20 and 10) into a byte array. This is marshalling.
3
Bytes travel over TCP/IP — The Remote Reference and Transport layers handle opening a TCP socket to the server machine and streaming the bytes. The network sees only raw bytes — it knows nothing about Java objects.
4
Skeleton unmarshals parameters — On the server side, the skeleton receives the bytes and deserializes them back into Java values: method name = "subtraction", a = 20, b = 10. This is unmarshalling.
5
Real method executes — The skeleton calls calculatorImpl.subtraction(20, 10). The actual business logic runs. Result = 10.
6
Return value marshalled and sent back — The skeleton serializes the integer 10 into bytes and sends it back to the stub via the TCP connection.
7
Stub unmarshals result — The stub deserializes the return bytes back into Java int value 10 and returns it to the client code. From the client's perspective, it just got 10 back from a method call.

What is Marshalling?

Marshalling is the process of converting Java objects and primitive values into a flat byte array suitable for transmission over a network. Unmarshalling is the reverse — reconstructing Java objects from received bytes. In Java RMI, java.io.ObjectOutputStream and java.io.ObjectInputStream handle this automatically. Objects must implement java.io.Serializable to be marshallable.

Marshalling (Serialization)

Java Object:
  int a = 20 → 0x00 0x00 0x00 0x14
  int b = 10 → 0x00 0x00 0x00 0x0A
  method = "subtraction"
→ bytes sent over TCP

Unmarshalling (Deserialization)

Received bytes:
  0x00 0x00 0x00 0x14 → int 20
  0x00 0x00 0x00 0x0A → int 10
  method identifier decoded
→ skeleton invokes method

Common Exceptions in RMI

ExceptionWhen it occurs
RemoteExceptionAny network/communication failure during a remote method call

AlreadyBoundExceptionServer calls bind() when that name is already registered
ConnectExceptionClient cannot reach the server (server not running, wrong host/port)

Q1 · 2 marks A client looks up a remote object using Naming.lookup("rmi://localhost/Calc") without specifying a port. On which default port does the RMI registry listen?
When no port is given, RMI uses its standard registry port.
Default RMI registry port = 1099 (started via rmiregistry or LocateRegistry.createRegistry(1099))
The answer is 1099.
Q2 · 2 marks A remote Calculator stub is invoked as shown. What value does the client receive in r?
Calculator c = (Calculator) Naming.lookup("rmi://localhost/Calc"); int r = c.add(45, 55);
The stub marshals the arguments, the server executes add, and the result is returned to the client.
add(45, 55) → 45 + 55 = 100
The client receives 100. (The remote call is transparent — it looks like a local method call.)
Q3 · 2 marks Using the same remote Calculator, what does this sequence return?
int a = c.multiply(12, 8); int b = c.power(2, 10); // computes 2^10 int result = b - a;
Each remote call returns its computed value; the subtraction happens on the client.
multiply(12, 8) = 96 · power(2, 10) = 2¹⁰ = 1024 · result = 1024 − 96 = 928
The final value is 928.
Q4 · 2 marks In RMI architecture, what is the role of the stub on the client side?
The stub is the client-side proxy: it marshals (serializes) the method name and arguments, ships them over the network, then unmarshals the returned value — making the remote call look local. The server-side counterpart (the skeleton, in older RMI) unmarshals the request and dispatches it to the real object. The answer is the client-side proxy that marshals/unmarshals.
Q5 · 2 marks Which pair of rules must a valid RMI remote interface satisfy?
A remote interface must extend java.rmi.Remote (a marker interface) and each method must declare throws RemoteException to account for network failures. The implementation class typically extends UnicastRemoteObject. The answer is extends Remote + throws RemoteException.
Q6 · 2 marks Which Naming method should a server use to register a remote object so that it replaces any existing binding under the same name (avoiding an AlreadyBoundException)?
bind() throws AlreadyBoundException if the name is taken, whereas rebind() binds the object and silently overwrites any existing binding — the safe choice for (re)starting a server. lookup() retrieves a stub and unbind() removes a binding. The answer is Naming.rebind(name, obj).
