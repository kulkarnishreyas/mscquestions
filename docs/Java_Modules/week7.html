1. Introduction to JDBC

JDBC (Java Database Connectivity) is a Java API that enables Java programs to connect to and execute queries against relational databases — regardless of the underlying database vendor.

What JDBC Provides

  • A standard interface for database connectivity
  • Execute SQL queries and updates
  • Retrieve and process query results
  • Handle database transactions
  • Call stored procedures

Key Principle: Write Once, Run Anywhere

The same Java application can connect to MySQL, Oracle, PostgreSQL, or SQLite — simply by swapping the JDBC driver. Business logic stays unchanged.

JDBC Driver: Provided by the database vendor. It translates generic JDBC calls into database-specific protocol calls.

JDBC in the Java Ecosystem

java.sql package

Core JDBC interfaces: Connection, Statement, ResultSet, PreparedStatement

javax.sql package

Enterprise extensions: DataSource, connection pooling, row sets, distributed transactions

JDBC Drivers

Third-party JARs: mysql-connector-java, ojdbc, postgresql, sqlite-jdbc

2. How JDBC Works

JDBC operates through four distinct aspects. Each aspect is handled by a different API object, and they work together in a pipeline:

🔌

Driver

Translates JDBC calls to DB-specific protocol

🔗

Connection

Active session with the database server

📝

SQL Execution

Statement / PreparedStatement to run queries

📊

Results

ResultSet — cursor over returned rows

Basic JDBC Structure

Java — BasicJDBC.java 
import java.sql.*;

public class BasicJDBC {
    public static void main(String[] args) throws Exception {

        // Step 1: Load the JDBC driver
        Class.forName("com.mysql.cj.jdbc.Driver");

        // Step 2: Establish a connection
        Connection con = DriverManager.getConnection(
            "jdbc:mysql://localhost:3306/mydb",
            "root",
            "password"
        );

        // Step 3: Create a Statement
        Statement stmt = con.createStatement();

        // Step 4: Execute a query
        ResultSet rs = stmt.executeQuery("SELECT id, name FROM students");

        // Step 5: Process the results
        while (rs.next()) {
            int    id   = rs.getInt("id");
            String name = rs.getString("name");
            System.out.println(id + " → " + name);
        }

        // Step 6: Cleanup — always close in reverse order
        rs.close();
        stmt.close();
        con.close();
    }
}
1 → Alice 2 → Bob 3 → Charlie Connection closed successfully.
Note on Class.forName(): In modern JDBC (4.0+, Java 6+), the driver is loaded automatically via the Service Provider mechanism. You rarely need to call Class.forName() explicitly — but it is still commonly taught and used for clarity.

3. 2-Tier vs 3-Tier Architecture

2-Tier Architecture

The Java application communicates directly with the database using JDBC. Suitable for small desktop applications.

  • Simple to set up
  • Direct connection — low latency
  • Limited scalability
  • DB credentials embedded in client

3-Tier Architecture

A middle-tier application server sits between the client and the database. Standard for enterprise web applications.

  • High scalability — handle many clients
  • Security — DB never directly exposed
  • Load balancing at app server tier
  • Easier maintainability and updates

Comparison Table

4. JDBC API Components

The JDBC API is organised around several core interfaces and classes, each responsible for a distinct part of database interaction.

DriverManager

java.sql.DriverManager

The factory that creates Connection objects. It maintains a list of registered JDBC drivers and tries each in order when a connection URL is requested.

DriverManager.getConnection(url, user, pass)

Driver

Vendor-Implemented

Each database vendor provides a Driver implementation that handles the actual network communication in the database's native protocol. Registers itself with DriverManager at class load time.

Connection

java.sql.Connection

Represents an active session with a specific database. Responsibilities:

  • Factory for Statement, PreparedStatement, CallableStatement
  • Transaction management: commit(), rollback()
  • Set auto-commit mode with setAutoCommit(boolean)
  • Must be closed after use

Statement (3 kinds)

Statement PreparedStatement CallableStatement

Statement: For simple, one-off SQL. No parameters. Slower for repeated execution.

PreparedStatement: Pre-compiled SQL with ? placeholders. Faster for repeated calls. Prevents SQL injection.

CallableStatement: Extends PreparedStatement. Used to call stored procedures in the database.

ResultSet

java.sql.ResultSet

An iterator over the rows returned by a SELECT query. A cursor points to the current row; call next() to advance it.

MethodReturns
getString("col")String value
getInt("col")int value
getDouble("col")double value
getBoolean("col")boolean value
getDate("col")java.sql.Date

SQLException

java.sql.SQLException

Thrown when a database access error occurs. Key methods:

  • getErrorCode() — vendor-specific error code
  • getSQLState() — standard SQL state string
  • getMessage() — human-readable message
  • getNextException() — chain of exceptions

DataSource

javax.sql.DataSource

The preferred alternative to DriverManager in enterprise apps. Provides connection pooling — reuses connections instead of creating new ones each time, dramatically improving performance.

5. The Six Steps of JDBC

Every JDBC program follows the same six fundamental steps. Master these and you understand JDBC.

1

Load the Driver

Class.forName("com.mysql.cj.jdbc.Driver"); — Dynamically loads the driver class, which registers itself with DriverManager. In JDBC 4.0+, drivers in the classpath auto-register, but explicit loading is common for portability.

2

Establish a Connection

Connection con = DriverManager.getConnection(url, user, password); — Opens a live TCP connection to the database server. The URL encodes the database type, host, port, and database name.

3

Create a Statement

Statement stmt = con.createStatement(); — Creates an object that will send SQL to the database. Use prepareStatement(sql) for parameterised queries.

4

Execute the SQL

Two main methods: executeQuery(sql) for SELECT (returns ResultSet) and executeUpdate(sql) for INSERT/UPDATE/DELETE (returns row count). execute(sql) handles both.

5

Process the Results

Iterate over the ResultSet with while (rs.next()) { ... }. Extract column values using typed getters: getString(), getInt(), getDouble(), etc.

6

Cleanup — Close Resources

Always close in reverse order: rs.close() → stmt.close() → con.close(). Use try-with-resources (Java 7+) for automatic closing. Failing to close connections causes resource leaks and exhaustion of the connection pool.

Java — SixSteps.java (try-with-resources) 
import java.sql.*;

public class SixSteps {
    static final String URL  = "jdbc:mysql://localhost:3306/school";
    static final String USER = "root";
    static final String PASS = "secret";

    public static void main(String[] args) {
        // Step 1 — (JDBC 4+ auto-loads; shown for completeness)
        try {
            Class.forName("com.mysql.cj.jdbc.Driver");
        } catch (ClassNotFoundException e) {
            System.err.println("Driver not found: " + e.getMessage());
            return;
        }

        // Steps 2-6 inside try-with-resources (auto-close)
        try (Connection con  = DriverManager.getConnection(URL, USER, PASS);  // Step 2
             Statement  stmt = con.createStatement();                         // Step 3
             ResultSet  rs   = stmt.executeQuery("SELECT * FROM students")) {   // Step 4

            while (rs.next()) {                                               // Step 5
                System.out.printf("%-5d %-20s %d%n",
                    rs.getInt("id"),
                    rs.getString("name"),
                    rs.getInt("age"));
            }
            // Step 6 — auto-close handled by try-with-resources

        } catch (SQLException e) {
            System.err.println("SQL Error [" + e.getSQLState() + "]: " + e.getMessage());
        }
    }
}
1 Alice 18 2 Bob 20 3 Charlie 19 4 Diana 21 All resources closed automatically by try-with-resources.

7. PreparedStatement — INSERT

A PreparedStatement is a pre-compiled SQL template with ? placeholders. Parameters are bound with typed setters indexed from 1.

Why PreparedStatement over Statement?
  • SQL is compiled once, executed many times — faster for batch operations
  • Automatically escapes special characters — prevents SQL injection
  • Cleaner code — no string concatenation with user data
  • Supports binary data (BLOB) and Java objects natively

INSERT with PreparedStatement

Java — InsertStudent.java 
import java.sql.*;

public class InsertStudent {
    public static void main(String[] args) throws Exception {
        Connection conn = DriverManager.getConnection(
            "jdbc:mysql://localhost:3306/school", "root", "pass");

        // SQL template with ? placeholders
        String sql = "INSERT INTO Students (id, name, age) VALUES (?, ?, ?)";

        PreparedStatement pstmt = conn.prepareStatement(sql);

        // Bind parameters — index starts at 1
        pstmt.setInt   (1, 1001);           // parameter 1 → id
        pstmt.setString(2, "John");         // parameter 2 → name
        pstmt.setInt   (3, 18);             // parameter 3 → age

        int rowsAffected = pstmt.executeUpdate();
        System.out.println("Rows inserted: " + rowsAffected);

        // Insert another row — reuse the same PreparedStatement
        pstmt.setInt   (1, 1002);
        pstmt.setString(2, "Alice");
        pstmt.setInt   (3, 20);
        rowsAffected = pstmt.executeUpdate();
        System.out.println("Rows inserted: " + rowsAffected);

        pstmt.close();
        conn.close();
    }
}
Rows inserted: 1 Rows inserted: 1

Benefits of PreparedStatement Summary

Security

Parameters are escaped automatically. User input cannot break out of the parameter context — SQL injection is prevented at the API level.

Performance

The database parses and compiles the SQL once. Subsequent executions with different parameters reuse the compiled plan — faster for loops and batch operations.

Readability

No brittle string concatenation. The SQL template is clearly separate from its parameters, making code easier to read, debug, and maintain.

9. Retrieving Data

SELECT queries return a ResultSet — a cursor that starts before the first row. Call rs.next() to advance; it returns false when there are no more rows.

Java — RetrieveData.java 
import java.sql.*;

public class RetrieveData {
    public static void main(String[] args) throws Exception {
        Connection conn = DriverManager.getConnection(
            "jdbc:mysql://localhost:3306/school", "root", "pass");

        Statement stmt = conn.createStatement();

        // Execute SELECT — returns a ResultSet
        ResultSet rs = stmt.executeQuery("SELECT * FROM myTable");

        // Iterate — rs.next() advances cursor, returns false at end
        while (rs.next()) {
            String data = rs.getString("column1Name");
            System.out.println(data);
        }

        rs.close();
        stmt.close();
        conn.close();
    }
}
row_value_1 row_value_2 row_value_3

ResultSet Access Styles

By column name (recommended):
rs.getString("name")
rs.getInt("age")
Readable, resilient to column reordering.
By column index (1-based):
rs.getString(1)
rs.getInt(2)
Slightly faster, but fragile if schema changes.

11. LoginApp — JavaFX + JDBC

A real-world JavaFX login window that authenticates users against a database using a secure PreparedStatement query.

Login Window Mockup

The authenticate() Method

The key to secure authentication: use a PreparedStatement with ? parameters so that username and password values can never alter the SQL structure.

The query returns a ResultSet. If rs.next() returns true, a matching row was found — login succeeds.

Full LoginApp Source

Java — LoginApp.java (JavaFX + JDBC) 
import javafx.application.Application;
import javafx.geometry.Insets;
import javafx.scene.Scene;
import javafx.scene.control.*;
import javafx.scene.layout.*;
import javafx.stage.Stage;
import java.sql.*;

public class LoginApp extends Application {

    private static final String DB_URL  = "jdbc:mysql://localhost:3306/appdb";
    private static final String DB_USER = "root";
    private static final String DB_PASS = "secret";

    private TextField     usernameField;
    private PasswordField passwordField;
    private Label         statusLabel;

    // ── JavaFX UI setup ──────────────────────────────────────
    @Override
    public void start(Stage primaryStage) {
        VBox root = new VBox(12);
        root.setPadding(new Insets(30));
        root.setStyle("-fx-background-color: #1a1d2e;");

        Label title = new Label("User Login");
        title.setStyle("-fx-font-size:20; -fx-text-fill:#6c63ff; -fx-font-weight:bold;");

        usernameField = new TextField();
        usernameField.setPromptText("Username");

        passwordField = new PasswordField();
        passwordField.setPromptText("Password");

        Button loginBtn = new Button("Login");
        loginBtn.setOnAction(e -> handleLogin());

        statusLabel = new Label();
        statusLabel.setStyle("-fx-text-fill: #00d4aa;");

        root.getChildren().addAll(title,
            new Label("Username"), usernameField,
            new Label("Password"), passwordField,
            loginBtn, statusLabel);

        primaryStage.setScene(new Scene(root, 300, 250));
        primaryStage.setTitle("Login");
        primaryStage.show();
    }

    // ── Login button handler ──────────────────────────────────
    private void handleLogin() {
        String username = usernameField.getText().trim();
        String password = passwordField.getText();

        if (username.isEmpty() || password.isEmpty()) {
            statusLabel.setText("Please enter both fields.");
            statusLabel.setStyle("-fx-text-fill:#ff5555;");
            return;
        }

        if (authenticate(username, password)) {
            statusLabel.setText("Authentication successful! Welcome, " + username);
            statusLabel.setStyle("-fx-text-fill:#00d4aa;");
        } else {
            statusLabel.setText("Invalid username or password.");
            statusLabel.setStyle("-fx-text-fill:#ff5555;");
        }
    }

    // ── JDBC authentication with PreparedStatement ────────────
    private boolean authenticate(String username, String password) {
        // Safe: parameters are bound — SQL structure cannot be changed
        String sql = "SELECT id FROM users WHERE username = ? AND password = ?";

        try (Connection conn = DriverManager.getConnection(DB_URL, DB_USER, DB_PASS);
             PreparedStatement pstmt = conn.prepareStatement(sql)) {

            pstmt.setString(1, username); // bind username parameter
            pstmt.setString(2, password); // bind password parameter

            ResultSet rs = pstmt.executeQuery();
            return rs.next(); // true if a matching row exists

        } catch (SQLException e) {
            System.err.println("Auth error: " + e.getMessage());
            return false;
        }
    }

    public static void main(String[] args) {
        launch(args);
    }
}

authenticate() walkthrough:
  1. SQL template "SELECT id FROM users WHERE username = ? AND password = ?" is compiled once.
  2. setString(1, username) and setString(2, password) bind the values safely.

12. SQL Injection Prevention

SQL injection is one of the most critical web security vulnerabilities. It occurs when user-supplied input is concatenated directly into a SQL query, allowing attackers to alter the query's logic.

The Attack

Suppose a login form has username and password fields. An attacker enters:

Username:  admin' OR '1'='1
Password:  anything

If the code uses string concatenation, this transforms the query in a devastating way:

Bad — String Concatenation

Java — VULNERABLE code INSECURE 
// BAD: Never do this!
String username = request.getParameter("user");
String password = request.getParameter("pass");

String sql = "SELECT * FROM users WHERE username = '"
           + username + "' AND password = '"
           + password + "'";

Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery(sql);
// rs.next() returns true — attacker is logged in!
What the DB actually executes:
SELECT * FROM users WHERE username = 'admin' OR '1'='1' AND password = 'anything'

'1'='1' is always TRUE. The WHERE clause always matches, returning all users. The attacker gains access as the first user in the table — often the admin.

Good — PreparedStatement

Java — SECURE code SECURE 
// GOOD: Always use PreparedStatement
String username = request.getParameter("user");
String password = request.getParameter("pass");

PreparedStatement pstmt = conn.prepareStatement(
    "SELECT * FROM users WHERE username = ? AND password = ?"
);
pstmt.setString(1, username);
pstmt.setString(2, password);

ResultSet rs = pstmt.executeQuery();
// rs.next() returns false — attack failed!
Why this is safe:
The JDBC driver escapes the username before embedding it. The actual query sent becomes:
SELECT * FROM users WHERE username = 'admin'' OR ''1''=''1' AND password = 'anything'

The single quotes are escaped (doubled), so the entire attacker string is treated as a literal username that doesn't exist — login fails.

Other SQL Injection Payloads to Know

PayloadIntentEffect on vulnerable system
' OR '1'='1Authentication bypassAlways-true WHERE clause
'; DROP TABLE users; --DestructiveDeletes entire table
' UNION SELECT username, password FROM admin --Data exfiltrationLeaks credentials
' AND 1=0 UNION SELECT version() --ReconnaissanceReveals DB version

Defence Summary

Use PreparedStatement

Always use parameterised queries. Never concatenate user input into SQL strings. This is the primary defence.

Input Validation

Validate and whitelist user input on the server side. Reject unexpected characters. Use length limits on all input fields.

Least Privilege

The DB user used by the app should only have the minimum permissions needed (SELECT, INSERT) — never DROP or ALTER in production.

Q1 · 2 marks A students table has 10 rows; exactly 4 of them have grade = 'F'. The statement below runs successfully. What int does executeUpdate() return?
PreparedStatement ps = conn.prepareStatement("UPDATE students SET grade = ? WHERE grade = ?"); ps.setString(1, "P"); ps.setString(2, "F"); int n = ps.executeUpdate();

rows with grade = 'F' = 4 → returns 4
Answer: 4 (B). Option A is the trap of counting all rows.
Q2 · 2 marks You prepare the INSERT below. How many ? placeholders must you bind — i.e. how many setXxx(...) calls are required before executing?
String sql = "INSERT INTO employees " + "(id, name, dept, salary) VALUES (?, ?, ?, ?)"; PreparedStatement ps = conn.prepareStatement(sql);
One setXxx call per placeholder, indexed 1..N.
placeholders = 4 (id, name, dept, salary) → 4 setXxx calls
Answer: 4 (C).
Q3 · 2 marks A query returns a ResultSet containing 5 rows. The cursor starts before the first row. How many times does the body of the loop below execute?
while (rs.next()) { System.out.println(rs.getString("name")); }
rs.next() advances onto each row, returning true once per row and false after the last.
5 rows → next() true 5 times → body runs 5 times
Answer: 5 (B). Off-by-one answers (4 or 6) are the common errors.
Q4 · 2 marks Why is a PreparedStatement with bind parameters the recommended defence against SQL injection, compared with building a query by string concatenation in a Statement?
The SQL structure is fixed when the statement is prepared; bound values travel separately as data and are never re-parsed as SQL. So input like ' OR '1'='1 is treated as a literal string, not executable code. Answer: B.
Q5 · 2 marks A Java desktop program connects directly to the database using the JDBC driver, with no application/middle server between them. What architecture is this?
Client talking straight to the DB is two-tier (client + database). A three-tier design inserts a middle application/business layer (e.g. a server or app server) between client and database. Answer: A.
Q6 · 2 marks You have a default ResultSet obtained from executeQuery() and the cursor is on the 3rd row. You call rs.previous() to step back to row 2. What happens?
A default ResultSet is TYPE_FORWARD_ONLY — you can only move forward with next(). Calling previous() throws a SQLException. To scroll backward you must request TYPE_SCROLL_INSENSITIVE/SENSITIVE when creating the statement. Answer: B.
