🌐 1. Basics of Networking

A computer network is a system of computers and devices that can communicate with each other. Networking underpins everything from loading a webpage to sending an email to video calls.

Key Terms

Network

Computers/devices that communicate via electromagnetic waves, radio signals, or fibre-optic cables.

Node

Any computer or device connected to the network — laptops, phones, servers, routers, printers.

Address

A unique string that identifies each node on the network (e.g., 192.168.1.1 or a domain name).

Protocol

Guidelines defining format, packet structure, and addressing rules. Examples: IEEE 802.3, HTTP, FTP.

Packets

Data split into independently managed units, each carrying sender and recipient info for routing.

TCP/IP Four-Layer Model

The TCP/IP model organises networking into four distinct layers, each with specific responsibilities:

Encapsulation: Data travels down the layers when sending (each layer wraps the data with its own header). When receiving, it travels up the layers, stripping headers at each step.

🌐 2. Internet Protocol (IP)

The Internet Protocol (IP) is the principal communications protocol in the Network Layer. It delivers packets from source to destination across multiple networks.

Key Characteristics

  • Packet routing: Transports individual packets from source to destination host
  • Addressing: Assigns IP addresses to uniquely identify each device
  • Packet structure: Each packet contains header (source IP, destination IP, TTL) and payload
  • Connectionless service: IP does not establish a connection before sending; each packet is routed independently
  • Best-effort delivery: IP makes no guarantees — packets may arrive out of order, be duplicated, or be dropped

IP Versions

IPv4

32-bit address space. Format: 192.168.0.1
~4.3 billion unique addresses. Nearly exhausted.

IPv6

128-bit address space. Format: 2001:db8::1
340 undecillion addresses. The future of IP.

Two Transport Protocols Built on IP

TCP/IP (Transmission Control Protocol)

Adds reliability on top of IP. Guarantees delivery, ordering, and error checking. The reliable option.

Connection-oriented

UDP/IP (User Datagram Protocol)

Lightweight protocol on top of IP. No delivery guarantees. The fast option.

Connectionless

⚖ 3. TCP vs UDP Comparison

TCP and UDP are the two main transport protocols. Understanding their trade-offs is essential for choosing the right one.

TCP

Transmission Control Protocol

  • Connection-oriented — handshake before data
  • Reliable delivery — guaranteed arrival
  • Ordered packets — arrives in sequence
  • Acknowledges each packet — ACK/NACK mechanism
  • Flow & congestion control
  • Slower — overhead from reliability
  • Use case: file transfer, web browsing, email, database queries

UDP

User Datagram Protocol

  • Connectionless — no handshake
  • No delivery guarantee — fire and forget
  • No ordering — packets may arrive scrambled
  • No acknowledgment — no ACK/NACK
  • No flow control
  • Faster, lower latency — minimal overhead
  • Use case: video streaming, online gaming, VoIP, DNS
Feature TCP UDP
Connection Connection-oriented (3-way handshake) Connectionless
Reliability Guaranteed delivery No delivery guarantee
Ordering Packets arrive in order No ordering guarantee
Acknowledgment ACK for every packet None
Speed Slower (overhead) Faster, lower latency
Header size 20–60 bytes 8 bytes
Error checking Full error checking + recovery Basic checksum only
Best for File transfer, email, HTTP Video streaming, gaming, VoIP
Rule of thumb: Use TCP when data correctness matters. Use UDP when speed matters more than perfect delivery (e.g., a dropped video frame is acceptable; a corrupted file transfer is not).

🔌 4. Ports

An IP address identifies a computer on the network. A port number identifies a specific application on that computer. Together they form a socket address.

Port numbers are 16-bit unsigned integers, ranging from 0 to 65535.
• 0–1023: Well-known ports (reserved for standard services, require root/admin)
• 1024–49151: Registered ports (for common software)
• 49152–65535: Dynamic/ephemeral ports (used by clients for outgoing connections)

🔭 6. Sockets

A socket is an endpoint for communication between two machines. It is uniquely identified by:

Socket = IP Address + Port Number
A TCP connection has two endpoints (two sockets): one at each end. Data flows in both directions.

Connection Lifecycle

1
Server creates ServerSocket on a specific port — begins listening for connections
2
Server calls accept() — the thread blocks, waiting for a client to connect
3
Client creates Socket(hostname, port) — this triggers the TCP 3-way handshake (connection request)
4
Server's accept() returns — a new Socket object is created specifically for this client
5
Both sides have a Socket — bidirectional communication via getInputStream() / getOutputStream()

⏱ 7. TCP Time Server / Client

A classic first networking example: a server that sends the current date/time to any client that connects. The client simply connects, reads one line, and prints it.

Time Server

TimeServer.java
import java.net.*;
import java.io.*;
import java.util.Date;

public class TimeServer {
    public static void main(String[] args) throws IOException {
        ServerSocket timeSocket = new ServerSocket(6868);
        System.out.println("Server is listening on port 6868");

        while (true) {
            Socket socket = timeSocket.accept(); // blocks here
            System.out.println("New client connected");

            PrintWriter writer = new PrintWriter(
                socket.getOutputStream(), true);
            writer.println(new Date().toString());
            socket.close();
        }
    }
}
Server is listening on port 6868 New client connected

Time Client

TimeClient.java
import java.net.*;
import java.io.*;

public class TimeClient {
    public static void main(String[] args) throws IOException {
        Socket socket = new Socket("localhost", 6868);

        BufferedReader reader = new BufferedReader(
            new InputStreamReader(socket.getInputStream()));

        String time = reader.readLine();
        System.out.println(time);
        socket.close();
    }
}

↩ 8. Echo Reverse Server / Client

A more interactive example: the client sends a string; the server reverses it and sends it back. This demonstrates bidirectional communication over a persistent TCP connection.

Echo Reverse Server

EchoReverseServer.java
import java.net.*;
import java.io.*;

public class EchoReverseServer {
    public static void main(String[] args) throws IOException {
        ServerSocket serverSocket = new ServerSocket(6868);
        System.out.println("Echo Reverse Server started on port 6868");

        Socket socket = serverSocket.accept();
        System.out.println("Client connected: " + socket.getInetAddress());

        BufferedReader reader = new BufferedReader(
            new InputStreamReader(socket.getInputStream()));
        PrintWriter writer = new PrintWriter(
            socket.getOutputStream(), true);

        String line;
        while ((line = reader.readLine()) != null) {
            String reversed = new StringBuilder(line).reverse().toString();
            System.out.println("Received: " + line + " → Sending: " + reversed);
            writer.println(reversed);
        }
        socket.close();
        serverSocket.close();
    }
}
Echo Reverse Server started on port 6868 Client connected: /127.0.0.1 Received: heloe → Sending: eoleh Received: prakash → Sending: hsakarp

📤 9. UDP Datagrams

UDP (User Datagram Protocol) sends data in self-contained messages called datagrams. Unlike TCP, there is no connection setup, no acknowledgment, and no guarantee of delivery or ordering.

No Guarantee
Packets may be lost in transit. The sender never knows if the data arrived.
No Ordering
Packets may arrive out of sequence. The application must handle reordering if needed.
Low Overhead
Minimal header (8 bytes). No handshake. Much faster than TCP for time-sensitive data.

📤 10. UDP Server / Client Code

UDP programming uses DatagramSocket and DatagramPacket instead of Socket. The server binds to a port; the client sends to that port.

UDP Server

UDPServer.java
import java.net.*;
import java.io.*;

public class UDPServer {
    public static void main(String[] args) throws Exception {
        DatagramSocket ds = new DatagramSocket(6868);
        System.out.println("UDP Server listening on port 6868");

        byte[] receiveData = new byte[1024];

        while (true) {
            DatagramPacket dpReceive = new DatagramPacket(
                receiveData, receiveData.length);
            ds.receive(dpReceive); // blocks until data arrives

            String str = new String(dpReceive.getData(),
                0, dpReceive.getLength());
            System.out.println("Data from client [" +
                dpReceive.getAddress() + "] : " + str);
        }
    }
}
UDP Server listening on port 6868 Data from client [/127.0.0.1] : Hello, UDP Server! Data from client [/127.0.0.1] : How are you?

UDP Client

UDPClient.java
import java.net.*;
import java.io.*;

public class UDPClient {
    public static void main(String[] args) throws Exception {
        DatagramSocket ds = new DatagramSocket(); // no port needed for client
        InetAddress ip = InetAddress.getByName("localhost");

        BufferedReader input = new BufferedReader(
            new InputStreamReader(System.in));

        while (true) {
            String msg = input.readLine();
            byte[] buf = msg.getBytes();

            DatagramPacket dpSend = new DatagramPacket(
                buf, buf.length, ip, 6868);
            ds.send(dpSend);

            if (msg.equals("bye")) break;
        }
        ds.close();
    }
}

⚖ 11. TCP vs UDP Use Cases

Choosing between TCP and UDP depends on whether reliability or speed is the higher priority for your application.

✅ Use TCP When...

Correctness matters more than speed

  • File downloads (HTTP/FTP)
    Every byte must arrive correctly. A missing block corrupts the file.
  • Web browsing (HTTP/HTTPS)
    HTML, CSS, JS must arrive intact for the page to render correctly.
  • Email (SMTP/IMAP/POP3)
    Messages must not be truncated or lost.
  • Database queries (JDBC/SQL)
    Query results must be complete and correct.
  • Remote login (SSH)
    Every keystroke and response must arrive in order.
  • Financial transactions
    A lost payment instruction could be catastrophic.

⚡ Use UDP When...

Speed and low latency matter more than reliability

  • Live video streaming (YouTube Live, Twitch)
    A dropped frame is barely noticeable; buffering is worse.
  • Online gaming
    Player position updates need to be real-time; stale data is useless.
  • VoIP / Video calls (Zoom, Teams)
    A brief audio glitch is acceptable; a multi-second TCP retransmit delay is not.
  • DNS lookups
    Small, single-packet queries/responses. If it fails, just retry.
  • IoT sensor broadcasts
    Temperature readings — losing one in a thousand is fine.
  • Network discovery (DHCP, mDNS)
    Broadcast/multicast to unknown addresses — TCP can't do this.

Decision Guide

Summary

Application Protocol Reason
File download (HTTP/FTP)TCPEvery byte must be correct
Web browsing (HTTPS)TCPReliable page delivery

Q1 · 2 marks A file of 5000 bytes is sent over a link whose Maximum Transmission Unit (MTU) payload is 1500 bytes per packet. How many packets are required?
The number of packets is the ceiling of data divided by MTU.
5000 / 1500 = 3.33… → ceil(3.33) = 4 (3 full packets of 1500 = 4500, plus 500 leftover → 1 more)
So 4 packets are needed.
Q2 · 2 marks How long does it take to transfer a 4 MB file over a link with a bandwidth of 8 Mbps (1 byte = 8 bits)?
Convert the file size to bits, then divide by bandwidth (time = size / bandwidth).
4 MB × 8 = 32 Mbit → 32 Mbit ÷ 8 Mbps = 4 s
The transfer takes 4 seconds.
Q3 · 2 marks An IPv4 address is 32 bits wide. How many bytes is that, and what is the range of "well-known" port numbers?
Bytes are computed by dividing the bit width by 8; well-known ports are a fixed reserved range.
32 bits ÷ 8 = 4 bytes · well-known ports = 0 to 1023
An IPv4 address is 4 bytes and the well-known port range is 0–1023 (e.g., HTTP 80, FTP 21).
Q4 · 2 marks Which set of properties correctly describes UDP (compared with TCP)?
TCP is connection-oriented and reliable (handshake, acknowledgements, ordering, retransmission) at the cost of overhead. UDP is connectionless and unreliable — it just fires datagrams with no handshake or delivery guarantee, which makes it fast and lightweight (good for streaming, gaming, DNS). The answer is connectionless, unreliable, fast, low overhead.
Q5 · 2 marks In Java socket programming, what is the role of ServerSocket versus Socket?
In TCP, the server creates a ServerSocket bound to a port and calls accept(), which blocks until a client connects and then returns a Socket. The client creates a Socket to connect. So ServerSocket listens/accepts while a Socket is one live connection used for the actual I/O streams.
Q6 · 2 marks The TCP/IP reference model has 4 layers. Which option lists them from top (application-facing) to bottom?
The TCP/IP model has exactly 4 layers: Application → Transport → Internet → Network Access (Link). (Presentation/Session/Physical/Data-Link are OSI-model terms.) The transport layer hosts TCP and UDP; the internet layer hosts IP. The answer is Application → Transport → Internet → Network Access.
