1. Polymorphism Overview

Etymology: Greek poly (many) + morphe (shape/form). In Java: the same method name behaves differently depending on context.

Polymorphism is one of the four pillars of OOP (alongside Encapsulation, Abstraction, and Inheritance). It lets you write flexible, extensible code by programming to a general interface rather than a specific implementation.

Two Types of Polymorphism in Java

Compile-time Polymorphism

Also called static binding or early binding. The compiler resolves the method call at compile time.

Achieved through Method Overloading — same name, different parameter lists.

Static Binding

Runtime Polymorphism

Also called dynamic binding or late binding. The JVM resolves the method call at runtime based on the actual object type.

Achieved through Method Overriding — subclass redefines a superclass method.

Dynamic Binding

2. Compile-time Polymorphism — Method Overloading

Method overloading means defining multiple methods in the same class with the same name but different parameter lists (number, type, or order of parameters).

Rules: Overloaded methods must differ in parameter list. Return type alone is not sufficient to overload. The compiler selects the best match based on argument types.
Example.java 
class Example {
    // Overload 1: single int
    void demo(int a) {
        System.out.println("int: " + a);
    }

    // Overload 2: two ints
    void demo(int a, int b) {
        System.out.println("ints: " + a + "," + b);
    }

    // Overload 3: double — also returns the value
    double demo(double a) {
        System.out.println("double: " + a);
        return a;
    }
}

public class OverloadTest {
    public static void main(String[] args) {
        Example ex = new Example();
        ex.demo(5);           // calls Overload 1
        ex.demo(3, 7);        // calls Overload 2
        ex.demo(9.5);         // calls Overload 3
    }
}
int: 5 ints: 3,7 double: 9.5

How the Compiler Picks the Right Overload

Overload Resolver — Which method gets called?

Select the argument type(s) you pass to demo():

Key Points

  • Overloading is a form of static polymorphism — resolved at compile time.
  • Java performs automatic widening: int → long → float → double if no exact match exists.
  • Varargs (int... nums) can also create overloads but are matched last.
  • Constructors can also be overloaded — this is very common in Java.

3. Runtime Polymorphism — Method Overriding

Method overriding occurs when a subclass provides its own implementation of a method already defined in its superclass. The method signature (name + parameters) must be identical.

@Override annotation: Always add @Override. The compiler verifies you are actually overriding — it will error if you accidentally mistype the method name.
AnimalDemo.java 
class Animal {
    void makeSound() {
        System.out.println("Some generic sound");
    }
}

class Dog extends Animal {
    @Override
    void makeSound() {
        System.out.println("Bark");
    }
}

class Cat extends Animal {
    @Override
    void makeSound() {
        System.out.println("Meow");
    }
}

public class AnimalDemo {
    public static void main(String[] args) {
        Animal a = new Animal();
        Animal d = new Dog();   // superclass ref, subclass object
        Animal c = new Cat();

        a.makeSound();  // Animal's method
        d.makeSound();  // Dog's method — resolved at runtime!
        c.makeSound();  // Cat's method — resolved at runtime!
    }
}
Some generic sound Bark Meow

4. Circle and Cylinder Inheritance

A classic textbook example: Cylinder extends Circle. The cylinder is a circle with an added height dimension. This demonstrates how inheritance allows code reuse.

UML Inheritance Diagram

Circle.java + Cylinder.java 
class Circle {
    private double radius;

    public Circle(double radius) {
        this.radius = radius;
    }

    public double getRadius() { return radius; }

    public double getArea() {
        return Math.PI * radius * radius;
    }

    @Override
    public String toString() {
        return "Circle[radius=" + radius + "]";
    }
}

class Cylinder extends Circle {
    private double height;

    public Cylinder(double radius, double height) {
        super(radius);      // call Circle's constructor
        this.height = height;
    }

    public double getVolume() {
        return getArea() * height;   // reuses Circle.getArea()
    }

    @Override
    public String toString() {
        return "Cylinder[radius=" + getRadius() + " Height=" + height + "]";
    }
}

public class TestCircleCylinder {
    public static void main(String[] args) {
        Circle   c  = new Circle(12.0);
        Cylinder cy = new Cylinder(2.0, 5.0);

        System.out.println(c);
        System.out.printf("Area is %.2f%n", c.getArea());

        System.out.println(cy);
        System.out.printf("Base area is %.2f, Volume is %.2f%n",
                          cy.getArea(), cy.getVolume());
    }
}
Circle[radius=12.0] Area is 452.39 Cylinder[radius=2.0 Height=5.0] Base area is 12.57, Volume is 62.83

What Cylinder Inherits from Circle

  • radius field (private — accessed via getRadius())
  • getArea() method — fully reused in getVolume()
  • toString() — overridden to add cylinder-specific output

5. Hierarchical Inheritance — BankAccount

Hierarchical inheritance: multiple subclasses share one common superclass. Here, SavingsAccount and CheckingAccount both extend BankAccount.

Inheritance Tree

BankAccount.java / SavingsAccount.java / CheckingAccount.java / AccountTest.java 
class BankAccount {
    protected String name;
    protected double balance;

    public BankAccount(String name, double balance) {
        this.name    = name;
        this.balance = balance;
    }
    public void deposit(double amount)  { balance += amount; }
    public void withdraw(double amount) { balance -= amount; }
    public double getBalance()           { return balance; }
    public String getName()              { return name; }
}

class SavingsAccount extends BankAccount {
    private double interestRate;   // e.g. 0.03 for 3 %

    public SavingsAccount(String name, double balance, double rate) {
        super(name, balance);
        this.interestRate = rate;
    }
    public void addInterest() {
        balance += balance * interestRate;
    }
}

class CheckingAccount extends BankAccount {
    private double overdraftLimit;

    public CheckingAccount(String name, double balance, double limit) {
        super(name, balance);
        this.overdraftLimit = limit;
    }
    public void transfer(BankAccount target, double amount) {
        if (balance - amount >= -overdraftLimit) {
            withdraw(amount);
            target.deposit(amount);
        } else {
            System.out.println("Transfer denied: overdraft limit exceeded");
        }
    }
}

public class AccountTest {
    public static void main(String[] args) {
        SavingsAccount  mum   = new SavingsAccount("Mummy", 8000.0, 0.05);
        CheckingAccount harry = new CheckingAccount("Harry",  500.0, 200.0);

        harry.deposit(50.0);  // Harry deposits £50
        mum.addInterest();     // 5 % interest on Mummy's account

        System.out.println(mum.getName()   + "'s Account balance: " + mum.getBalance());
        System.out.println(harry.getName() + "'s Account balance: " + harry.getBalance());

        harry.transfer(mum, 30.0);  // Harry sends £30 to Mummy
        System.out.println("After transfer — Harry: " + harry.getBalance()
                         + "  Mummy: " + mum.getBalance());
    }
}
Mummy's Account balance: 8400.0 Harry's Account balance: 550.0 After transfer — Harry: 520.0 Mummy: 8430.0

6. Dynamic Method Dispatch / Upcasting

Upcasting: Storing a subclass object in a superclass reference variable. Animal a = new Dog(); — legal because a Dog is-an Animal.

When you call an overridden method through a superclass reference, Java's JVM looks at the actual runtime type of the object — not the reference type — and calls the correct version. This is called Dynamic Method Dispatch.

DispatchDemo.java 
class A {
    void display() { System.out.println("Inside A"); }
}
class B extends A {
    @Override
    void display() { System.out.println("Inside B"); }
}
class C extends A {
    @Override
    void display() { System.out.println("Inside C"); }
}

public class DispatchDemo {
    public static void main(String[] args) {
        A ref;                   // superclass reference

        ref = new A();           // ref points to A object
        ref.display();

        ref = new B();           // ref now points to B object
        ref.display();            // JVM dispatches to B.display()

        ref = new C();           // ref now points to C object
        ref.display();            // JVM dispatches to C.display()
    }
}
Inside A Inside B Inside C

Reference Type vs Actual Type

Key rule: The reference type (left-hand side) determines which members are accessible at compile time. The actual object type (right-hand side, what was new'd) determines which overridden method runs at runtime.

7. Access Modifiers

Java has four access levels that control visibility of classes, fields, methods, and constructors.

Modifier Within Class Same Package Subclass (any pkg) Outside Package
public ✓ ✓ ✓ ✓

default (no modifier) ✓ ✓ ✗ ✗
private ✓ ✗ ✗ ✗
AccessDemo.java 
class Parent {
    public    int pub  = 1;   // visible everywhere
    protected int prot = 2;   // class + package + subclasses
              int def  = 3;   // class + same package only
    private   int priv = 4;   // class only
}

class Child extends Parent {
    void show() {
        System.out.println("pub="  + pub);    // OK
        System.out.println("prot=" + prot);   // OK (inherited)
        // def is accessible here because same package
        System.out.println("def="  + def);    // OK (same package)
        // System.out.println(priv); // COMPILE ERROR
    }
}

public class AccessDemo {
    public static void main(String[] args) {
        new Child().show();
    }
}
pub=1 prot=2 def=3

Best Practices

  • Prefer private for fields — expose only via getters/setters (encapsulation).
  • Use protected when subclasses need direct access (e.g., balance in BankAccount).
  • Make methods public only if they form part of the class's API contract.
  • Avoid default (package-private) for long-lived code — package structure can change.

8. The super Keyword

The super keyword refers to the immediate parent class. It has three distinct uses:

1. Access Parent Variable

When a child field shadows a parent field, use super.fieldName to reach the parent's version.

2. Call Parent Constructor

Use super(args) to invoke the parent constructor. Must be the first statement in the child constructor.

3. Call Parent Method

Use super.methodName() to call the overridden version from the parent class.

SuperDemo.java 
class Vehicle {
    String name = "Vehicle";   // parent field

    public Vehicle(String name) {
        this.name = name;
        System.out.println("Vehicle constructor: " + name);
    }

    void display() {
        System.out.println("I am a vehicle: " + name);
    }
}

class Car extends Vehicle {
    String name = "Car";       // shadows parent field

    public Car(String model) {
        super(model);            // USE 2: call parent constructor
    }

    @Override
    void display() {
        super.display();         // USE 3: call parent method
        System.out.println("Parent name field: " + super.name);  // USE 1
        System.out.println("Child  name field: " + name);
    }
}

public class SuperDemo {
    public static void main(String[] args) {
        Car c = new Car("Tesla");
        c.display();
    }
}
Vehicle constructor: Tesla I am a vehicle: Tesla Parent name field: Tesla Child name field: Car
Important: super() must always be the first statement in a constructor. Java automatically inserts a no-arg super() if you do not write one explicitly, but if the parent has no no-arg constructor you must call the correct one yourself.

9. Abstract Class

An abstract class is a class that cannot be instantiated directly. It may contain abstract methods (declared but not implemented) alongside concrete methods. Subclasses must implement all abstract methods.

Key Rules:
  • Declared with the abstract keyword.
  • Can have both abstract and non-abstract methods.
  • Can have constructors, fields, and static methods.
  • A class with even one abstract method must itself be abstract.
  • You cannot create objects of an abstract class with new.

Shape Hierarchy

ShapeDemo.java 
abstract public class Shape {
    private String color;

    public Shape(String color) { this.color = color; }

    // abstract method — no body, subclass MUST implement
    abstract public double getArea();

    @Override
    public String toString() {
        return "Shape[color=" + color + "]";
    }
}

class Rectangle extends Shape {
    private int length, width;

    public Rectangle(String color, int l, int w) {
        super(color);
        length = l; width = w;
    }
    @Override
    public double getArea() { return length * width; }
}

class Triangle extends Shape {
    private int base, height;

    public Triangle(String color, int b, int h) {
        super(color);
        base = b; height = h;
    }
    @Override
    public double getArea() { return 0.5 * base * height; }
}

public class ShapeDemo {
    public static void main(String[] args) {
        Shape[] shapes = {
            new Rectangle("red",    4, 5),
            new Triangle("blue",   4, 5)
        };

        for (Shape s : shapes) {
            System.out.println(s + " area = " + s.getArea());
        }
    }
}
Shape[color=red] area = 20.0 Shape[color=blue] area = 10.0

10. Interface

An interface defines a contract — a set of methods that implementing classes must provide. Before Java 8 all methods were implicitly abstract; Java 8+ allows default and static methods in interfaces.

Key Characteristics

All methods abstract

Unless declared default or static. Implementing class must provide a body for every abstract method.

Multiple Inheritance

A class can implement many interfaces. A class can only extend one superclass. Interfaces solve Java's single-inheritance limitation.

Constants only

All fields in an interface are implicitly public static final. No instance variables are allowed.

TaxableDemo.java 
interface Taxable {
    double taxRate = 0.06;           // implicitly public static final
    double calculateTax();           // implicitly public abstract

    // Java 8+ default method (shared implementation)
    default String taxSummary(String item) {
        return item + " tax: $" + calculateTax();
    }
}

class Toy implements Taxable {
    double cost;
    public Toy(double cost) { this.cost = cost; }

    @Override
    public double calculateTax() { return cost * taxRate; }
}

class Book implements Taxable {
    double price;
    public Book(double price) { this.price = price; }

    @Override
    public double calculateTax() { return price * taxRate; }
}

public class Store {
    public static void main(String[] args) {
        Taxable[] items = {
            new Toy(25.00),
            new Book(12.50)
        };

        for (Taxable t : items) {
            System.out.printf("Tax = $%.2f  |  %s%n",
                t.calculateTax(),
                t.taxSummary(t.getClass().getSimpleName()));
        }
        System.out.printf("Tax rate in use: %.0f%%%n", Taxable.taxRate * 100);
    }
}
Tax = $1.50 | Toy tax: $1.5 Tax = $0.75 | Book tax: $0.75 Tax rate in use: 6%

Interface: default and static Methods (Java 8+)

ModernInterface.java 
interface Printable {
    void print();                       // abstract

    default void printTwice() {         // default — has a body
        print(); print();
    }

    static void info() {               // static — belongs to interface
        System.out.println("Printable interface v1.0");
    }
}

class Document implements Printable {
    private String content;
    public Document(String s) { content = s; }

    @Override
    public void print() { System.out.println("[DOC] " + content); }
}

public class ModernInterface {
    public static void main(String[] args) {
        Printable.info();                    // static method on interface
        Document d = new Document("Hello World");
        d.printTwice();                      // uses default method
    }
}

12. The final Keyword

The final keyword can be applied to variables, methods, and classes, each with a different meaning:

final variable

Creates a constant. Value can only be assigned once. Must be initialised at declaration or in the constructor.

final double PI = 3.14159;

final method

Cannot be overridden by a subclass. Useful when you need to guarantee a specific behaviour in the inheritance hierarchy.

final void lock() { ... }

final class

Cannot be subclassed. String, Integer, and all other Java wrapper classes are final. Ensures immutability and security.

final class Immutable { ... }

FinalDemo.java 
// final CLASS — cannot be subclassed
final class MathConstants {
    public static final double PI  = 3.14159265358979;
    public static final double E   = 2.71828182845904;
}

class Parent {
    final void show() {   // final METHOD — cannot be overridden
        System.out.println("Parent.show() — cannot be overridden");
    }
    void info() {
        System.out.println("Parent.info() — can be overridden");
    }
}

class Child extends Parent {
    // @Override void show() { }  ← COMPILE ERROR: cannot override final
    @Override
    void info() {
        System.out.println("Child.info() — overriding is fine");
    }
}

public class FinalDemo {
    public static void main(String[] args) {
        // final VARIABLE
        final int MAX_SCORE = 100;
        // MAX_SCORE = 200;  ← COMPILE ERROR

        System.out.println("PI = "    + MathConstants.PI);
        System.out.println("MAX   = " + MAX_SCORE);

        Child ch = new Child();
        ch.show();   // calls Parent's final method
        ch.info();   // calls Child's overridden method
    }
}
PI = 3.14159265358979 MAX = 100 Parent.show() — cannot be overridden Child.info() — overriding is fine

String is a final Class

java.lang.String is declared final. This guarantees:

  • Immutability — no subclass can introduce mutable state or change string behaviour.
  • Security — class loaders rely on String; a malicious subclass could compromise the JVM.
  • Caching — the JVM can safely cache and reuse String objects (String pool).
  • Thread safety — immutable objects are inherently thread-safe.
StringFinal.java 
// class MyString extends String { }  ← COMPILE ERROR: String is final

public class StringFinal {
    public static void main(String[] args) {
        final String greeting = "Hello";
        // greeting = "Hi";  ← COMPILE ERROR: final variable

        String s1 = "Java";
        String s2 = "Java";   // same pool object
        String s3 = new String("Java");

        System.out.println(s1 == s2);          // true  (same pool ref)
        System.out.println(s1 == s3);          // false (new heap object)
        System.out.println(s1.equals(s3));     // true  (same content)
        System.out.println(greeting + " World"); // new String — original unchanged
    }
}
true false true Hello World

Packages — Brief Overview

A package is a namespace that groups related classes. Packages prevent naming conflicts, control access (default/package-private modifier), and make large codebases manageable.

com/example/shapes/Circle.java 
// File: com/example/shapes/Circle.java
package com.example.shapes;   // declare the package

public class Circle {
    private double radius;

    public Circle(double r) { this.radius = r; }

    public double area() { return Math.PI * radius * radius; }
}

// File: Main.java
import com.example.shapes.Circle;   // import the class
// import com.example.shapes.*;     // or import all from package

public class Main {
    public static void main(String[] args) {
        Circle c = new Circle(7.0);
        System.out.printf("Area = %.2f%n", c.area());
    }
}
Area = 153.94

📝 Exam-Style MCQs — 2 Marks Each

Each question is worth 2 marks — a mix of applied numerical and conceptual items. Click an option to check it; the correct answer and explanation appear instantly.

Q1 · 2 marks Given these two overloaded methods, which line is printed by the call f(7)?
static int f(int n) { return n * n; } static int f(long n) { return (int)(n + 1); } System.out.println(f(7));
Overload resolution prefers the most specific match without widening. The literal 7 is an int, so f(int) is chosen over f(long) (which would require a widening conversion).
f(int) chosen → 7 * 7 = 49
Had no f(int) existed, 7 would widen to long and give 8 (the distractor). The answer is 49.
Q2 · 2 marks A BankAccount starts with a balance of 2000. Its withdraw() rejects (does nothing to the balance) when the amount exceeds the balance. After the calls below, what is the final balance?
acc.withdraw(2500); // amount > balance acc.deposit(500); acc.withdraw(300);
The first withdraw is rejected because 2500 > 2000, so the balance is unchanged.
2000 (withdraw 2500 rejected, no change) 2000 + 500 = 2500 (deposit) 2500 - 300 = 2200 (withdraw)
Treating the rejected withdraw as if it applied (2000 − 2500 = −300) is the common mistake. The answer is 2200.
Q3 · 2 marks With dynamic method dispatch, which value does area hold after this code runs?
class Shape { int getArea() { return 0; } } class Square extends Shape { int side; Square(int s) { side = s; } int getArea() { return side * side; } } Shape s = new Square(9); int area = s.getArea();
The reference type is Shape, but the object is a Square. Overridden instance methods are dispatched on the runtime object (late binding), so Square.getArea() runs.
side = 9 getArea() = 9 * 9 = 81
Returning 0 would happen only with static binding to Shape (the distractor). The answer is 81.
Q4 · 2 marks Two methods named draw in the same class differ only in their parameter lists; a third draw in a subclass has the identical signature as one in the parent. Which statement correctly classifies these relationships?
Overloading = same name, different parameter lists, within one class — resolved by the compiler (static binding). Overriding = a subclass method with the identical signature as a parent method — resolved at runtime (dynamic dispatch). So the two same-class variants overload each other, and the matching-signature subclass method overrides. The answer is B.
Q5 · 2 marks You must model a capability (Taxable) that several unrelated classes can gain, while also sharing some common state and a partially-implemented method among a family of related shapes. Which design best fits?
A class can implement many interfaces, so a capability that crosses unrelated hierarchies (like Taxable) is best an interface. Shared instance fields plus a half-finished method that related subclasses complete is exactly what an abstract class provides (a class extends only one). The answer is C.
Q6 · 2 marks Which line below causes a compile-time error?
final class Engine { } class V8 extends Engine { } // line 1 class Account { final int id = 100; void reset() { id = 0; } // line 2 } final void run() { } // line 3 (inside a class)
final on a class forbids subclassing, so V8 extends Engine (line 1) fails. final on a field makes it non-reassignable, so id = 0 (line 2) fails. A final method (line 3) is perfectly legal — it merely cannot be overridden. So lines 1 and 2 are the errors. The answer is D.
