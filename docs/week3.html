1. Types of Errors

Errors in programming can be broadly classified based on when they occur and what causes them. Understanding error types is the first step to writing robust Java programs.

Error Classification Tree

Quick Reference

Key insight: Compile-time errors are caught by the compiler before the program runs. Runtime errors occur during execution — exceptions are the recoverable kind.

2. Compile-time Errors

Compile-time errors are caught by the Java compiler before your program ever runs. There are two main categories:

Syntax Error — Violates the grammatical rules of Java. E.g., missing semicolons, unmatched braces, misspelled keywords.
Semantic Error — Syntactically correct but logically meaningless to the compiler. E.g., assigning a String to an int variable.

Syntax Error Example — Circle.java

The following class has two syntax errors: a missing semicolon and a missing return type on a method declaration.

Java 
public class Circle {
    double radius;

    Circle(double r) {
        radius = r;
    }

    double area() {
        return 3.14 * radius * radius   // <-- missing semicolon on line 9
    }

    circumference() {                    // <-- missing return type on line 12
        return 2 * 3.14 * radius;
    }
}
Compiler Output

Semantic Error Example — Type Mismatch

The compiler understands the syntax but rejects the assignment because the types are incompatible.

Java 
public class TypeDemo {
    public static void main(String[] args) {
        int num_1;
        num_1 = "10";  // Semantic Error: cannot assign String to int
        System.out.println(num_1);
    }
}
Compiler Output
TypeDemo.java:4: error: incompatible types: String cannot be converted to int num_1 = "10"; ^ 1 error
Note: Semantic errors do NOT mean the logic is wrong — they mean the code violates the type system or scope rules of the language, even though the syntax is valid. The compiler catches these before runtime.

3. Runtime Errors & Exception Hierarchy

Runtime errors occur while the program is executing. Among them, exceptions are special objects that Java can throw and catch. The entire exception system is built on the Throwable class hierarchy.

4. Try-Catch-Finally

Java provides the try-catch-finally construct to handle exceptions gracefully — preventing abnormal termination and allowing the program to respond meaningfully to errors.

try { }

Encloses code that might throw an exception. The JVM monitors this block. If an exception is thrown, control immediately jumps to the matching catch block.

catch (E e) { }

Handles a specific exception type. You can have multiple catch blocks for different exception types. Only the matching block executes.

finally { }

Always executes — whether or not an exception occurred. Ideal for cleanup code: closing files, releasing database connections, etc.

Basic Try-Catch-Finally Example

Java 
public class TryCatchDemo {
    public static void main(String[] args) {
        try {
            int data = 100 / 0;   // throws ArithmeticException
        } catch (ArithmeticException e) {
            System.out.println("Cannot divide by zero!");
        } finally {
            System.out.println("This is the finally block.");
        }
    }
}
Output
Cannot divide by zero! This is the finally block.

Multiple Catch Blocks

You can catch different exception types with separate catch blocks. The JVM checks them in order — only the first matching block executes.

Java 
public class MultiCatch {
    public static void main(String[] args) {
        try {
            int[] arr = new int[5];
            arr[10] = 50;  // ArrayIndexOutOfBoundsException
        } catch (ArithmeticException e) {
            System.out.println("Arithmetic error: " + e.getMessage());
        } catch (ArrayIndexOutOfBoundsException e) {
            System.out.println("Array error: " + e.getMessage());
        } finally {
            System.out.println("Cleanup done.");
        }
    }
}
Output
Array error: Index 10 out of bounds for length 5 Cleanup done.
Remember: The finally block runs even if a return statement is inside the try or catch block. The only exception is if the JVM exits (e.g., System.exit() is called).

5. ArrayExcep Example

This classic example demonstrates how an ArrayIndexOutOfBoundsException is thrown when we access index 10 on an array of size 5, and how the program continues normally after the exception is caught.

Java 
class ArrayExcep {
    public static void main(String args[]) {
        int a[] = {10, 20, 30, 40, 50};
        try {
            System.out.println("Just before exception..");
            System.out.println(a[10]);      // Index 10 does not exist!
            System.out.println("Not executed"); // Skipped
        } catch(ArrayIndexOutOfBoundsException e) {
            System.out.println("Array boundary crossed" + e);
        }
        System.out.println("After handling exception: Normal flow");
    }
}
Output
Just before exception.. Array boundary crossedjava.lang.ArrayIndexOutOfBoundsException: Index 10 out of bounds for length 5 After handling exception: Normal flow

Execution Flow Explained

1
"Just before exception.." prints — the try block starts executing normally.
2
a[10] is accessed — array only has indices 0–4, so ArrayIndexOutOfBoundsException is thrown immediately.
3
"Not executed" is skipped — control jumps straight to the catch block.
4
The catch block prints the exception details — e.toString() gives the exception class and message.
5
Execution continues after the try-catch block — the program does NOT crash.
Key observation: Without the try-catch block, the program would terminate with an unhandled exception stack trace. With it, execution continues normally after the handler — demonstrating that exceptions are recoverable.

6. Average Program — Progressive Exception Handling

We build a program that calculates the average of passing marks over 4 versions, each adding better exception handling. This shows how real programs evolve to become more robust.

Version 1: No Exception Handling — Crashes on Bad Input

The basic program calculates the average of marks. It crashes with ArithmeticException when there are zero passing marks (divide by zero) or when invalid input is provided.

Java 
import java.util.Scanner;

class Average {
    public static void main(String[] args) {
        Scanner sc = new Scanner(System.in);
        int[] marks = new int[5];
        int sum = 0, count = 0;
        System.out.println("Enter 5 marks:");
        for (int i = 0; i < 5; i++) {
            marks[i] = sc.nextInt();
            if (marks[i] >= 40) { sum += marks[i]; count++; }
        }
        System.out.println("Average of passing marks: " + (sum / count));
    }
}
Output — when no student passes (count=0)
Enter 5 marks: 10 20 30 15 25 Exception in thread "main" java.lang.ArithmeticException: / by zero at Average.main(Average.java:13)

Version 2: Handle ArithmeticException (Divide by Zero)

We add a try-catch for ArithmeticException so that the program handles the case when no students pass (count = 0).

Java 
import java.util.Scanner;

class Average {
    public static void main(String[] args) {
        Scanner sc = new Scanner(System.in);
        int[] marks = new int[5];
        int sum = 0, count = 0;
        System.out.println("Enter 5 marks:");
        for (int i = 0; i < 5; i++) {
            marks[i] = sc.nextInt();
            if (marks[i] >= 40) { sum += marks[i]; count++; }
        }
        try {
            System.out.println("Average of passing marks: " + (sum / count));
        } catch (ArithmeticException e) {
            System.out.println("No students passed! Cannot compute average.");
        }
    }
}
Output — when no student passes
Enter 5 marks: 10 20 30 15 25 No students passed! Cannot compute average.

If the user enters "89f" (a string that isn't a valid integer), Integer.parseInt() throws NumberFormatException. We now handle that as well.

Version 4: Add ArrayIndexOutOfBoundsException (too many inputs via command line)

When marks are passed via command-line arguments and more than 5 are provided, the loop can go out of bounds. We handle this with a third catch block.

Java 
class AverageArgs {
    public static void main(String[] args) {
        int[] marks = new int[5];
        int sum = 0, count = 0;
        try {
            for (int i = 0; i < 5; i++) {
                marks[i] = Integer.parseInt(args[i]);  // args[i] may not exist
                if (marks[i] >= 40) { sum += marks[i]; count++; }
            }
            System.out.println("Average of passing marks: " + (sum / count));
        } catch (ArithmeticException e) {
            System.out.println("No students passed! Cannot compute average.");
        } catch (NumberFormatException e) {
            System.out.println("Invalid input! Please enter integers only.");
        } catch (ArrayIndexOutOfBoundsException e) {
            System.out.println("Please provide exactly 5 marks as arguments.");
        }
    }
}

7. The throw Keyword

While exceptions are usually thrown automatically by the JVM, you can also throw them manually using the throw keyword. This is useful for enforcing business rules and validating input.

Syntax: throw new ExceptionType("message");
You can throw any object that extends Throwable. The throw statement creates a new exception object and throws it immediately.

Person — Age Validation Example

Java 
public class Person {

    public static void validateAgeToVote(int age) {
        try {
            if (age < 18 || age > 120)
                throw new IllegalArgumentException("Invalid age: " + age);
            System.out.println("Go ahead to vote...");
        } catch (IllegalArgumentException e) {
            System.out.println("Invalid age to vote: " + age);
        }
    }

    public static void main(String[] args) {
        validateAgeToVote(25);
        validateAgeToVote(-5);
        validateAgeToVote(130);
    }
}
Output
Go ahead to vote... Invalid age to vote: -5 Invalid age to vote: 130

Interactive: Age Validator

throw vs throws: throw (lowercase, no 's') is the statement that actually throws an exception object at a specific point in your code. throws (with 's') is a method signature declaration — covered in Section 10.

8. Rethrowing Exceptions

Sometimes a catch block cannot fully handle an exception. In that case, you can rethrow the exception — catch it, do some partial processing, and then throw it again so that an outer handler can deal with it.

Why rethrow? Useful when a method catches an exception to log it or clean up, but needs the caller to also know about the failure. The same exception object is re-thrown using throw e; inside the catch block.

Voting Counter with Rethrow

The inner method validates age and rethrows the exception. The outer loop catches it per iteration and reports the result, then continues to the next voter.

Java 
public class VotingCounter {

    public static void validateAndVote(int age) throws IllegalArgumentException {
        try {
            if (age < 18 || age > 120)
                throw new IllegalArgumentException("Invalid age: " + age);
            System.out.println("Voted (age " + age + ")");
        } catch (IllegalArgumentException e) {
            System.out.println("Caught in inner: logging error...");
            throw e;  // rethrow to outer caller
        }
    }

    public static void main(String[] args) {
        int[] ages = {25, -5, 30, 130, 21};
        for (int age : ages) {
            try {
                validateAndVote(age);
            } catch (IllegalArgumentException e) {
                System.out.println("Outer handler: " + e.getMessage());
            }
        }
    }
}
Output
Voted (age 25) Caught in inner: logging error... Outer handler: Invalid age: -5 Voted (age 30) Caught in inner: logging error... Outer handler: Invalid age: 130 Voted (age 21)

Rethrow as a Different Exception Type

You can also catch one exception type and rethrow it wrapped in a different type — useful for layered architectures where lower-level exceptions are converted to higher-level ones.

Java 
public class RethrowDemo {
    public static void process() throws RuntimeException {
        try {
            int result = 10 / 0;
        } catch (ArithmeticException e) {
            System.out.println("Wrapping ArithmeticException...");
            throw new RuntimeException("Processing failed: " + e.getMessage(), e);
        }
    }

    public static void main(String[] args) {
        try {
            process();
        } catch (RuntimeException e) {
            System.out.println("Caught: " + e.getMessage());
        }
    }
}
Output
Wrapping ArithmeticException... Caught: Processing failed: / by zero

9. Catching Sub & Super Class Exceptions

Since ArrayIndexOutOfBoundsException is a subclass of Exception, you can catch both — but the subclass catch must come BEFORE the superclass catch. If the superclass catch is first, it would catch all exceptions (including the subclass), making the subclass catch block unreachable — a compile-time error.

Rule: Always place more specific (subclass) exception handlers BEFORE more general (superclass) handlers. Java enforces this at compile time — an unreachable catch block is a compile error.

ExDemo — Sub and Super Class Catch

Java 
class ExDemo {
    public static void main(String[] args) {
        int[] a = {2, 4, 8, 16};
        String[] s = {"No matching", "No matching"};

        for (int i = 0; i < 6; i++) {
            try {
                if (i == 1) throw new Exception("Some Exception");
                System.out.println(a[i]);
            } catch (ArrayIndexOutOfBoundsException e) {
                // More specific: triggered when i >= a.length (i.e. i=4 or 5)
                System.out.println(s[i - 4]);
            } catch (Exception e) {
                // More general: triggered for all other exceptions (i=1 here)
                System.out.println(e.getMessage());
            }
        }
    }
}
Output
2 Some Exception 4 8 Some Exception 16 No matching No matching

10. The throws Keyword

The throws keyword is used in a method declaration to indicate that the method might throw one or more exceptions that it does not handle internally. It is a warning to the caller: "handle this or declare it too."

throw (no 's'): Used inside a method body to actually throw an exception object.
throw new ExceptionType(...);
throws (with 's'): Used in method signature to declare exceptions a method may propagate to its callers.
void method() throws ExType { ... }

BankAccount — withdraw() with throws

Java 
class BankAccount {
    private double balance;

    BankAccount(double initialBalance) {
        this.balance = initialBalance;
    }

    // Declares that it may throw IllegalArgumentException
    public void withdraw(double amount) throws IllegalArgumentException {
        if (amount <= 0 || amount > balance) {
            throw new IllegalArgumentException(
                "Trying to withdraw invalid amount: " + amount);
        }
        balance -= amount;
        System.out.println("Withdrew: " + amount + ". Balance: " + balance);
    }
}

class AccountTest {
    public static void main(String[] args) {
        BankAccount acc = new BankAccount(1000.0);
        try {
            acc.withdraw(200.0);   // valid
            acc.withdraw(-50.0);  // invalid — throws
        } catch (IllegalArgumentException e) {
            System.out.println(e);
        }
    }
}
Output
Withdrew: 200.0. Balance: 800.0 java.lang.IllegalArgumentException: Trying to withdraw invalid amount: -50.0

When to Use throws vs try-catch

ScenarioPreferred Approach
You CAN recover from the exception in the same methodUse try-catch locally
The caller should decide how to handle the exceptionUse throws in method signature
Checked exception from a called library (e.g. FileNotFoundException)Use throws or wrap in try-catch
Documenting that a method may propagate unchecked exceptionsOptional throws for clarity
Note: For unchecked exceptions (RuntimeException subclasses), the throws declaration is optional — but using it serves as documentation for callers.

11. User-Defined Exceptions

Java lets you create your own exception classes to represent domain-specific error conditions. This makes error handling more expressive and meaningful for your application.

Template: Extending Exception

Java — Template 
// Step 1: Define your custom exception by extending Exception
class MyCustomException extends Exception {

    // Constructor with message
    public MyCustomException(String message) {
        super(message);  // passes message to Exception
    }

    // Optional: constructor with cause (for exception chaining)
    public MyCustomException(String message, Throwable cause) {
        super(message, cause);
    }
}

// Step 2: Throw it in your business logic
class BusinessService {
    public void process(int value) throws MyCustomException {
        if (value < 0)
            throw new MyCustomException("Value cannot be negative: " + value);
        System.out.println("Processing: " + value);
    }
}

// Step 3: Catch it like any other exception
class Main {
    public static void main(String[] args) {
        BusinessService svc = new BusinessService();
        try {
            svc.process(-3);
        } catch (MyCustomException e) {
            System.out.println("Custom exception caught: " + e.getMessage());
        }
    }
}
Output
Custom exception caught: Value cannot be negative: -3

VotingException — Full Example

Java 
class VotingException extends Exception {
    private int age;

    public VotingException(int age) {
        super("Not eligible to vote. Age: " + age);
        this.age = age;
    }

    public int getAge() { return age; }
}

class VotingSystem {
    public static void castVote(String name, int age)
            throws VotingException {
        if (age < 18)
            throw new VotingException(age);
        System.out.println(name + " has voted successfully.");
    }

    public static void main(String[] args) {
        String[][] voters = {
            {"Alice", "25"},
            {"Bob", "16"},
            {"Carol", "30"},
            {"Dave", "15"}
        };
        for (String[] v : voters) {
            try {
                castVote(v[0], Integer.parseInt(v[1]));
            } catch (VotingException e) {
                System.out.println("Error: " + e.getMessage());
            }
        }
    }
}
Output
Alice has voted successfully. Error: Not eligible to vote. Age: 16 Carol has voted successfully. Error: Not eligible to vote. Age: 15

PANException — PAN Number Validator

A PAN (Permanent Account Number) must match the format: 5 uppercase letters, 4 digits, 1 uppercase letter — e.g., ABCDE1234F. We create a PANException to signal invalid PAN numbers.

Java 
class PANException extends Exception {
    public PANException(String message) {
        super(message);
    }
}

class PANValidator {

    // PAN format: [A-Z]{5}[0-9]{4}[A-Z]
    public static void validatePAN(String name, String pan)
            throws PANException {
        if (pan == null || !pan.matches("[A-Z]{5}[0-9]{4}[A-Z]")) {
            throw new PANException("Invalid PAN number: " + name + " " + pan);
        }
        System.out.println("Valid PAN" + name + " " + pan);
    }

    public static void main(String[] args) {
        String[][] people = {
            {"John Doe",   "ABCDE1234F"},
            {"John Doe",   "12345XYZ"}
        };
        for (String[] p : people) {
            try {
                validatePAN(p[0], p[1]);
            } catch (PANException e) {
                System.out.println("Error: " + e.getMessage());
            }
        }
    }
}
Output
Valid PANJohn Doe ABCDE1234F Error: Invalid PAN number: John Doe 12345XYZ

Benefits of User-Defined Exceptions

  • Clarity: Exception names like VotingException or PANException immediately communicate intent to other developers.
  • Domain-specific context: Custom exceptions can carry additional fields (e.g., the invalid age, the invalid PAN string) beyond just a message.
  • Separation of concerns: Different layers of your application can define and handle their own exceptions independently.
  • Better testing: You can test that specific exception types are thrown under specific conditions.

Q1 · 2 marks What value does compute() return?
static int compute() { int x = 5; try { return x; } finally { x = 20; } }
return x first evaluates x (=5) and stores it as the pending return value, then the finally block runs. Reassigning the local x = 20 does not change the already-captured value (the finally has no return of its own to override it).
return value captured = 5 finally sets local x = 20 (ignored for the return) method returns 5
A return 20 inside finally would override it — but here there is none. The answer is 5.

Q4 · 2 marks A method dereferences an object that may be null and also reads a file that may be missing. Which classification of the resulting exceptions is correct?
Unchecked exceptions extend RuntimeException (programming errors like NullPointerException) — the compiler does not force handling. Checked exceptions extend Exception but not RuntimeException (recoverable external conditions like IOException) — the compiler requires a try/catch or a throws clause. The answer is B.
Q5 · 2 marks In Java's exception hierarchy, what is the relationship between Throwable, Error, Exception, and RuntimeException?
Throwable sits at the top of everything that can be thrown or caught. Its two direct children are Error (serious JVM-level problems you normally don't catch) and Exception (application-level conditions). RuntimeException is a subclass of Exception and marks the unchecked branch.
Throwable ├── Error └── Exception └── RuntimeException
The answer is C.
Q6 · 2 marks Consider the two underlined uses of the keywords below. Which statement is correct?
void process() throws IOException { // (1) if (bad) throw new IOException(); // (2) }
throws appears in a method's signature and declares the checked exceptions the method may propagate to its caller. throw is a statement that actually raises a specific exception object (throw new IOException()) at runtime. They are different keywords with different roles. The answer is C.
