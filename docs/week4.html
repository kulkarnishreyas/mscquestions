📂 1. Multiprocessing

Modern computers achieve high performance by exploiting parallelism — performing multiple tasks simultaneously. Java's concurrency model is built on two complementary abstractions: multiprocessing and multithreading.

Multiple CPUs in a Single System

A multiprocessor system contains two or more CPUs sharing main memory and I/O devices. Each CPU can independently execute a stream of instructions, so several computations literally run at the same instant — true parallelism.
  • Modern servers routinely have 32–256 logical cores.
  • Even a typical laptop has 8–16 cores (physical + hyper-threaded).
  • Java's Runtime.getRuntime().availableProcessors() reports available cores at runtime.

Process-Based vs Thread-Based Multitasking

Process-Based Multitasking

The OS runs multiple programs concurrently. Each program lives in its own address space. Switching between processes (context switch) saves and restores the full CPU state plus memory maps.

  • Heavyweight — each process has its own heap, stack, code, and data segments
  • Protected memory boundaries prevent one process crashing another
  • Communication requires IPC: pipes, sockets, shared memory, or message queues
  • Example: running IntelliJ, Chrome, and a Gradle build simultaneously

Thread-Based Multitasking

Multiple threads of control within one program share the same heap and static memory. Switching threads is cheap because only the CPU registers and stack pointer need saving.

  • Lightweight — threads share heap; only their stacks are private
  • Direct method calls for communication — no OS overhead
  • Errors in one thread can corrupt shared data (hence synchronization)
  • Example: a word processor spell-checks while you type

♾ 3. Thread Lifecycle

A Java thread passes through five well-defined states during its lifetime. Understanding these states is essential for writing correct concurrent code.

The Five States Explained

New

A Thread object has been created with new Thread() but start() has not yet been called. The thread occupies memory but no OS-level thread exists yet.

Runnable

After start() is called the thread enters the ready queue. It is eligible to run but waiting for the thread scheduler to pick it. In Java the Runnable state encompasses both ready and running-on-CPU from the JVM spec perspective.

Running

The thread has been selected by the scheduler and its run() method is actively executing on a CPU core. Only one thread per core runs at any instant.

Blocked / Waiting

The thread temporarily suspends. Causes:

  • Thread.sleep(ms) — timed wait
  • Object.wait() — waiting for notification
  • Waiting to acquire a synchronized lock
  • join() — waiting for another thread to finish

Returns to Runnable when the condition resolves.

Dead

The run() method has returned normally, or an uncaught exception terminated it. A dead thread cannot be restarted — calling start() again throws IllegalThreadStateException.

🔨 5. Creating Threads — Method 1: Extending Thread

The simplest way to create a thread is to extend java.lang.Thread and override its run() method. When start() is called, the JVM invokes your run() in a new thread.

Basic Example

JAVA 
class MyThread extends Thread {
    public void run() {
        System.out.println("My thread is running.");
    }
    public static void main(String args[]) {
        MyThread myThread = new MyThread();
        myThread.start();   // do NOT call run() directly!
    }
}
My thread is running.

⚙️ 6. Creating Threads — Method 2: Implementing Runnable

Because Java supports only single inheritance, extending Thread prevents your class from extending anything else. Implementing the Runnable interface is the preferred approach in modern Java.

When to use extends Thread
  • Quick prototypes or teaching examples
  • When no other superclass is needed
  • When you need to override other Thread methods (e.g., interrupt())
When to use implements Runnable
  • Your class already extends another class
  • Production code (recommended by Java best practices)
  • Works with ExecutorService, lambdas, and thread pools
  • Promotes separation of task logic from thread management

Basic Runnable Example

JAVA 
class MyRunnable implements Runnable {
    public void run() {
        System.out.println("My runnable is running.");
    }
}

public class RunnableDemo {
    public static void main(String args[]) {
        MyRunnable r = new MyRunnable();
        Thread t = new Thread(r);  // wrap in Thread
        t.start();
    }
}
My runnable is running.

🔒 7. Thread Synchronization

The Race Condition Problem

When two or more threads access shared mutable state without coordination, the result depends on the unpredictable scheduling order — a race condition. The final value can be wrong and the bug is nearly impossible to reproduce consistently.

Classic Example: Two threads both read a counter value (say 100), both increment to 101, and both write 101 back — net effect: +1 instead of +2. With N threads this rapidly corrupts data.

The synchronized Keyword

Java's built-in lock mechanism ensures that only one thread at a time executes a critical section. Every Java object has an intrinsic monitor lock.

Synchronized Method

JAVA
public synchronized void methodName() {
    // only one thread at a time
    criticalSection();
}

The calling thread acquires the lock on this before entering and releases it on exit (normal or exception).

Synchronized Block

JAVA
public void methodName() {
    // unsynchronized code here
    synchronized(this) {
        criticalSection();
    }
    // more unsynchronized code
}

Finer-grained — only the critical part is locked, improving throughput. Can lock on any object, not just this.

Auction Example

Multiple bidder threads call placeBid() concurrently. Without synchronized the highest bidder could be overwritten by a lower bid arriving a millisecond later.

JAVA 
class Auction {
    private double highestBid = 0.0;
    private String highestBidder = "";

    public synchronized void placeBid(String bidder, double amount) {
        if (amount > highestBid) {
            highestBid   = amount;
            highestBidder = bidder;
            System.out.println(bidder + " placed a bid of $" + amount);
        } else {
            System.out.println(bidder + "'s bid of $" + amount + " was too low");
        }
    }

    public synchronized String getHighestBidder() { return highestBidder; }
    public synchronized double getHighestBid()    { return highestBid; }
}

class Bidder implements Runnable {
    private Auction auction;
    private String name;
    private double bidAmount;

    Bidder(Auction a, String n, double b) {
        auction = a; name = n; bidAmount = b;
    }

    public void run() {
        auction.placeBid(name, bidAmount);
    }
}

public class BiddingExample {
    public static void main(String args[]) throws InterruptedException {
        Auction auction = new Auction();

        Thread[] bidders = {
            new Thread(new Bidder(auction, "Alice",  200.00)),
            new Thread(new Bidder(auction, "Bob",    450.00)),
            new Thread(new Bidder(auction, "Charlie",300.00)),
            new Thread(new Bidder(auction, "Diana",  500.00)),
            new Thread(new Bidder(auction, "Eve",    475.00))
        };

        for (Thread t : bidders) t.start();
        for (Thread t : bidders) t.join();

        System.out.println("\n=== Auction Result ===");
        System.out.println("Winner: " + auction.getHighestBidder());
        System.out.println("Winning Bid: $" + auction.getHighestBid());
    }
}

💬 8. Inter-Thread Communication

Synchronized access prevents race conditions, but sometimes threads need to coordinate: one thread should pause until another signals that a condition has been met. Java provides three Object methods for this — they must be called inside a synchronized context.

MethodWhat it does
wait() Releases the lock and suspends the calling thread. The thread waits until another thread calls notify() or notifyAll() on the same object.
notify() Wakes up one arbitrary thread waiting on this object's monitor. The awakened thread must re-acquire the lock before continuing.
notifyAll() Wakes up all threads waiting on this object's monitor. Preferred when multiple threads might be eligible to continue.

Producer-Consumer Pattern

The classic inter-thread communication pattern: a Producer generates data and places it in a shared buffer; a Consumer removes and processes it. They must coordinate so the consumer doesn't read an empty buffer and the producer doesn't overflow a full one.

JAVA 
class SharedBuffer {
    private int data;
    private boolean hasData = false;

    public synchronized void produce(int value) throws InterruptedException {
        while (hasData) {
            wait();  // buffer full — wait for consumer
        }
        data    = value;
        hasData = true;
        System.out.println("Produced: " + value);
        notifyAll();
    }

    public synchronized int consume() throws InterruptedException {
        while (!hasData) {
            wait();  // buffer empty — wait for producer
        }
        hasData = false;
        System.out.println("Consumed: " + data);
        notifyAll();
        return data;
    }
}

class Producer implements Runnable {
    private SharedBuffer buffer;
    Producer(SharedBuffer b) { buffer = b; }
    public void run() {
        for (int i = 1; i <= 5; i++) {
            try { buffer.produce(i * 10); Thread.sleep(200); }
            catch (InterruptedException e) {}
        }
    }
}

class Consumer implements Runnable {
    private SharedBuffer buffer;
    Consumer(SharedBuffer b) { buffer = b; }
    public void run() {
        for (int i = 0; i < 5; i++) {
            try { buffer.consume(); Thread.sleep(300); }
            catch (InterruptedException e) {}
        }
    }
}

public class ProducerConsumerDemo {
    public static void main(String args[]) throws InterruptedException {
        SharedBuffer buf = new SharedBuffer();
        new Thread(new Producer(buf)).start();
        new Thread(new Consumer(buf)).start();
    }
}

RandNumber Example — Accumulating Sum

The producer generates random numbers and places them in a shared object. The consumer accumulates the running total until the sum reaches or exceeds 999, then stops both threads.

JAVA 
import java.util.Random;

class RandBuffer {
    private int number;
    private boolean ready = false;

    public synchronized void set(int n) throws InterruptedException {
        while (ready) wait();
        number = n;
        ready  = true;
        notifyAll();
    }

    public synchronized int get() throws InterruptedException {
        while (!ready) wait();
        ready = false;
        notifyAll();
        return number;
    }
}

class RandProducer implements Runnable {
    private final RandBuffer buf;
    private final int count;
    RandProducer(RandBuffer b, int c) { buf = b; count = c; }
    public void run() {
        Random rng = new Random();
        try {
            for (int i = 0; i < count; i++) {
                int val = rng.nextInt(200) + 1;
                buf.set(val);
                System.out.println("Produced: " + val);
            }
        } catch (InterruptedException e) {}
    }
}

class RandConsumer implements Runnable {
    private final RandBuffer buf;
    private final int count;
    RandConsumer(RandBuffer b, int c) { buf = b; count = c; }
    public void run() {
        int sum = 0;
        try {
            for (int i = 0; i < count; i++) {
                int val = buf.get();
                sum += val;
                System.out.println("Consumed: " + val + "  | Running sum: " + sum);
                if (sum >= 999) {
                    System.out.println("Sum reached " + sum + ". Consumer stopping.");
                    return;
                }
            }
        } catch (InterruptedException e) {}
    }
}

public class RandNumber {
    public static void main(String args[]) {
        RandBuffer buf = new RandBuffer();
        new Thread(new RandProducer(buf, 20)).start();
        new Thread(new RandConsumer(buf, 20)).start();
    }
}

📌 10. Byte vs Character Streams

Java I/O is organised into two parallel stream hierarchies. Byte streams deal with raw 8-bit binary data; character streams handle 16-bit Unicode text with automatic encoding conversion.

Stream Hierarchy Diagram

Byte Streams — read/write raw bytes (int 0–255). Use for: binary files (images, audio, compiled code), network protocols, serialization.
Character Streams — read/write Unicode characters. Use for: text files, CSV, JSON, XML. Handles charset encoding (UTF-8, ISO-8859-1, etc.) automatically.

ByteStreamExample — Copy File Byte by Byte

JAVA 
import java.io.FileInputStream;
import java.io.FileOutputStream;
import java.io.IOException;

public class ByteStreamExample {
    public static void main(String args[]) {
        try (
            FileInputStream  fis = new FileInputStream("input.bin");
            FileOutputStream fos = new FileOutputStream("output.bin")
        ) {
            int byteData;
            long count = 0;
            while ((byteData = fis.read()) != -1) {
                fos.write(byteData);
                count++;
            }
            System.out.println("Copied " + count + " bytes successfully.");
        } catch (IOException e) {
            System.err.println("Error: " + e.getMessage());
        }
    }
}
Copied 2048 bytes successfully.

CharStreamExample — Copy File Character by Character

JAVA 
import java.io.FileReader;
import java.io.FileWriter;
import java.io.IOException;

public class CharStreamExample {
    public static void main(String args[]) {
        try (
            FileReader fr = new FileReader("input.txt");
            FileWriter fw = new FileWriter("output.txt")
        ) {
            int charData;
            long count = 0;
            while ((charData = fr.read()) != -1) {
                fw.write(charData);
                count++;
            }
            System.out.println("Copied " + count + " characters successfully.");
        } catch (IOException e) {
            System.err.println("Error: " + e.getMessage());

Performance tip: Single-byte/char reads are slow. Wrap streams in BufferedInputStream / BufferedReader to use an internal 8 KB buffer, dramatically reducing system-call overhead. Or use transferTo() in Java 9+ for even faster copies.

📄 11. BufferedReader + PrintWriter

BufferedReader wraps any Reader and buffers input, enabling the key method readLine() which returns one text line at a time (stripping the newline). PrintWriter wraps any Writer or OutputStream and adds convenient println() and printf() methods.

Line-by-Line File Copy

JAVA 
import java.io.BufferedReader;
import java.io.FileReader;
import java.io.PrintWriter;
import java.io.IOException;

public class LineFileCopy {
    public static void main(String[] args) {
        String inputFilePath  = "source.txt";
        String outputFilePath = "destination.txt";

        try (
            BufferedReader bufferedReader = new BufferedReader(
                                                 new FileReader(inputFilePath));
            PrintWriter    printWriter    = new PrintWriter(outputFilePath)
        ) {
            String line;
            int    lineCount = 0;
            while ((line = bufferedReader.readLine()) != null) {
                printWriter.println(line);
                lineCount++;
            }
            System.out.println("Copied " + lineCount + " lines.");
        } catch (IOException e) {
            e.printStackTrace();
        }
        // try-with-resources: both streams closed automatically here
    }
}
Copied 42 lines.

try-with-resources Pattern

try-with-resources (Java 7+): any object declared in the try(...) parentheses that implements AutoCloseable is automatically closed when the block exits — whether by normal completion or by exception. This eliminates the common bug of forgetting to close streams in a finally block, and handles the tricky case where the close() itself throws.
JAVA 
// Old pattern (error-prone)
BufferedReader br = null;
try {
    br = new BufferedReader(new FileReader("file.txt"));
    // ... use br ...
} finally {
    if (br != null) br.close(); // easy to forget, may itself throw
}

// Modern pattern (try-with-resources)
try (BufferedReader br = new BufferedReader(new FileReader("file.txt"))) {
    // ... use br ...
} // br.close() called automatically — even if exception thrown

NP2Total after synchronized increments. A shared counter starts at 0. 5 threads each call a synchronized increment() method 200 times. Because the method is synchronized (no lost updates), what is the final counter value?
Synchronization guarantees every increment is applied exactly once:
total = threads * increments = 5 * 200 = 1000
Result = 1000.
NP3Sum across threads. Three worker threads each add the integers 1 through 100 into a shared, properly-synchronized total (so no updates are lost). What is the final total?
Sum 1..100 per thread, then times 3 threads:
sum(1..100) = 100 * 101 / 2 = 5050 total = 3 * 5050 = 15150
Result = 15150.
NP4Sleep / total time. A single thread runs a loop 8 times, calling Thread.sleep(250) once per iteration. Ignoring overhead, roughly how long (in milliseconds) does the thread spend sleeping in total?
total = 250 ms * 8 = 2000 ms
Result = 2000 ms.
NP5OutputStream.write(int) — byte written. OutputStream.write(int) writes only the least-significant byte of its argument (the upper 3 bytes are discarded). If you call out.write(700), what byte value (0–255) is actually written?
Keep only the low 8 bits → value mod 256:
700 mod 256 = 700 - 2*256 = 700 - 512 = 188
Result = 188.
NP6Writer.write(int) — character written. A character Writer.write(int) writes only the lower 2 bytes (16 bits) of its argument; the upper 2 bytes are discarded. If you call writer.write(70000), what code-point value (0–65535) is actually written?
Keep only the low 16 bits → value mod 65536:
70000 mod 65536 = 70000 - 65536 = 4464
Result = 4464.
NP7Bytes read from a stream. A text file contains exactly the ASCII string Multithreading (no newline). A byte-stream copy loop while ((b = in.read()) != -1) reads one byte per ASCII character until -1. How many bytes are read before the loop ends?
Each ASCII character is 1 byte. The word has 14 letters:
M-u-l-t-i-t-h-r-e-a-d-i-n-g = 14 characters = 14 bytes (the 15th read returns -1, ending the loop)
Result = 14 bytes.
NP8Reader.read() return range. Reader.read() returns one character as an int, or -1 at end-of-stream. What is the largest non-EOF value it can return?
A Java char is a 16-bit unsigned Unicode code unit, so read() returns a value in 0 .. 65535 (or -1 at end of stream). The maximum is 2¹⁶ − 1 = 65535.

📝 Exam-Style MCQs — 2 Marks Each

Each question is worth 2 marks — a mix of applied numerical and conceptual items. Click an option to check it; the correct answer and explanation appear instantly.

Q1 · 2 marks A shared counter starts at 0. 4 threads each call a synchronized increment() method 2500 times. Because the method is synchronized (no lost updates), what is the final counter value?
Synchronization makes each increment atomic, so every one of the increments is applied exactly once — no lost updates.
total = threads * increments = 4 * 2500 = 10000
Without synchronization the result would be unpredictable (the distractor D), but the method is synchronized. The answer is 10000.
Q2 · 2 marks Five worker threads each add the integers 1 through 50 into a shared, properly-synchronized total (no updates lost). What is the final total?
Sum 1..50 once, then multiply by the 5 threads.
sum(1..50) = 50 * 51 / 2 = 1275 total = 5 * 1275 = 6375
Forgetting to multiply by 5 gives 1275 (the distractor). The answer is 6375.
Q3 · 2 marks A single thread runs a loop 6 times, calling Thread.sleep(150) once per iteration. Ignoring overhead, roughly how long (in milliseconds) does the thread spend sleeping in total?
for (int i = 0; i < 6; i++) { Thread.sleep(150); }
The loop body runs 6 times, sleeping 150 ms each pass.
total = 150 ms * 6 = 900 ms
Using 5 iterations (off-by-one) gives 750 ms (the distractor). The answer is 900 ms.
Q4 · 2 marks A thread has been created and start() has been called, but the OS scheduler has not yet given it CPU time. Soon after it begins executing, it calls Thread.sleep(1000). Which two thread states are involved, in order?
After start(), a thread is RUNNABLE — eligible to run and waiting for the scheduler. Calling sleep(1000) moves it to TIMED_WAITING until the timeout elapses, after which it returns to RUNNABLE. (The five states are NEW, RUNNABLE, BLOCKED, WAITING/TIMED_WAITING, and TERMINATED.) The answer is B.
Q5 · 2 marks In a producer–consumer setup, the consumer must pause when the buffer is empty and resume only once the producer adds an item. Which mechanism implements this correctly inside synchronized code?
Inter-thread coordination uses wait()/notify() on the shared monitor. The waiting thread (consumer) calls wait(), which releases the lock and suspends it; the producer calls notify()/notifyAll() after producing, waking the consumer to re-check the condition. sleep() would not release the lock and yield() gives no guarantee. The answer is B.
Q6 · 2 marks You need to (i) copy a JPEG image file faithfully and (ii) read a UTF-8 text file line by line. Which stream choice is most appropriate for each task?
Byte streams handle raw binary data (images, audio) without any encoding transformation — essential so the JPEG is copied byte-for-byte. Character streams apply a charset (e.g. UTF-8) and are built for text, with helpers like BufferedReader.readLine(). Using character streams on a JPEG would corrupt it. The answer is C.
