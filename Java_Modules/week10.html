🆕 1. Introduction to Java Frameworks

Definition: A Java framework is a pre-built code skeleton (set of classes, interfaces, and conventions) that developers build upon. Instead of writing everything from scratch, you fill in the blanks and let the framework handle the plumbing.

Think of it like buying a house shell — the walls, roof, and foundation are done; you just decorate the inside to your needs.

Types of Java Frameworks

🌐 Web Frameworks

Handle HTTP requests, routing, templating. e.g., Spring MVC, Struts, JSF.

🗃 ORM Frameworks

Map Java objects to database tables. e.g., Hibernate, JPA, MyBatis.

✅ Testing Frameworks

Automate unit/integration tests. e.g., JUnit, TestNG, Mockito.

📄 Logging Frameworks

Structured application logging. e.g., Log4j, SLF4J, Logback.

4 Key Benefits of Using Frameworks

🔒

Security

Built-in protection against SQL injection, XSS, CSRF. Security best practices baked in by experts.

👥

Community Support

Large communities, extensive documentation, Stack Overflow answers, tutorials, and plugins.

♻

Code Reuse

Avoid reinventing the wheel. Reusable components accelerate development cycles dramatically.

📈

Scalability

Designed to handle growth — from small apps to enterprise systems serving millions of users.

Drawbacks of Frameworks

  • API Restrictions: You must follow the framework's rules and conventions — limited freedom to do things your own way.
  • Public Code: Open-source frameworks expose internals; potential vulnerabilities if not kept updated.
  • Encapsulation: Much of the logic is hidden inside the framework, making debugging harder and increasing the learning curve.

Popular Java Frameworks

Spring Hibernate Grails JSF (JavaServer Faces) GWT (Google Web Toolkit) Struts Micronaut Quarkus

🌿 2. Spring Framework Overview

Spring Framework is an open-source, lightweight, enterprise application framework for Java. Its primary goal is to manage business objects — wiring them together and handling cross-cutting concerns so you don't have to.

Created by Rod Johnson in 2003, Spring has grown into the most widely-used Java enterprise framework, with a vast ecosystem of sub-projects.

🔗 7. Coupling in Java

Why Coupling Matters

In real projects, requirements change, technologies are swapped, and frameworks are upgraded. How tightly your classes depend on each other determines how easily you can make those changes.

Analogy:
  • Loose coupling — like a bike seat cover. Easy to swap any time, doesn't affect the rest of the bike.
  • Tight coupling — like a bike engine. Welded in; replacing it requires dismantling half the bike.

Problem: Tight Coupling

Suppose we build an IPLTeam that is directly dependent on TeamCSK:

Java — Tight Coupling Problem
public class TeamCSK {
    public void captain() { System.out.println("MSD"); }
    public void ower()   { System.out.println("India Cements"); }
}

public class IPLTeam {
    TeamCSK team;     // TIGHTLY COUPLED — hard-coded to CSK!

    public IPLTeam(TeamCSK team) {
        this.team = team;
    }

    public void run() {
        team.captain();  // always prints MSD
        team.ower();     // always prints India Cements
    }
}

// Main
public class Main {
    public static void main(String[] args) {
        IPLTeam iplTeam = new IPLTeam(new TeamCSK());
        iplTeam.run();
        // To use TeamMI we'd have to change IPLTeam's type — BAD!
    }
}
MSD India Cements
Problem: To switch from CSK to MI, you must change IPLTeam's field type, constructor, and all call sites. The class is not reusable or extensible.

Solution: Loose Coupling via Interface

Extract the behaviour into an interface. IPLTeam now depends on the abstraction, not the concrete class:

Java — Interface-Based Loose Coupling
// Step 1: Define an interface (the contract)
public interface IPLConsole {
    void captain();
    void ower();
}

// Step 2: Implement for each team
public class TeamCSK implements IPLConsole {
    @Override public void captain() { System.out.println("MSD"); }
    @Override public void ower()   { System.out.println("India Cements"); }
}

public class TeamMI implements IPLConsole {
    @Override public void captain() { System.out.println("Ambani"); }
    @Override public void ower()   { System.out.println("Reliance"); }
}

// Step 3: IPLTeam depends on the INTERFACE, not the class
public class IPLTeam {
    IPLConsole team;   // LOOSELY COUPLED via interface

    public IPLTeam(IPLConsole team) {
        this.team = team;
    }

    public void run() {
        team.captain();
        team.ower();
    }
}

// Main — swap team with ZERO changes to IPLTeam!
public class Main {
    public static void main(String[] args) {
        IPLTeam csk = new IPLTeam(new TeamCSK());
        csk.run();   // MSD / India Cements

        IPLTeam mi = new IPLTeam(new TeamMI());
        mi.run();    // Ambani / Reliance
    }
}
--- CSK --- MSD India Cements --- MI --- Ambani Reliance

Beans with Records and Dependencies

Java — Records + Bean Declarations
// Record types — immutable data carriers (Java 16+)
record University(String uname, String city) {}
record Student(String name, int rollNo, University university) {}

@Configuration
public class AppConfig {

    // Approach 1: Direct — hardcode all values
    @Bean
    public University university() {
        return new University("VIT University", "Vellore");
    }

    // Approach 2: Call-by-method — call another bean method
    @Bean
    public Student studentByMethod() {
        return new Student("Alice", 101, university());
    }

    // Approach 3: Call-by-parameter — Spring injects the bean
    @Bean
    public Student studentByParam(University university) {
        return new Student("Bob", 102, university);
    }
}

// Retrieve and use
var context = new AnnotationConfigApplicationContext(AppConfig.class);

Student s1 = context.getBean("studentByMethod", Student.class);
Student s2 = context.getBean("studentByParam", Student.class);
System.out.println(s1);
System.out.println(s2);

// List ALL beans defined in context
Arrays.stream(context.getBeanDefinitionNames())
      .forEach(System.out::println);
Student[name=Alice, rollNo=101, university=University[uname=VIT University, city=Vellore]] Student[name=Bob, rollNo=102, university=University[uname=VIT University, city=Vellore]] --- All Bean Names --- org.springframework.context.annotation.internalConfigurationAnnotationProcessor org.springframework.context.annotation.internalAutowiredAnnotationProcessor appConfig university studentByMethod studentByParam

@Primary and @Qualifier

When multiple beans of the same type exist, Spring needs to know which one to inject:

Java — @Primary and @Qualifier
@Configuration
public class MultiConfig {

    @Bean
    @Primary  // preferred when no qualifier specified
    public Student studentPrimary() {
        return new Student("Primary-Alice", 1, null);
    }

    @Bean
    @Qualifier("secondStudent")
    public Student studentSecond() {
        return new Student("Second-Bob", 2, null);
    }
}

// Injection in another bean — get the second student explicitly
@Bean
public String report(@Qualifier("secondStudent") Student s) {
    return "Report for: " + s.name();
}

// getBean without qualifier returns @Primary
context.getBean(Student.class);     // → Primary-Alice
context.getBean("secondStudent", Student.class); // → Second-Bob
Primary bean: Primary-Alice Qualified bean: Second-Bob Report for: Second-Bob

🔁 9. Autowiring

Autowiring is Spring's ability to implicitly inject dependencies into beans — you don't have to manually wire beans together; Spring figures it out automatically based on your configuration.

4 Autowiring Modes

ModeHow It WorksUse Case
no Default. No autowiring — you wire dependencies manually in XML or with @Bean. When you need full explicit control over wiring.
byName Spring looks for a bean whose name matches the property name in the target bean. When property name and bean name follow the same convention.
byType Spring looks for a bean of the same type as the dependency. Fails if multiple beans of that type exist (use @Primary or @Qualifier). Most common modern approach with annotations.
constructor Similar to byType but applies to the constructor parameters. Spring injects matching beans into the constructor. Recommended for mandatory dependencies; supports immutability.

@Autowired — Field, Setter, Constructor

Java — @Autowired Examples
@Component
public class VehicleService {

    // Field injection (convenient but not recommended for testing)
    @Autowired
    private EngineService engineService;

    // Setter injection
    @Autowired
    public void setFuelService(FuelService fuelService) {
        this.fuelService = fuelService;
    }

    // Constructor injection (RECOMMENDED — immutable, testable)
    private final TyreService tyreService;

    @Autowired  // optional if only 1 constructor
    public VehicleService(TyreService tyreService) {
        this.tyreService = tyreService;
    }

    public String start() {
        return "Vehicle started with " + tyreService.getType();
    }
}
Spring context created. VehicleService injected with: EngineService, FuelService, TyreService Vehicle started with Tubeless

How Spring Resolves Autowiring

Step 1 — byType: Spring scans all beans for a type match to the dependency.
Step 2 — byName: If multiple matches, Spring tries to match by the property/parameter name.
Step 3 — Qualifier: If still ambiguous, @Qualifier("beanName") picks the exact bean. @Primary can also pre-select the default.

🌐 10. Spring MVC

Spring MVC (Model-View-Controller) is Spring's web framework. It follows the MVC design pattern, separating your application into three clearly defined layers: Model (data), View (presentation), and Controller (business logic).

Key Components

ComponentRole
DispatcherServlet Front Controller — a single servlet that intercepts all incoming HTTP requests and routes them to the correct handler. Configured in web.xml.
@Controller Marks a class as a Spring MVC controller. Methods inside handle specific URL requests and return view names or response data.
@RequestMapping Maps a URL pattern (and optionally HTTP method) to a specific controller method. Also @GetMapping, @PostMapping, etc.
ViewResolver Translates a view name (e.g., "home") returned by a controller into an actual view file (e.g., /WEB-INF/views/home.jsp).
Model A map-like object passed to the view. Controllers add attributes to the model; the JSP reads them using EL expressions like ${serverTime}.
HandlerMapping Determines which controller method handles a given request based on URL and method annotations.

HomeController.java

Java
package vit.com.example;

import java.util.*;
import org.springframework.stereotype.*;
import org.springframework.web.bind.annotation.*;
import org.springframework.ui.Model;
import org.springframework.validation.annotation.Validated;

@Controller
public class HomeController {

    /** GET /app → shows home.jsp with current server time */
    @RequestMapping(value = "/app", method = RequestMethod.GET)
    public String home(Locale locale, Model model) {
        Date date = new Date();
        DateFormat df = DateFormat.getDateTimeInstance(
                            DateFormat.LONG, DateFormat.LONG, locale);
        String formattedDate = df.format(date);
        model.addAttribute("serverTime", formattedDate);
        return "home";   // → /WEB-INF/views/home.jsp
    }

    /** GET /login → shows login.jsp */
    @RequestMapping(value = "/login", method = RequestMethod.GET)
    public String loginPage() {
        return "login";  // → /WEB-INF/views/login.jsp
    }

    /** POST /home → processes login form and shows user.jsp */
    @RequestMapping(value = "/home", method = RequestMethod.POST)
    public String login(@Validated User user, Model model) {
        model.addAttribute("userName", user.getUserName());
        model.addAttribute("password", user.getPassword());
        return "user";   // → /WEB-INF/views/user.jsp
    }
}
GET /app → renders home.jsp with serverTime attribute GET /login → renders login.jsp POST /home → renders user.jsp with userName and password attributes

NP1Singleton scope instance count. A bean uses Spring's default singleton scope. The container fetches it five times:
for (int i = 0; i < 5; i++) { context.getBean("name"); }
How many actual bean instances does the container create?
The default scope is singleton: exactly one shared instance per bean definition, no matter how many times getBean is called. All five calls return the same object.
Instances created = 1
Answer = 1.
NP2Counting @Bean definitions. How many beans does this @Configuration class register in the container?
@Configuration public class AppConfig { @Bean public String name() { return "RELU"; } @Bean public int rollNo() { return 2024; } @Bean public Student student() { ... } @Bean public Student student2() { ... } @Bean public University univ() { ... } @Bean(name="address") public University univ2() { ... } }
Each @Bean-annotated method contributes exactly one bean definition (the method name, or the explicit name=, becomes the bean id).
name, rollNo, student, student2, univ, address → 6 methods
Beans registered = 6.
NP3Retrieving a primitive bean. Given the config in NP2, what does this line print?
System.out.println(context.getBean("rollNo"));
getBean("rollNo") returns the value produced by the rollNo() @Bean method, which returns the literal 2024.
rollNo() returns 2024
Printed value = 2024.
NP4Dependency injection arithmetic. An @Bean unitPrice() returns 250 and an @Bean quantity() returns 12. A service is wired with both and computes the order total:
@Bean public int orderTotal() { return unitPrice() * quantity(); }
What value does getBean("orderTotal") return?
The wired dependencies supply 250 and 12; the service multiplies them.
250 * 12 = 3000
Order total = 3000.
NP5Autowiring modes. How many autowiring modes does the notes list for injecting a dependency (no, byName, byType, constructor)?
The four modes are no (default, none), byName, byType, and constructor.
no + byName + byType + constructor = 4
Count = 4.
NP6Service logic with integer division. A discount service is injected with price = 1200 and applies a 10% discount using Java integer math:
int price = 1200; int finalPrice = price - (price / 10);
What is finalPrice?
price / 10 is integer division.
1200 / 10 = 120 1200 - 120 = 1080
finalPrice = 1080.
NP7Counting MVC roles handled by the controller. In the Spring MVC HomeController from the notes, three @RequestMapping handler methods are defined — home() (GET /app), loginPage() (GET /login), and login() (POST /home). How many request-mapped handler methods is that?

home() + loginPage() + login() = 3

📝 Exam-Style MCQs — 2 Marks Each

Each question is worth 2 marks — a mix of applied numerical and conceptual items. Click an option to check it; the correct answer and explanation appear instantly.

Q1 · 2 marks A bean is defined with the default singleton scope. If context.getBean("car") is called 1000 times, how many instances of that bean does the Spring container create?
A singleton-scoped bean has exactly one shared instance per Spring container.
1000 getBean() calls → all return the same reference → instances created = 1
Each call returns the identical object, so the container creates 1 instance. (Prototype scope would create 1000.)
Q2 · 2 marks How many beans does this @Configuration class register in the container?
@Configuration class AppConfig { @Bean Engine engine() { return new Engine(); } @Bean Car car() { return new Car(engine()); } Wheel wheel() { return new Wheel(); } // no @Bean @Bean Driver driver() { return new Driver(); } }
Only methods annotated with @Bean become container-managed beans.
@Bean methods: engine, car, driver → 3 · wheel() has no @Bean → not registered
So 3 beans are registered. The plain wheel() method is ignored by the container.
Q3 · 2 marks A BillingService receives an injected unit price of 500 and a quantity of 3, then applies a 10% discount. What total does it return?
double total = price * qty; // 500 * 3 double payable = total - total * 0.10; return payable;
The dependency-injected values feed the service logic.
total = 500 × 3 = 1500 → discount = 1500 × 0.10 = 150 → payable = 1500 − 150 = 1350.0
The method returns 1350.0.
Q4 · 2 marks Which statement best describes Inversion of Control (IoC) as implemented by Spring?
With IoC, the responsibility for object creation and wiring is handed over to the Spring container instead of the class controlling it with new. Dependency Injection is the mechanism that realises IoC — the container supplies collaborators (via constructor or setter). This promotes loose coupling. The answer is the container creates objects and injects dependencies.
Q5 · 2 marks Two beans implement the same interface PaymentService. When @Autowired tries to inject one of them, what is the correct way to remove the ambiguity by naming a specific bean at the injection point?
When multiple candidates match a type, Spring throws NoUniqueBeanDefinitionException. @Qualifier("beanName") at the injection point selects one specific bean by name. @Primary sets a default winner but must mark exactly one bean (not both). The precise per-injection choice is @Qualifier("upiPayment").
Q6 · 2 marks In Spring MVC, which component is the front controller that receives every incoming request and delegates it to the appropriate handler?
Spring MVC follows the Front Controller pattern: the DispatcherServlet is the single entry point that consults the HandlerMapping, invokes the matching @Controller, then uses a ViewResolver to render the view. The controller, view resolver, and ModelAndView are downstream pieces it orchestrates. The answer is DispatcherServlet.
