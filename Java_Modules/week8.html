2. CGI vs Servlets

CGI — Common Gateway Interface

One new process per request. Each incoming HTTP request spawns a completely new OS process (shell/executable). When the request is done, that process terminates.
  • Heavyweight: process creation is expensive
  • No shared memory between requests
  • Does not scale under high load
  • Any language can be used (Perl, Python, C...)
  • Historically the first dynamic web technology

Servlets

One new thread per request. All requests share a single JVM instance. The servlet container manages a thread pool — threads are lightweight and reusable.
  • Lightweight: threads share JVM heap
  • Persistent objects across requests
  • Scales well under heavy load
  • Java only, but powerful APIs
  • Supports sessions, cookies, JDBC out-of-the-box

3. MVC Architecture

MVC (Model-View-Controller) is a software design pattern that separates an application into three interconnected components. It decouples business logic from the user interface, making code easier to maintain, test, and scale.

Model

Business logic and data management. Plain Java classes (POJOs). Interacts with the database. Notifies controller of state changes.

Data Logic

View

Presentation layer. Renders data to the user as HTML. No business logic. Typically JSP, HTML templates, or Thymeleaf in Java.

HTML JSP

Controller

Interface between Model and View. Receives user requests, invokes model operations, selects the appropriate view to render.

Servlet Flow

MVC Data Flow Diagram

Request Processing Steps

1
User Request — User submits a form or clicks a link; browser sends HTTP request to the server.
2
Controller receives — Servlet reads request parameters, determines what action to take.
3
Controller invokes Model — Calls model methods to retrieve or update data (queries database).
4
Model returns data — Business objects are returned to the controller.
5
Controller selects View — Forwards data to the appropriate JSP / HTML template.
6
View renders response — HTML page is generated and sent back to the user's browser.

4. MVC Example — Employee

A classic MVC example in pure Java (no web layer) to demonstrate the pattern. The Model holds employee data, the View prints it, and the Controller mediates between them.

Model — Employee.java

Java — Model 
// Model: holds employee data (POJO)
public class Employee {
    private String EmpName;
    private String EmpId;

    public String getEmpName() { return EmpName; }
    public void setEmpName(String EmpName) {
        this.EmpName = EmpName;
    }
    public String getEmpId() { return EmpId; }
    public void setEmpId(String EmpId) {
        this.EmpId = EmpId;
    }
}
// Model compiled. No output — just a data class.

View — EmployeeView.java

Java — View 
// View: displays employee details
public class EmployeeView {
    public void displayEmpDetails(String EmpName, String EmpId) {
        System.out.println("Employee Details: ");
        System.out.println("Name: " + EmpName);
        System.out.println("Employee ID: " + EmpId);
    }
}
// View compiled. Awaiting data from Controller.

Controller — EmployeeController.java

Java — Controller 
// Controller: mediates between Model and View
public class EmployeeController {
    private final Employee model;
    private final EmployeeView view;

    public EmployeeController(Employee model, EmployeeView view) {
        this.model = model;
        this.view = view;
    }

    public void setEmpName(String name) { model.setEmpName(name); }
    public String getEmpName() { return model.getEmpName(); }
    public void setEmpId(String id) { model.setEmpId(id); }
    public String getEmpId() { return model.getEmpId(); }

    public void updateView() {
        view.displayEmpDetails(model.getEmpName(), model.getEmpId());
    }
}
// Controller compiled and ready to orchestrate Model + View.

Client — MVCExample.java

5. Servlet Lifecycle

The servlet container (e.g., GlassFish, Tomcat) fully manages the lifecycle of a servlet. Developers override specific methods to plug in custom behaviour at each phase.

Lifecycle Diagram

Five Phases in Detail

1
Loading & Instantiation: The servlet container loads the servlet class (via ClassLoader) and creates a single instance using the no-arg constructor. This happens once, usually on first request or server startup if <load-on-startup> is set.
2
Initialisation — init(ServletConfig config): Called once after instantiation. The container passes a ServletConfig object containing initialisation parameters from web.xml. Override this to set up database connections, thread pools, caches, etc.
3
Request Handling — service(HttpServletRequest req, HttpServletResponse res): Called for every HTTP request. The default implementation in HttpServlet dispatches to doGet(), doPost(), doPut(), doDelete(), etc., based on the HTTP method.
4
Destruction — destroy(): Called once when the container decides to remove the servlet (server shutdown, redeploy, idle timeout). Release all resources here: close DB connections, stop background threads, flush caches.
5
Garbage Collection: After destroy() returns, the servlet instance is eligible for garbage collection. The JVM reclaims memory. The container may reload the servlet fresh on the next request.
Key point: Only one servlet instance exists per JVM (by default). Multiple threads concurrently call service() on this same instance. This is why servlet instance variables are dangerous — use local variables or synchronisation for thread safety.

web.xml — Servlet Mapping

XML — WEB-INF/web.xml 
<?xml version="1.0" encoding="UTF-8"?>
<web-app xmlns="http://xmlns.jcp.org/xml/ns/javaee"
         version="3.1">

    <!-- Declare the servlet -->
    <servlet>
        <servlet-name>helloServlet</servlet-name>
        <servlet-class>helloServlet</servlet-class>
    </servlet>

    <!-- Map a URL pattern to the servlet -->
    <servlet-mapping>
        <servlet-name>helloServlet</servlet-name>
        <url-pattern>/helloServlet</url-pattern>
    </servlet-mapping>

</web-app>
web.xml parsed. URL /helloServlet mapped to class helloServlet. Servlet container will invoke helloServlet for all matching requests.

helloServlet.java — Processing Request

Java — helloServlet.java 
import java.io.*;
import javax.servlet.*;
import javax.servlet.http.*;

@WebServlet("/helloServlet")
public class helloServlet extends HttpServlet {

    @Override
    protected void processRequest(HttpServletRequest request,
                                   HttpServletResponse response)
            throws ServletException, IOException {

        response.setContentType("text/html;charset=UTF-8");

        try (PrintWriter out = response.getWriter()) {
            out.println("<!DOCTYPE html>");
            out.println("<html><head><title>Hello Servlet</title></head>");
            out.println("<body>");
            out.println("<h1>Hello from helloServlet!</h1>");
            out.println("</body></html>");
        }
    }

    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse res)
            throws ServletException, IOException {
        processRequest(req, res);
    }

    @Override
    protected void doPost(HttpServletRequest req, HttpServletResponse res)
            throws ServletException, IOException {
        processRequest(req, res);
    }
}
GlassFish 4.1.1 started on port 8080 Servlet helloServlet loaded at /FirstServlet/helloServlet HTTP GET /FirstServlet/helloServlet → 200 OK Response: <h1>Hello from helloServlet!</h1>
Tip: In modern Servlet 3.0+ you can use the @WebServlet annotation instead of web.xml mappings. Both approaches achieve the same result. NetBeans generates both by default.

loginServlet.java — Reading Form Parameters

Java — loginServlet.java 
import java.io.*;
import javax.servlet.*;
import javax.servlet.http.*;

@WebServlet("/loginServlet")
public class loginServlet extends HttpServlet {

    @Override
    protected void doGet(HttpServletRequest request,
                          HttpServletResponse response)
            throws ServletException, IOException {

        // Read form parameters
        String username = request.getParameter("username");
        String password = request.getParameter("password");

        response.setContentType("text/html;charset=UTF-8");
        PrintWriter out = response.getWriter();

        out.println("<h2>Login Details</h2>");
        out.println("<p>Entered User Name: <b>"
                    + username + "</b></p>");
        out.println("<p>Entered Password: <b>"
                    + password + "</b></p>");
    }
}

9. Navigation — RequestDispatcher

RequestDispatcher is an interface that enables one servlet to hand off control to another servlet, JSP, or HTML resource. It supports two key operations:

forward()

Completely transfers request/response control to the target resource. The original servlet can no longer write to the response after calling forward(). The client browser URL does not change (server-side redirect).
Java — forward() 
// In Servlet-I doGet()
RequestDispatcher rd =
    request.getRequestDispatcher("/Servlet2");

// Optionally pass data as attribute
request.setAttribute("message", "Hello from S1");

// Forward — control moves to Servlet2
rd.forward(request, response);

// Code here is NOT executed after forward
Servlet-I: request received, forwarding... Servlet-II: took over, generating response Client sees: response from Servlet-II only Browser URL: unchanged (still shows /Servlet1)

include()

Includes the output of another resource in the current response. Control returns to the calling servlet after the included resource finishes. Used to compose pages from reusable header/footer components.
Java — include() 
// In Servlet-I doGet()
PrintWriter out = response.getWriter();
out.println("<h1>From Servlet-I (before)</h1>");

RequestDispatcher rd =
    request.getRequestDispatcher("/Servlet2");

// Include Servlet2 output here
rd.include(request, response);

out.println("<p>Back in Servlet-I (after)</p>");
<h1>From Servlet-I (before)</h1> [output from Servlet-II inserted here] <p>Back in Servlet-I (after)</p> Combined response sent to client.

10. Session Tracking — 4 Techniques

HTTP is stateless — each request is independent and the server has no memory of previous interactions. Session tracking techniques maintain state across multiple requests from the same user.

1. Cookies

Small name-value pairs stored by the browser. The server sends a Set-Cookie header; the browser sends them back on subsequent requests.

Java — Cookies 
// Create and send cookie
Cookie c = new Cookie("username", "PrakashP");
c.setMaxAge(60 * 60 * 24); // 1 day (persistent)
response.addCookie(c);

// Read cookies from browser
Cookie[] cookies = request.getCookies();
for (Cookie ck : cookies) {
    if (ck.getName().equals("username")) {
        String user = ck.getValue();
    }
}
Set-Cookie: username=PrakashP; Max-Age=86400; Path=/ Next request includes: Cookie: username=PrakashP Value read: PrakashP

Session vs Persistent Cookies

TypeLifetimeStorage
Session cookieBrowser session (tab close = gone)Memory
Persistent cookiesetMaxAge(seconds)Disk file

2. Hidden Form Field

Embed user state in a hidden HTML input. The data is sent back to the server with each form submission. Works even if cookies are disabled, but only survives form submissions.

HTML — hidden field 
<!-- In the response HTML from Servlet -->
<form method="post" action="nextPage">

  <!-- Hidden: carries user identity silently -->
  <input type="hidden"
         name="username"
         value="PrakashP"/>

  <input type="text" name="city"
         placeholder="Your city"/>
  <input type="submit" value="Next"/>
</form>
Form submitted: POST /nextPage Parameters: username=PrakashP, city=Chennai Servlet reads: request.getParameter("username") → "PrakashP"
Limitation: Only works for page flows that use forms. Not suitable for tracking across navigation links or non-form requests.

3. URL Rewriting

Append session data as query parameters to every URL. The server encodes user info into every link it generates. Works without cookies but exposes data in URLs.

URL Rewriting Example 
// URL with appended session data
http://example.com/profile?username=vit&password=pass

// Every link in the page is rewritten:
<a href="cart?username=vit&sessionId=ABC123">
  View Cart
</a>

// In servlet: use encodeURL for automatic rewriting
String url = response.encodeURL("/cart");
out.println("<a href='" + url + "'>Cart</a>");
URL encoded: /cart;jsessionid=ABC123XYZ789 Works even if cookies are disabled. Warning: session ID visible in browser history & server logs.
Security note: Exposing session IDs in URLs can lead to session hijacking. Prefer HttpSession for sensitive applications.

4. HttpSession

The most powerful and preferred technique. The servlet container creates a unique session object per user, identified by a session ID (usually stored in a cookie). Attributes are stored server-side.

Java — HttpSession 
// Get (or create) session for this user
HttpSession session = request.getSession();

// Store attributes in session
session.setAttribute("username", "PrakashP");
session.setAttribute("role", "admin");
session.setMaxInactiveInterval(1800); // 30 min

// In another servlet — retrieve attributes
HttpSession session = request.getSession(false);
if (session != null) {
    String user = (String) session.getAttribute("username");
}

// Destroy session (logout)
session.invalidate();
Session created: ID = A3F7B2C1D9E4... setAttribute("username", "PrakashP") → stored server-side Browser stores: JSESSIONID=A3F7B2C1D9E4... (cookie) Next request: container matches JSESSIONID → returns same session getAttribute("username") → "PrakashP" session.invalidate() → session destroyed (logout)

Q1 · 2 marks A page-counter servlet stores a hit count in the HttpSession. On the first request it initialises the count to 1, and on every later request it does count = count + 1. After the same browser session makes 6 requests, what value does count hold?
The session persists across requests, so the count accumulates one per request.
req1→1, req2→2, … req6→6
count = 6 (answer B). Option D would be the bug of using a request attribute instead of the session.
Q2 · 2 marks A single servlet instance receives 25 requests from various clients over its lifetime. By default (single instance, multithreaded), how many times does the container call its init() method?
init() runs once when the servlet is first loaded; service() (then doGet/doPost) runs once per request, and destroy() runs once at unload.
init() calls = 1 (regardless of the 25 requests)
Answer: 1 (A). 25 would be the number of service() calls.
Q3 · 2 marks You want a "Remember me" cookie to live for exactly 7 days. What value (in seconds) should you pass to cookie.setMaxAge(...)?
Cookie c = new Cookie("user", name); c.setMaxAge( /* 7 days in seconds */ ); response.addCookie(c);
setMaxAge takes seconds.
7 × 24 × 60 × 60 = 604800 s
Answer: 604800 (C). Options A and B stop at hours and minutes respectively.
Q4 · 2 marks Place these servlet lifecycle methods in the order the container invokes them across the servlet's life.
The lifecycle is: init() once at load → service() for each request (dispatching to doGet/doPost) → destroy() once at unload. Answer: B.
Q5 · 2 marks A servlet processes a login POST and then needs the browser's address bar to change to /dashboard on a fresh GET request (so a refresh won't re-submit the form). Which navigation mechanism is correct?
sendRedirect sends a 302 to the browser, which issues a brand-new GET — so the URL updates and a refresh re-runs the harmless GET (the Post/Redirect/Get pattern). forward/include are server-side and leave the URL unchanged. Answer: C.
Q6 · 2 marks Which of the following is not one of the four standard servlet session-tracking techniques?
The four classic techniques are cookies, URL rewriting, hidden form fields, and the HttpSession API. Background DB polling is not a session-tracking mechanism. Answer: D.
