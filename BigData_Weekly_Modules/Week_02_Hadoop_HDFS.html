1️⃣ Introduction to Hadoop

Apache Hadoop is an open-source framework for storing and processing massive datasets across clusters of commodity hardware. It's the foundation of the modern Big Data ecosystem.

Two Core Components

🗄️ HDFS (Hadoop Distributed File System)

Storage layer — stores files across many machines, with built-in replication for reliability.

⚙️ MapReduce / YARN

Processing layer — distributes computation across the cluster (covered in Week 3).

💡 Key idea: Hadoop moves computation to the data (data locality), instead of moving large data to compute. This drastically reduces network traffic.

2️⃣ Core Features of Hadoop

🌐 Distributed Processing

Tasks split across many nodes — work happens in parallel across the cluster.

📈 Scalability

Handles large datasets; scales by adding more nodes (horizontal scaling).

🛡️ Fault Tolerance

Data replicated across multiple nodes — survives hardware failures gracefully.

📖 Open Source

Free, large community, extensive tooling, customizable.

3️⃣ Advantages of Hadoop

💰 Cost-Effective

Uses cheap commodity hardware instead of expensive specialized servers.

🚀 High Throughput

Optimized for processing large amounts of data quickly via parallelism.

📍 Data Locality

Hadoop moves computation to the data — minimizes network traffic.

🎭 Versatility

Can store & process structured, semi-structured, and unstructured data.

Data Locality Example: If a 100GB log file is split across DataNodes A, B, C — Hadoop sends the MapReduce code to A, B, C to process locally instead of sending all 100GB over the network to one machine.

4️⃣ History of Hadoop

Origins

Hadoop is inspired by two seminal Google papers:

  • Google File System (GFS) paper (2003) — distributed file system design
  • MapReduce paper (2004) — programming model for processing large data

Doug Cutting and Mike Cafarella created Hadoop in 2005 (named after Cutting's son's toy elephant 🐘). It became an Apache project in 2008.

Key Milestones

VersionYearKey Change
Hadoop 1.x2011MapReduce + HDFS; JobTracker bottleneck
Hadoop 2.x2013YARN introduced; supports non-MR frameworks
Hadoop 3.x2017Erasure coding, GPU support, multiple NameNodes

5️⃣ HDFS (Hadoop Distributed File System)

HDFS is the primary data storage system for Hadoop applications. Designed to:

  • Store very large files reliably
  • Provide high-throughput access to data
  • Work across a distributed cluster of commodity servers
  • Detect and recover from hardware failures automatically

HDFS Design Principles

📦 Large Files

Optimized for files of GB to TB sizes (not millions of small files).

📝 Write-Once, Read-Many

Files are written once then read many times — append-only model.

🌊 Streaming Access

Optimized for reading entire files sequentially — not random access.

💻 Commodity Hardware

Assumes hardware will fail — builds in redundancy.

6️⃣ HDFS Architecture — Master/Slave

HDFS uses a master-slave architecture:

👑 NameNode
(Master)
Metadata only — filenames, permissions, block locations
🔧 DataNode 1
Stores actual blocks
🔧 DataNode 2
Stores blocks + replicas
🔧 DataNode 3
3rd replica (RF=3 default)

NameNode — The Master

📋 Manages Namespace

The file system namespace (directory tree)

📚 Stores Metadata

File permissions, sizes, locations of blocks

🚫 No Actual Data

Does NOT store the actual data — only the index

⚠️ Single Point of Failure

In Hadoop 1.x. Hadoop 2.x added standby NameNode for HA.

DataNodes — The Workers

💾 Store Blocks

Actual data is stored as blocks on the local filesystem

🔄 Replicate Data

Each block stored on multiple DataNodes (default: 3 copies)

📨 Serve R/W

Handle read/write requests directly from clients

💓 Heartbeat

Send heartbeats to NameNode to signal they're alive

🛠️ Block Creation / Deletion / Replication

Perform block creation, deletion, and replication on command from the NameNode

ASCII Architecture Diagram

+-------------------+ | NameNode | | (Metadata) | +-------------------+ | +----+----+----+ | | | | +------+ +------+ +------+ |Data | |Data | |Data | |Node1 | |Node2 | |Node3 | |Block1| |Block1| |Block1| |Block2| |Block3| |Block2| |Block4| |Block4| |Block3| +------+ +------+ +------+ ^ ^ ^ +-- Replication ---+ ^ +---------+ | Client | +---------+

Key Advantages of HDFS Architecture

🛡️ Fault Tolerance

Replication ensures data not lost on DataNode failure

📈 Scalability

Add more DataNodes → more storage and throughput

🚀 High Throughput

Parallel reads from multiple DataNodes

📍 Data Locality

Compute happens where data resides

7️⃣ Blocks, Replication & Fault Tolerance

HDFS Blocks

  • HDFS breaks files into blocks — the smallest storage unit
  • Default block size: 128 MB (configurable, often 256 MB)
  • Storing in blocks lets HDFS distribute large files across DataNodes
Example: A 500 MB file with 128 MB blocks splits into 4 blocks (128 + 128 + 128 + 116 MB), each stored on different DataNodes with replicas.

Replication Factor

  • Each block is replicated across multiple DataNodes
  • Replication factor determines how many copies — default = 3
  • If a DataNode fails, data remains accessible from other replicas

Replica Placement Strategy

  1. First replica — on local DataNode (where writer is)
  2. Second replica — on a different rack
  3. Third replica — on the same rack as the second (different node)

This balances reliability (rack failures) with bandwidth efficiency.

Why Larger Block Size?

💼 Less Metadata

Fewer blocks → smaller NameNode memory footprint

⚡ Less Seek Overhead

More time spent reading data, less seeking

📦 Better for Big Files

Reduces management overhead for huge files

8️⃣ HDFS Write Operation — Step by Step

Client Request: Client sends a write request to the NameNode for a new file.
NameNode Check: NameNode verifies file doesn't exist & client has write permission.
DataNode Allocation: NameNode returns a list of DataNodes for storing blocks (selected by proximity, available storage, and replication factor).
Data Streaming (Pipeline):
  • Client divides file into blocks
  • Streams data to the first DataNode
  • First DataNode stores its portion AND forwards to the next DataNode in the pipeline
  • Process repeats until all DataNodes in pipeline have the block
Acknowledgement: Each DataNode sends ACK back through the pipeline. The first DataNode sends final ACK to the client.
NameNode Update: After receiving all ACKs, client tells NameNode the write is complete. NameNode updates metadata with file info and block locations.
💡 Why a pipeline? Instead of the client copying the same block 3 times to 3 DataNodes (3× upload bandwidth), the data flows through a chain: Client → DN1 → DN2 → DN3 (1× upload bandwidth from client).

9️⃣ HDFS Read Operation — Step by Step

Client Request: Client sends a read request to the NameNode.
NameNode Lookup: NameNode checks its metadata to find block locations.
DataNode Information: NameNode returns a list of DataNode locations for each block. Multiple locations per block (due to replication).
Data Retrieval: Client contacts DataNodes directly (parallel reads, preferring the closest DataNode to minimize latency).
Data Assembly: DataNodes stream blocks to client. Client assembles blocks in correct order to reconstruct the file.

Key Considerations for Both Operations

ConceptDescription
Fault ToleranceIf a DataNode fails, data read from another replica
Data LocalityHDFS tries to read from the closest DataNode
Concurrency ControlHDFS supports concurrent reads. Writes are exclusive — only one writer per file to maintain data consistency
PipeliningWrites use a DataNode pipeline (DN1 → DN2 → DN3) instead of separate connections — saves client bandwidth and improves write efficiency

🔟 HDFS Commands

General Command Structure

hdfs dfs <command> <options> <arguments>

(hadoop fs works interchangeably.)

File & Directory Management

hdfs dfs -ls / # list root hdfs dfs -ls /user/data # list directory hdfs dfs -mkdir /user/data # create directory hdfs dfs -mkdir -p /a/b/c # create with parents hdfs dfs -rm /user/data/file.txt # remove file hdfs dfs -rm -r /user/data # recursive remove hdfs dfs -rmdir /empty_dir # remove empty dir hdfs dfs -mv /source /dest # move/rename hdfs dfs -cp /source /dest # copy

Uploading & Downloading

hdfs dfs -put local.txt /user/data/ # upload local → HDFS hdfs dfs -copyFromLocal local.txt /data/ # same as put hdfs dfs -get /user/data/file.txt ./ # download HDFS → local hdfs dfs -copyToLocal /data/file.txt ./ # same as get

Viewing File Content

hdfs dfs -cat /user/data/file.txt # full content hdfs dfs -tail /user/data/file.txt # last 1KB hdfs dfs -head /user/data/file.txt # first 1KB

Other Useful Commands

hdfs dfs -du /user/data # disk usage hdfs dfs -du -h /user/data # human-readable hdfs dfs -count /user/data # count files/dirs/bytes hdfs dfs -chmod 755 /user/data/file.txt # permissions hdfs dfs -chown user:group /user/data # owner hdfs dfs -chgrp group1 /user/data/file2.txt # change group

HDFS Admin Commands

hdfs dfsadmin -report # cluster status hdfs fsck / # filesystem check hdfs balancer # rebalance data hdfs dfsadmin -safemode get # check safe mode

1️⃣1️⃣ Hadoop Ecosystem

Hadoop is more than HDFS + MapReduce. The ecosystem includes many specialized tools:

Visual Representations of the Ecosystem

Two common ways to depict the Hadoop ecosystem (both highlighted in the course PDF):

(a) Layered Diagram

HDFS at the bottom (storage); YARN above HDFS (resource management); processing frameworks (MapReduce, Spark, Pig, Hive) above YARN; other components (HBase, Flume, Sqoop, ZooKeeper) placed appropriately by function.

+----------------------------------------------------+ | Hive | Pig | HBase | Flume | Sqoop | ZooKeeper | <- Tools +----------------------------------------------------+ | MapReduce | Spark | Tez | <- Processing +----------------------------------------------------+ | YARN | <- Resource Mgmt +----------------------------------------------------+ | HDFS | <- Storage +----------------------------------------------------+

(b) Graph-Based Diagram

Nodes represent each component; edges show relationships and data flow between them — e.g., Sqoop → HDFS → Hive → BI Tool, or Flume → HDFS → Spark → HBase.

[RDBMS] [Log Sources] | | Sqoop Flume | | v v +----------------------------+ | HDFS | <- central storage hub +----------------------------+ | | | v v v Hive Spark MapReduce ZooKeeper | | (coordinates v v HDFS HA & [BI Tool] HBase <--- Spark HBase) | v [Real-time App] Legend: A --> B means "A sends / feeds data to B"
📌 Key Points the diagrams emphasise:
  • Interoperability: Components are designed to work together as a comprehensive platform.
  • Evolving ecosystem: New projects and tools are added continuously.
  • Choosing the right tools: The tool selection depends on specific application requirements.

1. Core Components

🗄️ HDFS

Storage layer — base of everything

⚙️ YARN (Yet Another Resource Negotiator)

Manages CPU + memory across cluster. Allows MR + Spark + Tez to coexist. Components: ResourceManager, NodeManager, ApplicationMaster, Container.

🔀 MapReduce

Original parallel processing framework. Map + Reduce phases.

2. Data Processing

⚡ Spark

Fast, general-purpose in-memory processing. Batch + streaming + ML + SQL.

🐷 Pig

High-level data flow language (Pig Latin). Translates to MapReduce.

🐝 Hive

Data warehouse with SQL-like queries (HiveQL). Translates to MR/Tez/Spark.

3. NoSQL & Storage

📊 HBase

Column-oriented NoSQL DB on HDFS. Real-time R/W, large sparse tables.

4. Data Ingestion

🌊 Flume

Distributed service for collecting & moving large amounts of log data.

🔄 Sqoop

Transfers data between Hadoop and relational databases (MySQL, Oracle, PostgreSQL).

5. Coordination

🐘 ZooKeeper

Centralized service for distributed configuration, synchronization, and naming. Used for HDFS NameNode HA, HBase coordination.

📝 Summary & Exam Tips

🎯 Key Takeaways

  • Hadoop = open-source framework for distributed storage + processing of large datasets
  • HDFS = master-slave: 1 NameNode (metadata) + many DataNodes (actual data)
  • Block size = 128 MB default; Replication factor = 3 default
  • Write: 6 steps, uses pipeline; Read: 5 steps, direct from DataNodes
  • Data locality: compute moves TO data (key Hadoop principle)
  • Ecosystem layers: HDFS → YARN → MR/Spark → Hive/Pig/HBase + Flume/Sqoop + ZooKeeper
  • Hadoop 2.x added YARN — decoupled resource mgmt from MR
📌 Common Exam Pitfalls:
  • Saying NameNode stores data (it stores ONLY metadata)
  • Forgetting that writes use a pipeline (not separate connections)
  • Confusing Flume (logs) vs Sqoop (RDBMS)
  • Confusing Pig (Pig Latin) vs Hive (HiveQL/SQL-like)
