1️⃣ MapReduce Overview

MapReduce is a programming model + associated implementation for processing large datasets in parallel across a distributed system.

The Core Idea

📦 Big Dataset
split into
🧩 Chunks
processed
⚡ in parallel
combined
📊 Final Result

Why MapReduce?

🚀 Parallelism

Hundreds/thousands of machines work simultaneously

🛡️ Fault Tolerance

Failed tasks automatically re-run on other nodes

📍 Data Locality

Computation moves to where data lives (in HDFS)

📈 Scalability

Linear scaling with cluster size

💡 Origin: MapReduce was published by Google in 2004. Inspired by functional programming concepts (map and reduce functions in Lisp).

2️⃣ The Three Stages of MapReduce

① Map
transform
→
② Shuffle & Sort
group by key
→
③ Reduce
aggregate

① Map Stage

  • Input data is split into independent chunks
  • Each chunk is processed by a Mapper
  • Mapper takes input (key, value) pairs and emits intermediate (key, value) pairs
  • Mappers run in parallel across many nodes

② Shuffle & Sort Stage

  • Intermediate pairs from all Mappers are shuffled (transferred across the network) to Reducer nodes
  • Pairs with the same key are grouped together
  • Keys are sorted
  • This stage happens automatically — you don't write code for it

③ Reduce Stage

  • Each Reducer receives a key and the list of all values for that key
  • Reducer aggregates/processes these values
  • Produces the final output
Visual Example — Word Count:
Input: "Deer Bear River | Car Car River | Deer Car Bear" Map: "Deer Bear River" → (Deer,1)(Bear,1)(River,1) "Car Car River" → (Car,1)(Car,1)(River,1) "Deer Car Bear" → (Deer,1)(Car,1)(Bear,1) Shuffle/Sort groups same keys: Bear → [1,1] Car → [1,1,1] Deer → [1,1] River → [1,1] Reduce: sum values per key Bear=2, Car=3, Deer=2, River=2

3️⃣ Reducer in Detail

The Reducer's primary role is to process the intermediate data generated by the Mappers.

What the Reducer Does

  • Takes shuffled and sorted data as input
  • Processes each key + list of values
  • Produces the final output

Important: Number of Reducers

  • Configurable — set by the developer
  • Impacts performance and scalability
  • Too few → bottleneck; too many → overhead
  • Rule of thumb: 0.95 × (nodes × max reducers per node)
Reducer Logic (Pseudocode):
reduce(key, [value1, value2, ...]): aggregated = combine_function(value1, value2, ...) emit (key, aggregated)

4️⃣ Combiner — The Mini-Reducer

A Combiner is an optional optimization that performs local reduction on the Mapper's output BEFORE shuffle.

Purpose

  • Reduces the volume of data shuffled across the network
  • Improves MapReduce job efficiency
  • Effectively a "mini-Reducer" run on each Mapper's local output
Without Combiner:
Mapper 1: (Deer,1)(Deer,1)(Bear,1) → shuffle 3 pairs
With Combiner:
Mapper 1: (Deer,1)(Deer,1)(Bear,1) Combiner: (Deer,2)(Bear,1) ← already locally aggregated Shuffle: only 2 pairs to send
⚠️ Combiner Caveats:
  • Combiner output must have same types as Reducer input
  • Function must be associative and commutative (e.g., sum, min, max — NOT average)
  • Combiner may run zero, one, or many times — your code can't depend on it running

5️⃣ Partitioner — Routing Keys to Reducers

The Partitioner determines which Reducer a given intermediate (key, value) pair should be sent to.

How It Works

  • Divides the key space among Reducers
  • All values with the same key MUST go to the same Reducer
  • Default: HashPartitioner uses hash(key) % numReducers
// Default HashPartitioner partition = key.hashCode() % numReducers

Custom Partitioner

You can write a custom partitioner to:

  • Group related keys to the same Reducer (e.g., all keys starting with same letter)
  • Distribute keys evenly across Reducers (avoid skew)
  • Implement custom business logic
Example — Custom Partitioner for sales by region:
if key starts with "US" → Reducer 0 if key starts with "EU" → Reducer 1 if key starts with "APAC" → Reducer 2

6️⃣ Classic MapReduce Framework (Hadoop 1.x)

Key Components

  1. Client Node — submits the job
  2. JobTracker (master) — schedules + manages all jobs
  3. TaskTrackers (workers) — execute Map/Reduce tasks
  4. HDFS — provides input/output storage

Workflow (6 Phases)

Job Submission — Client uploads code + config to HDFS, submits to JobTracker
Job Scheduling — JobTracker assigns tasks to TaskTrackers based on data locality
Map Phase — TaskTrackers run Map tasks on their assigned chunks
Shuffle & Sort Phase — Intermediate data redistributed by key
Reduce Phase — Reducers aggregate values per key
Job Completion — Results written to HDFS; client notified

Limitations of Classic Framework

🔗 Tight Coupling

JobTracker handles BOTH job scheduling AND resource management → bottleneck.

🚫 Limited Models

Designed only for batch processing. No streaming or interactive support.

📦 Resource Inefficiency

Tasks run in fixed "slots" — wastes resources if tasks have varying needs.

⚠️ Single Point of Failure

JobTracker failure halts entire cluster.

7️⃣ YARN (Yet Another Resource Negotiator)

Introduced in Hadoop 2.0 to fix Classic MR's limitations. Decouples resource management from processing.

YARN Architecture

👑 ResourceManager (RM)

Master daemon. Manages cluster resources, schedules applications across NodeManagers.

🔧 NodeManager (NM)

Per-node daemon. Manages local resources, launches containers, reports to RM.

🎯 ApplicationMaster (AM)

Per-application process. Negotiates resources from RM, works with NMs to execute tasks.

📦 Container

Unit of resource allocation (CPU + memory bundle) on a NodeManager.

How YARN Works (MapReduce on YARN)

Client submits MR job to ResourceManager
RM allocates a container for the ApplicationMaster on some NodeManager
ApplicationMaster starts; requests more containers from RM for Map/Reduce tasks
NodeManagers launch containers running Map/Reduce code
AM monitors progress, requests retries on failures
When job is complete, AM tells RM, RM releases resources

YARN's Benefits

🎭 Multi-Framework

MapReduce, Spark, Tez, Storm all share the same cluster

📈 Better Scalability

Up to ~10,000 nodes vs ~4,000 with classic

💪 High Availability

Standby ResourceManager (no SPOF)

📦 Flexible Resources

Container model adapts to task needs

8️⃣ MapReduce Daemons — 1.x vs 2.x

Hadoop 1.x — Classic MapReduce

  • JobTracker (master) — coordinates & manages MR jobs, assigns tasks to TaskTrackers, monitors progress
  • TaskTracker (worker, per node) — executes Map/Reduce tasks assigned by JobTracker

Drawback: JobTracker handles both job scheduling AND resource management — single point of failure & bottleneck.

Hadoop 2.x — YARN Era

  • NodeManager (worker, per node) — manages local resources, launches & monitors containers
  • ResourceManager (master) — allocates cluster resources to applications
  • ApplicationMaster (per app) — manages individual app lifecycle, negotiates containers with RM

Benefit: Decoupled architecture, multi-framework support, better scaling.

9️⃣ MapReduce Programming Examples

Example 1: Word Count

Problem: Count frequency of each word in a text corpus.

Map Function

map(line_number, line_text): for each word in line_text.split(): emit(word, 1)

Reduce Function

reduce(word, list_of_counts): total = sum(list_of_counts) emit(word, total)

Walkthrough

Input: "The cat sat on the mat" "The cat is fat" Map: (The,1)(cat,1)(sat,1)(on,1)(the,1)(mat,1) (The,1)(cat,1)(is,1)(fat,1) Shuffle/Sort: The → [1,1] cat → [1,1] sat → [1] on → [1] the → [1] mat → [1] is → [1] fat → [1] Reduce: (The,2)(cat,2)(sat,1)(on,1)(the,1)(mat,1)(is,1)(fat,1)

Example 2: Matrix Multiplication

Problem: Compute C = A × B where C[i][j] = Σₖ A[i][k] × B[k][j]

Map Function

For each element of A[i][k]: emit each j: ((i,j), ("A", k, value)) For each element of B[k][j]: emit each i: ((i,j), ("B", k, value))

Reduce Function

reduce((i,j), values): # Separate A's and B's A_values = { k: v for ("A", k, v) in values } B_values = { k: v for ("B", k, v) in values } # Multiply pairs with same k, sum result = sum(A_values[k] * B_values[k] for k in A_values) emit((i,j), result)

Example 3: SQL Join via MapReduce

Problem: Join Customers (id, name) with Orders (cust_id, amount) on customer ID.

Map Function

For each row in Customers: emit(cust_id, ("C", name)) For each row in Orders: emit(cust_id, ("O", amount))

Reduce Function

reduce(cust_id, values): customers = [v for ("C", v) in values] orders = [v for ("O", v) in values] # Cross-product for customer in customers: for order in orders: emit(cust_id, (customer, order))
💡 Key Pattern: Tag records with source ("A", "B" or "C", "O") so Reducer can distinguish them.

📝 Summary & Exam Tips

🎯 Key Takeaways

  • MapReduce = programming model for parallel processing of large data
  • 3 stages: Map → Shuffle/Sort → Reduce
  • Mapper emits (key, value); Reducer aggregates values per key
  • Combiner: optional mini-reducer running locally on Mapper output (must be associative + commutative)
  • Partitioner: routes keys to Reducers (default = hash partition)
  • Classic MR (1.x): JobTracker + TaskTrackers (limited, SPOF)
  • YARN (2.x): ResourceManager + NodeManager + ApplicationMaster + Container
  • YARN benefits: multi-framework support, scalability, HA
📌 Common Exam Pitfalls:
  • Forgetting that Shuffle/Sort happens between Map and Reduce
  • Saying Combiner replaces Reducer (it's optional optimization)
  • Using Combiner for non-associative operations (like average)
  • Mixing up JobTracker (1.x) vs ResourceManager (2.x)
