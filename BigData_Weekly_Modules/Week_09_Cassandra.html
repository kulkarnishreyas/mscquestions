Where does Cassandra fit?

Cassandra is the storage layer for streaming systems — high write throughput makes it perfect for ingesting massive streams of events.

2️⃣ Apache Cassandra Introduction

Cassandra is a distributed, decentralized, NoSQL column-oriented database designed for handling massive amounts of data across many commodity servers.

Origin

  • Developed at Facebook in 2008 for the inbox search feature
  • Open-sourced and donated to Apache
  • Inspired by Google's Bigtable (data model) and Amazon's Dynamo (architecture)

Cassandra in CAP Theorem

Cassandra is AP by default (Availability + Partition Tolerance):

  • Sacrifices strong consistency for high availability
  • Offers tunable consistency — you can choose stronger consistency per query if needed
💡 Cassandra vs HBase: Both column-oriented, but:
  • Cassandra: AP (high availability), peer-to-peer (no master)
  • HBase: CP (strong consistency), master-slave architecture

Use Cases

📊 Time-Series Data

IoT sensor readings, logs, metrics

💬 Messaging

Inbox systems, chat history

🌐 Large Web Apps

Twitter, Netflix, Instagram, Discord

🛒 Personalization

Recommendation engines, user profiles

3️⃣ Core Features of Cassandra

🌐 Distributed

Data partitioned across nodes; new nodes joinable seamlessly

🤝 Decentralized (Peer-to-Peer)

Every node is identical — no master/slave. No single point of failure.

📈 Elastic Scalability

Add nodes → throughput grows linearly

🛡️ Fault Tolerance

Replication ensures data availability if nodes fail

🎛️ Tunable Consistency

Per-query consistency level (ONE, QUORUM, ALL, etc.)

⚡ High Write Throughput

Designed for fast writes — ideal for streaming

📅 Time-Series Friendly

Partition design naturally fits time-stamped data

💻 Commodity Hardware

No need for specialized servers

4️⃣ Cassandra Ring Architecture

Cassandra organizes nodes in a logical ring. Each node owns a range of data based on the partition key's hash value.

How Data is Distributed

  1. Each row has a partition key
  2. Cassandra hashes the partition key (default: Murmur3)
  3. The hash determines which node owns that data (based on the ring position)
  4. Data is also replicated to N-1 neighbor nodes (per replication factor)

Gossip Protocol

Nodes communicate via gossip protocol:

  • Every second, each node randomly picks a few others and exchanges state
  • Spread cluster topology, node health, schema versions
  • Eventually consistent view of cluster — no central coordinator needed

Snitch

The snitch tells Cassandra about network topology:

  • Which nodes are in which rack / data center
  • Used by replication strategies to place replicas in different racks/DCs

5️⃣ Replication Factor

Replication Factor (RF) = how many copies of each piece of data are stored across the cluster.

Why Replication?

🛡️ Fault Tolerance

If a node fails, data still accessible from other replicas

⚡ Read Scaling

Reads can hit any replica — distributed load

🌍 Multi-DC

Replicas in different data centers for disaster recovery

Configuration

  • Set at the keyspace level (a keyspace is like a database)
  • Typical RF = 3 (3 copies of every piece of data)
  • Can be different for different data centers

Replication Strategies

StrategyUse Case
SimpleStrategySingle data center; replicates to N consecutive nodes on the ring
NetworkTopologyStrategyMulti-data-center; specify RF per DC

Tunable Consistency

For each query, you specify a consistency level:

LevelMeaning
ONEOne replica must respond
QUORUMMajority of replicas must respond (e.g., 2 of 3)
ALLAll replicas must respond (strong consistency)
LOCAL_QUORUMQuorum in local data center
EACH_QUORUMQuorum in EACH data center
💡 Trade-off:
  • ONE = fastest, weak consistency
  • ALL = strongest consistency, slowest, lowest availability
  • QUORUM = balanced sweet spot
Formula: To get strong consistency: R + W > RF where R = read consistency, W = write consistency

6️⃣ Handling Streaming Data in Cassandra

Why Cassandra for Streaming?

⚡ Fast Writes

Log-Structured Merge Tree (LSM) design → very fast writes (no disk seek for writes)

📈 Linear Scalability

Stream rate exceeds one node? Just add more.

📅 Time-Series Optimized

Partition by entity (sensor_id), cluster by timestamp

🌐 Always Available

AP design ensures writes never blocked

Time-Series Pattern

Common Cassandra pattern for streaming/time-series data:

CREATE TABLE sensor_readings ( sensor_id UUID, reading_time TIMESTAMP, temperature DOUBLE, humidity DOUBLE, PRIMARY KEY ((sensor_id), reading_time) ) WITH CLUSTERING ORDER BY (reading_time DESC);
  • Partition key: sensor_id — all readings for one sensor on same node
  • Clustering column: reading_time — readings sorted by time within partition
  • CLUSTERING ORDER DESC: newest first — fast access to recent data

7️⃣ Indexes in Cassandra

Primary Key — The Main Index

The primary key is the most important index in Cassandra. It determines:

  • Uniqueness of rows
  • Data distribution across nodes (via partition key)
  • Sort order within a partition (via clustering columns)

Primary Key Components

PRIMARY KEY (partition_key, clustering_col1, clustering_col2) -- or for composite partition key: PRIMARY KEY ((part_key1, part_key2), clustering_col1, clustering_col2)
ComponentRole
Partition KeyDetermines which node holds the data
Clustering ColumnsSort order within a partition

Secondary Indexes

Allow querying on columns that aren't part of the primary key.

CREATE INDEX ON users (email); -- Then query: SELECT * FROM users WHERE email = 'alice@example.com';

When to Use Secondary Indexes

✅ Good for

  • Low-to-moderate cardinality (e.g., status, country)
  • Small datasets
  • Specific column queries within a partition

❌ Avoid for

  • High cardinality (e.g., unique emails, UUIDs)
  • Very large datasets (slow scans across nodes)
  • Frequently updated columns
⚠️ Best Practice: Instead of secondary indexes on high-cardinality columns, create a second table with that column as the partition key. This is denormalization — common pattern in Cassandra.

SASI Indexes (Storage-Attached Secondary Index)

More efficient than regular secondary indexes:

  • Support range queries (e.g., WHERE age > 18)
  • Support LIKE queries on strings
  • Still avoid for high-write workloads

8️⃣ Cassandra Query Language (CQL)

CQL is Cassandra's primary interface — looks SQL-like but has different semantics underneath.

Key Differences from SQL

SQLCQL
JOINs supportedNo JOINs
SubqueriesNo subqueries
Transactions across rowsLightweight Transactions (single row)
WHERE on any columnWHERE only on primary key (or with ALLOW FILTERING / index)
DatabaseKeyspace
Schema-on-writeSchema-on-write (but no FK)

DDL — Data Definition Language

Keyspaces

-- Create keyspace CREATE KEYSPACE mykeyspace WITH REPLICATION = { 'class': 'SimpleStrategy', 'replication_factor': 3 }; -- Multi-DC strategy CREATE KEYSPACE retail WITH REPLICATION = { 'class': 'NetworkTopologyStrategy', 'DC1': 3, 'DC2': 2 } AND DURABLE_WRITES = TRUE; -- Use keyspace USE mykeyspace; -- Drop keyspace DROP KEYSPACE mykeyspace;

Tables

CREATE TABLE users ( user_id UUID PRIMARY KEY, first_name TEXT, last_name TEXT, email TEXT, signup_date TIMESTAMP ); -- Composite primary key CREATE TABLE temperature_readings ( device_id UUID, event_time TIMESTAMP, temperature FLOAT, PRIMARY KEY ((device_id), event_time) ) WITH CLUSTERING ORDER BY (event_time DESC); -- Alter table ALTER TABLE users ADD age INT; ALTER TABLE users DROP age; -- Drop table DROP TABLE users;

DML — Data Manipulation Language

-- INSERT INSERT INTO users (user_id, first_name, last_name, email) VALUES (uuid(), 'John', 'Doe', 'john@example.com'); -- INSERT with TTL (auto-delete after 3600 sec = 1 hour) INSERT INTO sessions (id, token) VALUES (uuid(), 'abc123') USING TTL 3600; -- SELECT SELECT * FROM users WHERE user_id = ?; SELECT first_name, last_name FROM users LIMIT 10; -- UPDATE UPDATE users SET last_name = 'Smith' WHERE user_id = ?; -- DELETE DELETE FROM users WHERE user_id = ?; DELETE last_name FROM users WHERE user_id = ?; -- Just one column -- COUNT (slow on big data!) SELECT COUNT(*) FROM users;

Batch Operations

BEGIN BATCH INSERT INTO users (user_id, name) VALUES (uuid(), 'Alice'); INSERT INTO users (user_id, name) VALUES (uuid(), 'Bob'); UPDATE users SET status = 'active' WHERE user_id = ?; APPLY BATCH;
⚠️ Batches in Cassandra: Not for performance optimization (unlike SQL batches). They're for atomicity — all succeed or all fail. Can be slower if not all in same partition.

Counter Tables — Special!

Counters are atomic and can only be modified via UPDATE (no INSERT):

CREATE TABLE pageviews ( page_id UUID, views COUNTER, PRIMARY KEY (page_id) ); -- Increment UPDATE pageviews SET views = views + 1 WHERE page_id = ?; -- Decrement UPDATE pageviews SET views = views - 1 WHERE page_id = ?;
⚠️ Counter restrictions:
  • Cannot mix counter and non-counter columns in same table
  • No regular INSERT — only UPDATE with arithmetic
  • No TTL allowed

📝 Summary & Exam Tips

🎯 Key Takeaways

  • Streaming Analytics = data in motion, real-time processing (vs batch)
  • Cassandra: distributed, decentralized (peer-to-peer), AP NoSQL column-oriented DB
  • From Facebook — inspired by Google Bigtable + Amazon Dynamo
  • Ring architecture: nodes form a logical ring; partition key hash determines node
  • Gossip protocol: nodes exchange cluster state randomly each second
  • Replication factor set at keyspace level (typical RF = 3)
  • Tunable consistency: ONE / QUORUM / ALL / LOCAL_QUORUM per query
  • Primary key = partition key + clustering columns
  • CQL = SQL-like but NO JOINs, NO subqueries; WHERE on PK only (mostly)
  • Counter type = atomic, only updatable (no insert), no TTL
  • Collections: list (ordered, dups), set (unique), map (K/V)
📌 Common Exam Pitfalls:
  • Saying Cassandra has a master (it doesn't — peer-to-peer!)
  • Confusing Cassandra (AP) with HBase (CP)
  • Forgetting RF is set at keyspace level, not table
  • Trying to use JOINs in CQL (not supported)
  • Mixing counter and regular columns in same table
