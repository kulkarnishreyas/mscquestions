5️⃣ MongoDB Introduction

MongoDB is a document-oriented NoSQL database designed for handling unstructured/semi-structured data at scale.

Why MongoDB for Big Data?

  • Handles variety (unstructured docs) naturally
  • Scales horizontally via sharding
  • High availability via replica sets
  • Flexible schema — fields can differ per document
  • Excellent for evolving data models (agile development)

Common Use Cases

📝 Content Management

Articles, blogs with varying attributes

🛒 E-commerce Catalogs

Products with very different attributes

🎮 Real-time Apps

Gaming leaderboards, IoT data

📱 Mobile Backends

User profiles, app state

📊 Analytics

Event tracking, logs aggregation

🤖 Catalog & Recommendation

Personalization engines

Database

  • Container for collections
  • Can have multiple collections
  • Created automatically when you first store data

Collection

  • A group of documents
  • Roughly analogous to a SQL table
  • Schema-less — documents in same collection can have different fields

Document

  • Single record in a collection
  • Composed of field-value pairs (like JSON objects)
  • Stored as BSON internally
  • Each document MUST have _id field (auto-generated ObjectId if not provided)

Example Document

{ "_id": ObjectId("507f1f77bcf86cd799439011"), "name": "Alice Smith", "age": 30, "active": true, "email": "alice@example.com", "tags": ["admin", "premium"], "address": { "city": "Vellore", "country": "India", "zip": "632014" }, "joined": ISODate("2023-01-15") }
💡 Embedded documents: MongoDB allows nested objects (like address here). This denormalizes data, often avoiding the need for joins.

7️⃣ BSON & Data Format

What is BSON?

BSON = Binary JSON. MongoDB's internal storage format.

BSON vs JSON

AspectJSONBSON
FormatTextBinary
StorageExternal / human-readableInternal MongoDB
SpeedSlow to parseFast traversal (typed)
TypesString, Number, Bool, Array, Object, null+ ObjectId, Date, BinData, Decimal128
SizeCompact textSometimes larger (length prefixes)

Why BSON over JSON?

⚡ Speed

Type/length prefixes allow fast scanning without parsing whole document

🎯 Rich Types

Native Date, Decimal128, ObjectId, Binary — critical for DB operations

🔍 Indexable

Binary format optimized for index lookups

🔄 JSON-compatible

Drivers convert between BSON ↔ JSON for app code

ObjectId Structure

The default _id — 12 bytes:

  • 4 bytes: timestamp (seconds since epoch)
  • 5 bytes: random value (per process)
  • 3 bytes: incrementing counter

This guarantees uniqueness across machines + time, without coordination.

7️⃣B Basic Operations & Indexes

Database & Collection Management

  • use <dbname> — switches to a database; creates it implicitly on first data insert
  • show dbs / show databases — list databases
  • Collections are created automatically when the first document is inserted
  • Explicit creation: db.createCollection("name")
  • Databases and collections can be dropped (db.dropDatabase(), db.collection.drop())

Finding Documents

db.collection.find() // returns a cursor to all documents db.collection.find({ field: value }) // documents matching criteria db.collection.findOne({ field: value }) // first matching document only

Indexes — Speeding Up Reads

Special data structures that store a small portion of the collection in an easy-to-traverse form. Like an index in the back of a book — find documents without scanning the whole collection.

  • MongoDB automatically creates a unique index on _id for every collection

Common User-Defined Index Types

🎯 Single Field

Index on one field. Ascending: { field: 1 }, Descending: { field: -1 }

🧩 Compound Index

Index on multiple fields — order matters. { field1: 1, field2: -1 }

📚 Multikey

Automatic when the indexed field is an array — indexes each element separately

🌍 Geospatial

For querying location data — points, polygons, near-queries

🔤 Text

For searching string content with text-search operators

#️⃣ Hashed

Hash of field value — used for hash-based sharding

⚠️ Trade-off: Indexes speed up reads (find, $match) but slow down writes (insert, update, delete) because the index must also be updated. They also consume storage. Choose based on common query patterns.

8️⃣ Replication — Replica Sets

Replication = synchronizing data across multiple servers.

Why Replication?

🛡️ High Availability

If primary fails, secondary becomes new primary (automatic failover)

📋 Data Redundancy

Multiple copies protect against data loss

🌍 Disaster Recovery

Copies in different locations

📈 Read Scaling

Read operations distributed to secondaries

🔧 No-Downtime Maintenance

Backup, index rebuild without disrupting service

Replica Set Structure

Typically 3+ nodes, with one Primary and others Secondary.

👑 Primary
Single node. Handles ALL writes. Records to oplog.
🔄 Secondary 1
Replicates from primary's oplog. Can serve reads.
🔄 Secondary 2
Replica + vote in elections.

How Replication Works

  1. Writes go to Primary; appended to oplog (capped collection)
  2. Secondaries asynchronously replay the oplog
  3. Heartbeats every 2 sec between members
  4. If Primary unreachable for ~10-12 seconds → election triggered
  5. Eligible secondary with majority votes becomes new Primary
  6. Old Primary rejoins as Secondary when recovered

Typical Configurations

P-S-S (Recommended)

Primary + 2 Secondaries. Two full data copies + primary. Good fault tolerance.

P-S-A

Primary + Secondary + Arbiter. Arbiter votes but holds no data. Cheaper but less redundancy.

💡 Odd number rule: Always use an odd number of voting members (3, 5, 7) to avoid election ties.

Replication Lag

  • Delay between Primary write and Secondary applying it
  • Causes: network latency, slow disk on secondaries, long-running ops on primary, heavy write load
  • Monitor: rs.printSecondaryReplicationInfo() / rs.printSlaveReplicationInfo() (legacy alias) / rs.status()
  • Network connectivity: all members must reach each other — firewalls & networking must be configured

Write Concerns

How MongoDB confirms writes to balance performance vs durability:

  • w: 1 — acknowledged by primary only (fastest, default)
  • w: majority — acknowledged by majority (safest)
  • w: 0 — no acknowledgment (fire-and-forget)

9️⃣ Sharding — Horizontal Scaling

Sharding = distributing data across multiple machines when data is too big for one replica set.

Why Shard?

💾 Storage Capacity

Single servers have storage limits — shards add unlimited capacity

⚡ High Throughput

Distribute writes/reads across many shards

📈 Horizontal Scaling

Add servers instead of upgrading one big server

Sharded Cluster Components

🚦 mongos
Query router. Client interface. Routes to correct shard.
📚 Config Servers (CSRS)
Deployed as a Config Server Replica Set (CSRS). Holds metadata: chunk → shard mapping. Authoritative source for cluster state.
🗄️ Shard 1
Replica set holding part of data
🗄️ Shard 2
Replica set with another part

How It Works

  1. Each shard = a replica set (data redundancy within shard)
  2. Data is split into chunks based on a shard key
  3. Chunks are distributed across shards
  4. Clients connect to mongos, not shards directly
  5. mongos consults config servers to route query to correct shard
  6. If query needs multiple shards, mongos aggregates results

Choosing a Shard Key

✅ Good Shard Key

  • High cardinality (many unique values)
  • Low frequency (no hot spots)
  • Monotonically non-changing

❌ Bad Shard Key

  • Low cardinality (e.g., gender)
  • Monotonically increasing (writes pile on last shard)
  • Unpredictable query patterns

Sharding + Replication Combined

+-------------------------+ | mongos (clients) | +-------------------------+ | +--------+--------+ | | +-----+----+ +-----+----+ | Config | | Config | ← Config Servers (replica set) +----------+ +----------+ | | +-----+----+ +-----+----+ | Shard 1 | | Shard 2 | ← Each shard is a replica set | (P+S+S) | | (P+S+S) | +----------+ +----------+
⚠️ Common pitfall: Sharding is complex. Only shard when your data truly exceeds what a single replica set can handle — usually multi-TB datasets or massive write loads.

📝 Summary & Exam Tips

🎯 Key Takeaways

  • Big Data Vs: Volume, Velocity, Variety (+ Veracity, Value, Variability)
  • Analytics cycle: 8 steps from Business Understanding → Feedback
  • MongoDB = document-oriented NoSQL DB for unstructured data
  • Hierarchy: Database → Collection → Document
  • BSON = Binary JSON; fast traversal, rich types (ObjectId, Date, etc.)
  • Every doc has _id field (auto ObjectId)
  • Replica Set: 1 Primary (writes) + N Secondaries (replicate via oplog, can read) ± Arbiter
  • Election: triggered ~10-12s after primary failure; majority wins
  • Sharding = horizontal scaling; mongos (router) + Config Servers + Shards (each a replica set)
  • Choose shard key: high cardinality, low frequency, non-monotonic
📌 Common Exam Pitfalls:
  • Forgetting that only Primary handles writes in replica set
  • Confusing arbiter (votes, no data) with config server (metadata)
  • Saying clients talk to shards directly (they go through mongos)
  • Forgetting each shard is itself a replica set
  • Mixing up JSON (text, external) vs BSON (binary, internal)
