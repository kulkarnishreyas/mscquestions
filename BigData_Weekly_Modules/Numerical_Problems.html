1. HDFS Block-Size & Replication Math (Week 2)

Problem 1 — Splitting a file into blocks

A 500 MB file is stored on HDFS with a default block size of 128 MB and replication factor 3. How many blocks does HDFS create and how many physical copies are stored?

Solution

  • 500 MB ÷ 128 MB = 3 full blocks + 1 partial block → 4 blocks total: 128 + 128 + 128 + 116 MB
  • Each block is replicated 3× → 4 × 3 = 12 physical block copies across the cluster

(Default block sizes in HDFS are typically 128 MB or 256 MB; default replication factor = 3.)

Problem 2 — Replica placement strategy

Where does HDFS place the 3 replicas of a given block?

Solution (rack-aware placement)

  1. Replica 1 — on the local DataNode (the writer's node)
  2. Replica 2 — on a DataNode in a different rack (rack-failure resilience)
  3. Replica 3 — on a different DataNode in the same rack as Replica 2 (saves cross-rack bandwidth on the second write)

3. K-Means Clustering with MapReduce (Week 11)

Problem — Sketch one full MR iteration of K-Means

Given an initial set of K centroids, describe the Map and Reduce phases and the output of one full pass.

Solution

  1. Initialization: Select K initial centroids.
  2. Map Phase: Each mapper receives a chunk of points. For each point, it computes the distance to each of the K centroids and assigns the point to the nearest centroid.
    Emits: (centroid.id, data.point)
  3. Combine Phase (optional): Partially aggregates points belonging to the same centroid within a mapper to reduce shuffle volume.
  4. Reduce Phase: Each reducer receives all points assigned to one centroid. It averages those points to compute the new centroid coordinates.
    Emits: (centroid.id, new_centroid_coordinates)
  5. Iterate: Repeat Map + Reduce phases until centroids converge or a maximum iteration count is reached.
💡 Numerical considerations from PDF:
  • Choosing K — needs to be set up-front or experimented
  • Distance metric — Euclidean or Manhattan, depending on data
  • Convergence — stop when assignments no longer change
  • Iteration cost — multiple MR passes can be expensive on very large data

4. Cassandra Replication Factor (Weeks 9 & 10)

Problem — Sizing a Cassandra keyspace

A keyspace is created with replication_factor = 3 in a 5-node cluster. How many copies of each row exist? What happens if 1 node fails?

Solution

  • 3 physical copies of each row spread across 3 distinct nodes.
  • With RF=3 and only 1 node failure, both reads and writes remain available — 2 replicas are still up.
  • Read/write consistency level controls how many replicas must acknowledge before the operation is considered successful (e.g., QUORUM = ⌊RF/2⌋ + 1 = 2 of 3).

Problem — Apriori on MapReduce: minimum-support filter (Week 12)

Frequencies of itemsets emitted by mappers are aggregated in the Reducer. The Reducer must keep only itemsets meeting minimum support. If min_support = 100 and three mappers contribute partial counts of 40, 35, 30 for an itemset, is it frequent?

Solution

Reducer sums partial counts: 40 + 35 + 30 = 105 ≥ 100 → itemset is frequent, kept for rule generation. (If the sum were less than 100, the itemset would be pruned.)

Multiple MapReduce passes may be required: pass 1 finds frequent 1-itemsets, pass 2 finds frequent 2-itemsets (using only items kept from pass 1), and so on (the Apriori "bottom-up" principle).
